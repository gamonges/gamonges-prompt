# 判定チェックリスト（回帰 vs 環境要因）

非2xx・失敗を検出した場合、以下のパターンに該当しないか先に確認してから「回帰」と結論する。

- **後続処理が非2xxだが実処理は成功している**: 直前の同一操作の再実行による重複・一意制約違反（例: 同名キー衝突）を疑う。別パラメータ（別名・別フォルダ等）で再実行し切り分ける。
- **複数経路が同時にネットワークエラーで失敗する**: 個別経路の不具合ではなく、バックエンド/依存サービスの再起動・停止を疑う。ヘルスチェックエンドポイントやプロセス状態を確認する。
- **インフラ復旧直後に認証エラーが出る**: セッション/トークンの失効を疑い、再認証してから再検証する。
- **想定と異なるリクエストを捕捉している**: 期待した書込み系リクエストではなく、無関係な読み取り系リクエストを誤って捕捉していないか、HTTPメソッドとURL形状で区別する。
- **非冪等な操作**: 同一パラメータでの再実行が失敗する経路固有の制約は、回帰ではなく仕様として記録に残す。
- **原則（症状の層で分岐する）**:
  - 複数経路が同時に**ネットワーク層（接続断・5xx・タイムアウト）で**落ちたらインフラ要因を疑う。障害期に取得した結果は破棄し、健全期の結果のみを採用する。
  - 一方、複数経路が**同一の 4xx / バリデーション / レスポンス内容で**落ちる場合は、コア共通関数の回帰を第一に疑う。「コア 1 つ × シンク N 種」の構造上、コアの回帰は複数経路に同時に現れるため、同時多発を機械的にインフラ要因と断じない。

上記いずれにも該当しない場合のみ、回帰として記録する。

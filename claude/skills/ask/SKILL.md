ユーザーの質問に対して、必要十分な調査と分析を行い回答する。
回答に必要な範囲で、調査対象のコード・構成・アーキテクチャの仕様を理解してから回答すること。

**重要**: コードの変更をしない。調査結果に基づいた回答のみを提供し、調査結果を`./tmp/research.md`に記録する。

### research.md への記録ルール

- **デフォルトは追記**: 既存の `./tmp/research.md` がある場合、末尾に `---`（区切り線）を挿入してから調査結果を追記する。既存の内容を削除・上書きしない。
- **上書きする場合**: ユーザーから「上書き」「リセット」「クリア」等の明示的な指示があった場合のみ、ファイルを上書きする。

## パラメーター

`$ARGUMENTS` で質問ファイルのパスを指定できる。省略時は `./tmp/context.md` を使用する。

## ワークフロー上の位置付け

本コマンドは独立して使用できる。`/design`、`/revise`、`/implement` の前後いずれでも利用可能。

## 実行条件

以下のファイルが存在することを確認する:

- `$ARGUMENTS`（指定がない場合は `./tmp/context.md`）— ユーザーの質問
- `openspec/specs/`（任意）— 既存の仕様ファイル。存在する場合は自動的に参照する

ファイルが存在しない場合:

- プロセスを即座に停止する
- どの条件が満たされていないかをユーザーに報告する
- 処理を続行しない

## ハルシネーション防止ガイドライン

回答の信頼性を確保するため、以下のルールに従う:

- [ ] コードベース・ドキュメント・調査結果で**直接確認した情報のみ**を使用する
- [ ] コードを引用する場合は**ファイルパスと行番号**を付ける（例: `src/module/file.ts:L42`）
- [ ] 根拠が見つからない場合は「**確認できませんでした**」と明記する
- [ ] **事実**と**推測**を明確に区別する（推測には「推測ですが」「可能性がありますが」等の前置きを付ける）
- [ ] 外部情報を引用する場合は**URL を付記**する

## 実行プロセス

### フェーズ 1: 質問の読み込みと整理

- [ ] 質問ファイル（`$ARGUMENTS` または `./tmp/context.md`）から質問を読み込む
- [ ] 個別の質問を識別する（区切り: `=====` やリスト形式など）
- [ ] ファイル参照やコンテキスト情報を抽出する（例: `tmp/plan.md:L144`）
- [ ] 各質問の種類を分類する:
  - **コードベースに関する質問**: 実装の詳細、アーキテクチャ、パターン
  - **技術的な質問**: ライブラリ、フレームワーク、ベストプラクティス
  - **設計に関する質問**: 設計判断、トレードオフ、代替案

### フェーズ 2: 調査

質問が複数あり相互に独立している場合、または複数ファイル・複数の情報源を横断する必要がある場合は、サブエージェントを同一ターンで複数 spawn して並列に調査する。単一の質問や 1 ファイルで完結する調査は直接実行する。各調査では以下を活用する:

- [ ] **コードのセマンティック検索（使い分け）**: 概念/自然言語で探す場合、対象の絶対パス（`git rev-parse --show-toplevel`）で claude-context `get_indexing_status` を確認する。応答に `✅` を含む（index 済み）なら `search_code(absPath, query)` を優先。`isError`（未 index）や `🔄`（進行中）なら未準備として Serena `search_for_pattern`・grep にフォールバックする（claude-context が効くのはそのパスが index 済みのときのみ。`🔄` は indexing 完了後の再実行で使える）。自動 index はせず、**未 index のときだけ** `/context-index` を案内する。シンボルの定義・参照・実装・構造把握は Serena（`find_symbol` / `find_referencing_symbols` / `get_symbols_overview`）を使う
- [ ] Web search tool で外部ドキュメント、事例、技術情報を検索する
- [ ] Context7 MCP でライブラリ・フレームワークの使用方法を確認する
- [ ] 参照されたファイルを直接確認し、コンテキストを理解する
- [ ] Notion MCP が利用可能な場合、ADR データベースを検索し、関連する過去の技術判断・設計決定の背景を調査する
- [ ] `openspec/specs/` が存在する場合、関連する既存仕様を参照し、仕様の背景や制約を確認する

### フェーズ 3: 分析・検証

- [ ] 収集した情報を整理し、各質問に対する回答を組み立てる
- [ ] **批判的思考を適用**する: 質問の前提自体が正しいかを検証する
- [ ] 複数の情報源から得た回答が整合するか確認する
- [ ] 回答に根拠があるか最終確認する（ハルシネーション防止ガイドラインに照らす）

### フェーズ 4: 回答

各質問に対して、以下の構造で回答する:

```
## 質問 1: [質問の要約]

### 回答
[調査結果に基づいた回答]

### 参照
- `file_path:line_number` — [参照内容の説明]
- [URL] — [外部情報の説明]

## 質問 2: [質問の要約]

### 回答
[調査結果に基づいた回答]

### 参照
- ...
```

回答のポイント:

- [ ] 簡潔で明確な回答を提供する
- [ ] 具体的なファイルパスと行番号を付記する
- [ ] 必要に応じてコードスニペットや例を含める
- [ ] 確認できた事実と推測を明確に区別する

#### HTML view 化 (オプション)

完了報告に続けて、以下をユーザーに尋ねる:

> **HTML 化しますか?** (人間レビュア向けのデザイン HTML を生成)

ユーザーが Yes と回答した場合、**Claude は自動実行せず**、次のコマンドを案内する:

```
/html-view ./tmp/research.md
```

ユーザーが明示的に slash command を入力することで HTML 生成 + ブラウザ自動起動が完了する。

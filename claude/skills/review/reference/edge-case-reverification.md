# Edge Case 再検証パターン（subagent 並列）

> SKILL.md Phase 3 完了後、特に **Critical / Minor Issue がある場合** に追加で起動する subagent パターン。
> 単体テストでカバーされにくいエッジケースを subagent に再シミュレートさせ、レビューの漏れを減らす。

## 適用判断

以下のいずれかに該当する場合、本パターンを適用する:

- Phase 3 のレビューで Critical Issue が 1 件以上
- 変更が **境界値処理** / **エラーハンドリング** / **非同期処理** / **並行処理** を含む
- 変更が **テナント分離 / 認可** / **トランザクション境界** を含む
- 既存テストのカバレッジが低い領域（テストなしファイルの変更）

該当しない場合（適用不要）:

- 純粋な UI スタイル変更
- ドキュメント / コメントのみの変更
- リネーム / ファイル移動のみ

## 実行プロセス

### 1. エッジケースカテゴリの選択

変更内容に応じて以下のカテゴリから 2-4 個を選択し、各カテゴリに対して 1 つの subagent を **並列起動** する:

| カテゴリ | エッジケース例 |
|---------|---------------|
| **境界値** | 0 / 1 / max / max-1 / 空配列 / 空文字 / null / undefined / 巨大入力 |
| **エラー伝播** | DB エラー / ネットワークタイムアウト / 認証失敗 / ロック競合 |
| **並行処理** | 同一リソースへの同時更新 / レースコンディション / デッドロック |
| **テナント越境** | 別 organization のデータ参照試行 / 認可ヘッダ欠落 / role 不足 |
| **データ不整合** | 外部キー欠落 / orphan record / マイグレーション失敗時のロールバック |
| **入力サニタイズ** | XSS payload / SQL injection 文字列 / 長すぎる文字列 / 制御文字 |

### 2. Subagent への指示テンプレート

各カテゴリ別 subagent には以下のフォーマットで指示する:

```markdown
**役割**: {category} カテゴリのエッジケース再検証

**スコープ**: PR #{number} で変更されたコード（`tmp/review/_pr-diff.snapshot` を参照）

**タスク**:
1. 変更されたコードパスをトレースし、{category} に該当するエッジケースを 3-5 件列挙する
2. 各エッジケースについて、**実行されたら何が起きるか** を具体的に記述する
3. 既存テストがカバーしているか確認する（`tests/` を grep）
4. カバーされていない場合、**Critical / Minor / Info** に分類する

**出力**: `tmp/review/_edge-{category}-review.md` に以下フォーマットで記述

## Edge Case Review — {category}

### {EC-1} 入力境界値での挙動
- **シナリオ**: ...
- **トレース**: src/foo/bar.ts:L42 → src/baz/qux.ts:L15
- **既存カバレッジ**: なし（tests/baz/qux.test.ts に該当ケース無し）
- **想定挙動**: ...
- **判定**: Critical / Minor / Info
- **修正案**: ...
```

### 3. 結果のマージ

各 `_edge-{category}-review.md` を読み込み、Phase 4 の unified.md に統合する:

- Critical 判定のエッジケースは Critical Issues セクションへ
- Minor 判定は Minor Issues セクションへ
- Info 判定は Info セクションへ
- カテゴリ名を `[edge-{category}]` のタグで明示

```markdown
### 1. {問題タイトル} [edge-boundary]
- **確信度**: Medium
- **カテゴリ**: edge-case
- **ファイル**: `src/foo/bar.ts:L42`
- **該当箇所**: 0 入力時の処理
- **理由**: 配列が空の場合、`array[0]` で undefined アクセスが発生
- **修正案**: 先頭の空チェック追加 `if (array.length === 0) return null`
```

## 注意事項

- **並列起動を必須化**: 4 カテゴリを直列でやると 4× 時間がかかる。同じ turn で複数 Agent 呼出を発行する
- **Phase 3 の通常 review と重複しないよう**、subagent には「**通常 review でカバーされていないエッジケースに特化**」を明示する
- 純粋な「if/else 分岐網羅」は code-quality-reviewer の責務なので本パターンの対象外
- 再検証で問題ゼロの場合も `tmp/review/_edge-{category}-review.md` を残し、レビュー完備の証跡とする

## 効果的な適用シナリオ例

| 変更内容 | 推奨カテゴリ |
|---------|--------------|
| 新規 API エンドポイント | テナント越境 / 境界値 / エラー伝播 |
| トランザクション境界の変更 | 並行処理 / エラー伝播 / データ不整合 |
| 認可ロジックの変更 | テナント越境 / エラー伝播 |
| バッチ処理の追加 | 境界値 / 並行処理 / データ不整合 |
| 入力フォームの追加 | 入力サニタイズ / 境界値 |

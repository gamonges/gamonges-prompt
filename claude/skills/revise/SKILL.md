既存の実装計画をユーザーのフィードバックに基づいて修正する。
初期設計と実装の間のギャップを埋め、ユーザーの知見を計画に反映する。

**重要**: 後述の「対象計画ファイルの自動判別」で決定された 1 ファイル（デフォルトは `./tmp/plan.md`、状況により `./tmp/fix-plan.md`）もしくはCursorのplanモードで作成した`***.plan.md`のみ変更する
**規約**: CLAUDE.md の Skills 共通規約に従う
## パラメーター

`$ARGUMENTS` で以下を指定できる:

- **第 1 引数**: フィードバックファイルのパス（省略時は `./tmp/feedback.md` → `./tmp/context.md` の順で検索）
- **第 2 引数**: 修正対象計画ファイルのパス（省略時は後述の「対象計画ファイルの自動判別」を実施）

例:
- `/revise` — フィードバック / 計画とも自動判別
- `/revise tmp/feedback.md` — フィードバックを明示、計画は自動判別
- `/revise tmp/feedback.md tmp/fix-plan.md` — フィードバック / 計画とも明示

## ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| `/design` 計画作成 | `/revise` 計画修正 | `/revise` 再修正 or `/implement` 実装 |

## 対象計画ファイルの自動判別

第 2 引数が省略された場合、以下のロジックで対象計画ファイルを決定する:

1. `./tmp/fix-plan.md` が存在し、`./tmp/plan.md` より mtime が新しい → `./tmp/fix-plan.md` を対象とする
2. それ以外で `./tmp/plan.md` が存在する → `./tmp/plan.md` を対象とする
3. 両方存在し判定が曖昧（例: mtime が同じ）→ AskUserQuestion でユーザーに確認

決定された計画ファイルを以下「対象計画ファイル」と呼ぶ。

## 実行条件

以下のファイルが存在することを確認する:

- 対象計画ファイル（自動判別または第 2 引数で指定）
- 第 1 引数（指定がない場合は `./tmp/feedback.md` → `./tmp/context.md` の順で検索）— ユーザーのフィードバック・修正要件

いずれかのファイルが存在しない場合:

- プロセスを即座に停止する
- どの条件が満たされていないかをユーザーに報告する
- 実装を続行しない

## 実行プロセス

### フェーズ 1: 現行計画の読み込み

- [ ] 「対象計画ファイルの自動判別」で決定した計画ファイル（あるいはCursorのplanモードで作成した実装計画書）から既存の実装計画を読み込む
- [ ] 実装計画書にNoteとして修正事項があれば、その内容を確認する
- [ ] 現在の構成・要件・アプローチを把握する
- [ ] 計画の各ステップと依存関係を特定する

### フェーズ 2: フィードバックの読み込みと分類

- [ ] フィードバック入力の優先順位:
  1. `$ARGUMENTS` で指定されたファイル
  2. `./tmp/feedback.md`（存在する場合）
  3. `./tmp/context.md`（フォールバック）
- [ ] `./tmp/plan-review.md` が存在する場合、追加フィードバックとして自動参照する
- [ ] 上記で決定したファイルからフィードバックを読み込む
- [ ] 個別のフィードバック項目を識別する（区切り: `=====` やリスト形式など）
- [ ] ファイル参照やコンテキスト情報を抽出する（例: `tmp/plan.md:L144`）
- [ ] 各フィードバックを以下に分類する:
  - **要件変更**: 機能要件・非機能要件の追加・変更・削除
  - **技術的懸念**: パフォーマンス・セキュリティ・保守性への懸念
  - **アプローチ変更**: 実装方式・設計方針の変更提案

### フェーズ 3: ユーザー確認（必要に応じて）

以下の問題を検出した場合、**プロセスを即座に停止**してユーザーに報告する:

- **曖昧なフィードバック**: 複数の解釈が可能な場合
  - どのフィードバックが不明確か、または複数の解釈が考えられるかを具体的に質問する
- **矛盾する要求**: フィードバック間で整合性がとれない場合
  - 矛盾を明確に提示し、優先度の判断を依頼する
- **技術的制約の不足**: 実装方式に影響する制約が不明な場合
  - 不足している技術的詳細や制約を具体的に質問する

複数の問題がある場合のフォーマット:

```
## 問題 1: [問題の要約]
[該当するフィードバックの引用と、何が問題なのかの説明]

## 問題 2: [問題の要約]
[該当するフィードバックの引用と、何が問題なのかの説明]
```

### フェーズ 4: 調査・分析

フィードバック項目が複数あり相互に独立している場合は、サブエージェントを同一ターンで複数 spawn して並列に調査する。項目が少数、または 1 ファイルで完結する場合は直接調査する。各項目で以下を実施する:

- [ ] Serena MCP でセマンティックなコード検索を行い、フィードバックに関連するコードパターンや実装を調査する
- [ ] Web search tool で外部ドキュメントや事例を検索する
- [ ] Context7 MCP でライブラリ・フレームワークの使用方法を確認する
- [ ] フィードバックで指摘されたファイルや箇所を直接確認する

### フェーズ 5: 計画の修正方針の決定

- [ ] 各フィードバックの実現可能性を検証する
- [ ] フィードバック間の矛盾を特定し解決する
- [ ] 既存計画の有効な部分を維持しつつ、修正方針を決定する
- [ ] 修正による他ステップへの波及影響を評価する
- [ ] **波及影響マッピング**: 各フィードバックに対し以下を列挙
  - 修正対象ステップ
  - 依存しているステップ（前提）
  - 依存されているステップ（後続）
  - 該当する 6 カテゴリ（統合点 / Decision 相互 / 並列性 / 根拠 / contract / 完了パス）

> 軽微な修正（typo / 文言調整等）は「該当なし」で OK。詳細テンプレートは `../design/reference/phase-details.md` の「構造的欠陥防止チェックリスト」を参照。

### フェーズ 6: 対象計画ファイルの更新

- [ ] 上書き前に、対象計画ファイルの現在の内容を `{name}.archived-{YYYYMMDD-HHmm}-{変更理由の要約}.md`（例: `tmp/plan.archived-20260722-1430-DB接続プール見直し.md`）にコピーし、旧版を保存する。archive ファイル冒頭に「なぜ前バージョンから変更したか」を1行で記す
- [ ] 「対象計画ファイルの自動判別」で決定した計画ファイル（`./tmp/plan.md` または `./tmp/fix-plan.md`）を修正後の実装計画で上書きする
- [ ] すべてのフィードバックを反映しつつ、元の構成を維持する
- [ ] 調査で得た最新情報（ファイルパス、URL、コード例など）で古い情報を更新する
- [ ] 修正履歴は含めない（計画として完結した状態にする）（archive ファイルには変更理由を残すが、対象計画ファイル本体はクリーンな状態を保つ）

### フェーズ 6.5: 6 カテゴリ自己検証

修正後の plan を自分でチェックし、修正が新規の構造的欠陥を生んでいないか確認する。

- [ ] **統合点トレース**: 修正で追加・変更した step が新たな skip / 短絡を含まないか
- [ ] **Decision 間整合性**: 新規 Decision が既存 Decision と矛盾しないか
- [ ] **並列性**: 並列実行に関する変更で共有リソース衝突が生じないか
- [ ] **「変更不要」根拠**: 修正で「透過」「変更不要」と書いた箇所に grep / read 根拠があるか
- [ ] **HTTP contract**: 修正で追加した抽象 IF に concrete 例（method / path / header / body / response + curl）があるか
- [ ] **完了パス**: 修正で増えた / 変わった完了パスに AC が割り当たっているか

> 軽微な修正の場合は「該当なし」と明記して OK。詳細テンプレートは `../design/reference/phase-details.md` の「構造的欠陥防止チェックリスト」を参照。

違反が見つかった場合は、フェーズ 6（plan 更新）に戻って再修正。再修正後にもう一度本フェーズを実施する。

### フェーズ 7: 完了報告

対象計画ファイル（`./tmp/plan.md` または `./tmp/fix-plan.md`）の差分はコミットしない（`tmp/` 配下はリポジトリ管理対象外）。

以下の内容をユーザーに報告する:

- 計画に加えた主な変更点
- 各フィードバックへの対応結果
- フェーズ 6.5 6 カテゴリ自己検証の結果（違反の有無、再修正回数）
- アーカイブした旧版ファイルのパス

複数のフィードバックがある場合のフォーマット:

```
## フィードバック 1: [フィードバックの要約]
[この修正で何をどう変更したかの説明]

## フィードバック 2: [フィードバックの要約]
[この修正で何をどう変更したかの説明]
```

- 次のステップの案内: `/revise` で再度修正、`/implement` で実装開始

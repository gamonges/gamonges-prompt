`/design` 着手前に、扱う概念・モジュール候補・想定される結合関係を `coupling-anatomy` の3軸（統合強度×距離×変動性）で洗い出し、疎結合/高凝集になるよう設計方針を整理する。

**重要**: ソースコードは一切変更しない。`./tmp/coupling-precheck.md` のみ出力する。
**規約**: CLAUDE.md の Skills 共通規約に従う

判定基準・訳語・出力フォーマットは `coupling-anatomy` スキルを参照する（本スキルでは再宣言しない）。

## `strategic-ddd` との関係

`strategic-ddd` は**戦略的DDD**（コンテキスト境界・サブドメイン分類・ユビキタス言語など、macro-levelな概念/境界の設計）を扱う。本スキルは**戦術的**な視点（実装単位・モジュール間の結合品質、micro-levelな設計判断）を扱う。ただし本スキルの手法はDDDに限定しない（DDDを採用していない設計にも適用可能な汎用スキル）。競合ではなく補完関係にある。

`./tmp/strategic-ddd.md` が存在する場合、Phase 5（コンテキスト境界の特定）で選定したDDD関係パターン（Conformist / Published Language / Shared Kernel / ACL）があれば、コントラクト結合相当のはずの選択が実際に実装詳細を漏らしていないか等をチェックする。

## パラメーター

`$ARGUMENTS` でコンテキストファイルパスを指定できる。省略時は `./tmp/context.md` を使用する。

## ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| `/ask`（任意）/ `/strategic-ddd`（任意、Phase 5完了時に本スキルへの案内あり） | `/coupling-precheck` 設計前整理 | `/design`（`./tmp/coupling-precheck.md` を任意参照） |

## 実行条件

- `$ARGUMENTS`（指定がない場合は `./tmp/context.md`）— 必須
- `./tmp/strategic-ddd.md`（任意）
- `./tmp/research.md`（任意）

コンテキストファイルが存在しない場合: プロセスを即座に停止し、ユーザーに報告する。

## 実行プロセス

### フェーズ1: 入力読み込み

- [ ] コンテキストファイルを読み込む
- [ ] `./tmp/strategic-ddd.md` が存在する場合はあわせて読み込む
- [ ] `./tmp/research.md` が存在する場合はあわせて読み込む

### フェーズ2: モジュール候補・想定結合関係の洗い出し

- [ ] 扱う概念・モジュール候補を洗い出す
- [ ] `strategic-ddd.md` が存在する場合、そのDDD関係パターン（Conformist / Published Language / Shared Kernel / ACL）を結合軸の言葉に変換する

### フェーズ3: 3軸での仮判定

- [ ] 各想定関係を `coupling-anatomy` の3軸で仮判定する。実装前のため「推測」ベースである旨を明記した上で、これから書くコードの設計方針として提示する

### フェーズ4: 設計方針の提案

- [ ] 危険域（強度高×距離遠、または変動性の高い箇所との組み合わせ）を避ける設計方針を提案する

### フェーズ5: レポート出力

`./tmp/coupling-precheck.md` に `coupling-anatomy` の標準フォーマットのうち**推測ベース変種**（想定モジュール名/概念名、根拠列は `plan.md:L{number}` または `(推測)` タグ）で出力する。

### フェーズ6: 完了報告

- 洗い出したモジュール候補・結合関係の件数
- 危険域を避けるための設計方針の要約
- `/design` の実行を案内する

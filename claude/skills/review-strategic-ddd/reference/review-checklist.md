# review-strategic-ddd 観点詳細チェックリスト + レポート出力テンプレ

> SKILL.md の 5 観点ごとの詳細チェック項目、Finding Format、レポート出力テンプレートを集約。

## レビューの思考フレームワーク

DDD の概念設計に特化した視座で設計書を評価:

- **ドメイン適合性**: モデルが実際のビジネスドメインを正確に反映しているか
- **モデルの表現力**: 集約・イベント・コマンドがビジネスルールを十分に表現できているか
- **境界の明確性**: コンテキスト境界が責務の漏洩なく引かれているか
- **言語の一貫性**: ユビキタス言語がコード・設計書・業務の間で統一されているか
- **実装可能性**: 概念設計が `/design` での戦術設計にスムーズに引き継げるか

## 観点 1: ドメインモデルの妥当性 → `ddd-expert`

- 集約の粒度は適切か（God Aggregate / 過小集約）
- 不変条件が定義されており、ビジネスルールを網羅しているか
- 「動詞か名詞か」テストが適用されているか
- 概念分類（集約ルート / 永続化モデル / Domain Service / Value Object / ReadModel）の振り分けが正しいか
- MECE チェックが必要な enum に適用されているか

## 観点 2: イベントとコマンドの整合性 → `strategic-ddd-designer`

- 全イベントに対応するコマンドがあるか（孤立イベントの検出。ただし外部システム起点のイベントは対応コマンドを持たない場合があり、これは正常）
- 全コマンドが集約に紐づいているか（浮遊コマンドの検出）
- イベントの消費者と副作用が明記されているか
- 暗黙のポリシー（イベント X → コマンド Y の自動化）が見落とされていないか

## 観点 3: ユビキタス言語の品質 → `strategic-ddd-designer`

- 同じ概念に複数の用語が使われていないか（同義語の検出）
- 既存コードベースの命名と矛盾していないか
- 定義が曖昧で複数解釈可能な用語がないか
- Phase 2-3 の議論内容と用語集が整合しているか

## 観点 4: コンテキスト境界の適切性 → `architect-reviewer`

- あるコンテキストが別コンテキストの関心事を持ち込んでいないか
- 関係パターン（Conformist / Published Language / Shared Kernel / ACL）の選択が妥当か
- 境界を越える通信が過剰でないか
- 新規コンテキストの必要性が論証されているか

## 観点 5: 設計全体の一貫性 → メインエージェント（統合）

- Phase 間で概念・用語が整合しているか（イベント一覧の用語 ↔ ユビキタス言語）
- 合意の経緯が記録されており、判断の根拠が追跡可能か
- 未決定事項に Phase 1-5 の論点が漏れなく記録されているか
- 問題空間のスコーピング（コア / サポーティング / ジェネリック）と設計の深さが整合しているか

## Finding Format Standard

すべての指摘で以下の形式を使用:

```
- `strategic-ddd.md:L{number}` — 指摘内容
  **関連 Phase**: Phase N
  **指摘元**: {エージェント名}
```

- 行番号が特定できない場合はセクション名で参照: `strategic-ddd.md (## 3. コマンドと集約)`
- 各指摘に**関連 Phase** を付記し、修正時にどの Phase を再実行すべきかを明示
- 各指摘に**指摘元**を付記し、どのエージェントが出した指摘かを追跡可能に

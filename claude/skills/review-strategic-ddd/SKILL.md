`/strategic-ddd` で作成した概念設計書をレビューし、`/design`（戦術設計・実装計画）に進む前に概念設計の品質を体系的に評価する。

**入力**: `./tmp/strategic-ddd.md`
**出力**: `./tmp/strategic-ddd-review.md`
**規約**: CLAUDE.md の Skills 共通規約に従う
## ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| `/strategic-ddd` 概念設計 | `/review-strategic-ddd` 設計レビュー | `/design` 実装計画 |

## 実行条件

- `./tmp/strategic-ddd.md` が存在すること

ファイルが存在しない場合はプロセスを即座に停止し、ユーザーに報告する。

## レビューの思考フレームワーク

DDD の概念設計に特化した視座で設計書を評価する:

- **ドメイン適合性**: モデルが実際のビジネスドメインを正確に反映しているか
- **モデルの表現力**: 集約・イベント・コマンドがビジネスルールを十分に表現できているか
- **境界の明確性**: コンテキスト境界が責務の漏洩なく引かれているか
- **言語の一貫性**: ユビキタス言語がコード・設計書・業務の間で統一されているか
- **実装可能性**: 概念設計が `/design` での戦術設計にスムーズに引き継げるか

## 5つのレビュー観点とサブエージェント割当

### デフォルト割当

| 観点 | 担当サブエージェント | 理由 |
|------|---------------------|------|
| 1. ドメインモデルの妥当性 | `ddd-expert` | 集約設計・Entity/VO 分類の専門家 |
| 2. イベントとコマンドの整合性 | `strategic-ddd-designer` | 戦略的 DDD の専門家、イベント/コマンド設計に精通 |
| 3. ユビキタス言語の品質 | `strategic-ddd-designer` | ユビキタス言語の定義・検証が専門領域 |
| 4. コンテキスト境界の適切性 | `architect-reviewer` | アーキテクチャ境界・統合パターンの評価が専門 |
| 5. 設計全体の一貫性 | メインエージェント（統合） | 各サブエージェントの結果を横断的に統合・矛盾検出 |

### プロジェクト固有サブエージェントの動的検出

プロジェクトによっては、ドメイン固有の知識を持つサブエージェントが用意されている場合がある（例: `hr-domain-expert`, `billing-reviewer` 等）。以下の手順で検出・活用する:

1. `.claude/agents/` および `.claude/subagents/` を Glob で走査し、description に DDD・ドメイン関連のキーワード（`domain`, `aggregate`, `bounded context`, `event`, `ubiquitous language` 等）を含むエージェントを検出する
2. 検出されたエージェントの description を読み、5つのレビュー観点のどれに関連するかを判定する
3. デフォルトのエージェントに**追加で**並列起動する（デフォルトを置き換えない）
4. 統合レビュー（観点 5）で、デフォルトエージェントとプロジェクト固有エージェントの指摘を統合する

検出されなかった場合はデフォルト割当のみで実行する。プロジェクト固有エージェントは必須ではない。

## レビュー観点の詳細

### 観点 1: ドメインモデルの妥当性 → `ddd-expert`

- 集約の粒度は適切か（God Aggregate / 過小集約）
- 不変条件が定義されており、ビジネスルールを網羅しているか
- 「動詞か名詞か」テストが適用されているか
- 概念分類（集約ルート / 永続化モデル / Domain Service / Value Object / ReadModel）の振り分けが正しいか
- MECE チェックが必要な enum に適用されているか

### 観点 2: イベントとコマンドの整合性 → `strategic-ddd-designer`

- 全イベントに対応するコマンドがあるか（孤立イベントの検出。ただし外部システム起点のイベントは対応コマンドを持たない場合があり、これは正常）
- 全コマンドが集約に紐づいているか（浮遊コマンドの検出）
- イベントの消費者と副作用が明記されているか
- 暗黙のポリシー（イベント X → コマンド Y の自動化）が見落とされていないか

### 観点 3: ユビキタス言語の品質 → `strategic-ddd-designer`

- 同じ概念に複数の用語が使われていないか（同義語の検出）
- 既存コードベースの命名と矛盾していないか
- 定義が曖昧で複数解釈可能な用語がないか
- Phase 2-3 の議論内容と用語集が整合しているか

### 観点 4: コンテキスト境界の適切性 → `architect-reviewer`

- あるコンテキストが別コンテキストの関心事を持ち込んでいないか
- 関係パターン（Conformist / Published Language / Shared Kernel / ACL）の選択が妥当か
- 境界を越える通信が過剰でないか
- 新規コンテキストの必要性が論証されているか

### 観点 5: 設計全体の一貫性 → メインエージェント（統合）

- Phase 間で概念・用語が整合しているか（イベント一覧の用語 ↔ ユビキタス言語）
- 合意の経緯が記録されており、判断の根拠が追跡可能か
- 未決定事項に Phase 1-5 の論点が漏れなく記録されているか
- 問題空間のスコーピング（コア / サポーティング / ジェネリック）と設計の深さが整合しているか

## 指摘の重要度

| レベル | 意味 |
|--------|------|
| ❌ Critical | 設計の修正が必須。このまま `/design` に進むと問題が生じる |
| ⚠️ Warning | 修正を推奨。モデルの品質・保守性に影響する |
| ℹ️ Note | 検討・確認を推奨。判断はユーザーに委ねる |
| 💡 代替案 | 検討に値する別のモデリングアプローチの提案 |
| ✅ Good | 設計の良い点（積極的に評価する） |

## Finding Format Standard

すべての指摘で以下の形式を使用する:

```
- `strategic-ddd.md:L{number}` — 指摘内容
  **関連 Phase**: Phase N
  **指摘元**: {エージェント名}
```

- 行番号が特定できない場合はセクション名で参照: `strategic-ddd.md (## 3. コマンドと集約)`
- 各指摘に**関連 Phase** を付記し、修正時にどの Phase を再実行すべきかを明示する
- 各指摘に**指摘元**を付記し、どのエージェントが出した指摘かを追跡可能にする

## 総合判定（3段階）

| 判定 | 意味 | 次のアクション |
|------|------|----------------|
| **LGTM** | 軽微な指摘のみ。`/design` に進んでよい | `/design` |
| **要修正** | 指摘事項を反映してから進むべき | `/strategic-ddd` で該当 Phase を再実行（指摘の「関連 Phase」を参照） |
| **要再設計** | 根本的なモデリングの見直しが必要 | `/strategic-ddd` を最初からやり直し |

## 実行プロセス

### 1. 設計書の読み込み

- [ ] `./tmp/strategic-ddd.md` を読み込む
- [ ] 全セクション（1〜6）の存在を確認する
- [ ] セクションが欠落している場合、ユーザーに報告し `/strategic-ddd` での補完を促す

### 2. 前調査とサブエージェント検出

- [ ] 既存コードの主要な命名・パターンを収集する（後続のサブエージェントに渡すコンテキスト）
- [ ] `.claude/agents/` および `.claude/subagents/` を Glob で走査し、ドメイン関連のサブエージェントを検出する
- [ ] 検出されたエージェントの description を読み、5つの観点への関連度を判定する
- [ ] 検出結果をユーザーに報告する: 「プロジェクト固有エージェント X, Y を検出しました。観点 N のレビューに追加で活用します」

### 3. サブエージェント並列レビュー

デフォルト3エージェント + 検出されたプロジェクト固有エージェントを `run_in_background: true` で並列起動する:

- **`ddd-expert`**: 観点 1（ドメインモデルの妥当性）を評価。設計書 + コードベースの集約パターンを入力として渡す
- **`strategic-ddd-designer`**: 観点 2-3（イベント/コマンド整合性 + ユビキタス言語品質）を評価。設計書 + コードベースの命名を入力として渡す
- **`architect-reviewer`**: 観点 4（コンテキスト境界の適切性）を評価。設計書 + コードベースのモジュール構造を入力として渡す
- **（検出されたプロジェクト固有エージェント）**: 判定された観点を評価。プロジェクト固有のドメイン知識を活かした指摘を出す

各エージェントには、指摘を `❌ Critical / ⚠️ Warning / ℹ️ Note / 💡 代替案 / ✅ Good` の重要度で分類し、`strategic-ddd.md:L{number}` 形式で参照を付けるよう指示する。

### 4. 統合レビュー

- [ ] 各サブエージェントの結果を収集する
- [ ] 観点 5（設計全体の一貫性）をメインエージェントが評価する
- [ ] デフォルトエージェントとプロジェクト固有エージェントの指摘を統合する
- [ ] サブエージェント間で矛盾する指摘があれば検出し、メインエージェントが判断して解消する

### 6. 完了報告

以下の内容をユーザーに報告する:

- **総合判定**（LGTM / 要修正 / 要再設計）
- 各区分の指摘件数
- 特に重要な指摘事項（Critical がある場合はその要約）
- 使用したサブエージェント一覧（デフォルト + プロジェクト固有）
- 次のステップの案内:
  - **LGTM の場合**: `/design` で実装計画に進む
  - **要修正の場合**: `/strategic-ddd` で該当 Phase を再実行
  - **要再設計の場合**: `/strategic-ddd` を最初からやり直し
- **リマインド**: 本スキルを新規インストールした場合は `./setup.sh install` を実行してください

`/strategic-ddd` で作成した概念設計書をレビューし、`/design`（戦術設計・実装計画）に進む前に概念設計の品質を体系的に評価する。

**入力**: `./tmp/strategic-ddd.md`
**出力**: `./tmp/strategic-ddd-review.md`
**規約**: CLAUDE.md の Skills 共通規約に従う

## 補助ドキュメントへの参照

| 補助ドキュメント | 読むタイミング |
|------------------|----------------|
| `./reference/review-checklist.md` | 5 観点ごとの詳細チェック項目を見る時 / Finding Format を確認する時 / レポート出力テンプレートを使う時 |

「念のため全部読む」は禁止。表のトリガー条件に該当する場合のみ読み込む。

## ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| `/strategic-ddd` 概念設計 | `/review-strategic-ddd` 設計レビュー | `/design` 実装計画 |

## 実行条件

- `./tmp/strategic-ddd.md` が存在すること

ファイルが存在しない場合はプロセスを即座に停止し、ユーザーに報告する。

## 5 つのレビュー観点とサブエージェント割当

### デフォルト割当

| 観点 | 担当サブエージェント | 理由 |
|------|---------------------|------|
| 1. ドメインモデルの妥当性 | `ddd-expert` | 集約設計・Entity/VO 分類の専門家 |
| 2. イベントとコマンドの整合性 | `strategic-ddd-designer` | 戦略的 DDD の専門家、イベント/コマンド設計に精通 |
| 3. ユビキタス言語の品質 | `strategic-ddd-designer` | ユビキタス言語の定義・検証が専門領域 |
| 4. コンテキスト境界の適切性 | `architect-reviewer` | アーキテクチャ境界・統合パターンの評価が専門 |
| 5. 設計全体の一貫性 | メインエージェント（統合） | 各サブエージェントの結果を横断的に統合・矛盾検出 |

各観点の詳細チェック項目は `./reference/review-checklist.md` を参照。

### プロジェクト固有サブエージェントの動的検出

プロジェクトによっては、ドメイン固有の知識を持つサブエージェントが用意されている場合がある（例: `hr-domain-expert`, `billing-reviewer` 等）。以下の手順で検出・活用する:

1. `.claude/agents/` および `.claude/subagents/` を Glob で走査し、description に DDD・ドメイン関連のキーワード（`domain`, `aggregate`, `bounded context`, `event`, `ubiquitous language` 等）を含むエージェントを検出する
2. 検出されたエージェントの description を読み、5 つのレビュー観点のどれに関連するかを判定する
3. デフォルトのエージェントに**追加で**並列起動する（デフォルトを置き換えない）
4. 統合レビュー（観点 5）で、デフォルトエージェントとプロジェクト固有エージェントの指摘を統合する

検出されなかった場合はデフォルト割当のみで実行する。プロジェクト固有エージェントは必須ではない。

## 指摘の重要度

| レベル | 意味 |
|--------|------|
| ❌ Critical | 設計の修正が必須。このまま `/design` に進むと問題が生じる |
| ⚠️ Warning | 修正を推奨。モデルの品質・保守性に影響する |
| ℹ️ Note | 検討・確認を推奨。判断はユーザーに委ねる |
| 💡 代替案 | 検討に値する別のモデリングアプローチの提案 |
| ✅ Good | 設計の良い点（積極的に評価する） |

Finding Format は `./reference/review-checklist.md` を参照。

## 総合判定（3 段階）

| 判定 | 意味 | 次のアクション |
|------|------|----------------|
| **LGTM** | 軽微な指摘のみ。`/design` に進んでよい | `/design` |
| **要修正** | 指摘事項を反映してから進むべき | `/strategic-ddd` で該当 Phase を再実行（指摘の「関連 Phase」を参照） |
| **要再設計** | 根本的なモデリングの見直しが必要 | `/strategic-ddd` を最初からやり直し |

## 実行プロセス

### 1. 設計書の読み込み

- [ ] `./tmp/strategic-ddd.md` を読み込む
- [ ] 全セクション（1〜6）の存在を確認する
- [ ] セクションが欠落している場合、ユーザーに報告し `/strategic-ddd` での補完を促す

### 2. 前調査とサブエージェント検出

- [ ] 既存コードの主要な命名・パターンを収集する（後続のサブエージェントに渡すコンテキスト）
- [ ] `.claude/agents/` / `.claude/subagents/` を Glob で走査し、ドメイン関連のサブエージェントを検出
- [ ] 検出されたエージェントの description を読み、5 つの観点への関連度を判定
- [ ] 検出結果をユーザーに報告: 「プロジェクト固有エージェント X, Y を検出。観点 N のレビューに追加で活用します」

### 3. サブエージェント並列レビュー

デフォルト 3 エージェント + 検出されたプロジェクト固有エージェントを `run_in_background: true` で並列起動する:

- **`ddd-expert`**: 観点 1 を評価。設計書 + コードベースの集約パターンを入力
- **`strategic-ddd-designer`**: 観点 2-3 を評価。設計書 + コードベースの命名を入力
- **`architect-reviewer`**: 観点 4 を評価。設計書 + コードベースのモジュール構造を入力
- **（検出されたプロジェクト固有エージェント）**: 判定された観点を評価

各エージェントには、指摘を 5 段階の重要度で分類し、`strategic-ddd.md:L{number}` 形式で参照を付けるよう指示する。

### 4. 統合レビュー

- [ ] 各サブエージェントの結果を収集
- [ ] 観点 5（設計全体の一貫性）をメインエージェントが評価
- [ ] デフォルトエージェントとプロジェクト固有エージェントの指摘を統合
- [ ] サブエージェント間で矛盾する指摘があれば検出し、メインエージェントが判断して解消

### 5. レポート生成

`./reference/review-checklist.md` のレポート出力テンプレートに従って `./tmp/strategic-ddd-review.md` に出力する。

### 6. 完了報告

以下の内容をユーザーに報告する:

- **総合判定**（LGTM / 要修正 / 要再設計）
- 各区分の指摘件数
- 特に重要な指摘事項（Critical がある場合はその要約）
- 使用したサブエージェント一覧（デフォルト + プロジェクト固有）
- 次のステップの案内:
  - **LGTM**: `/design` で実装計画に進む
  - **要修正**: `/strategic-ddd` で該当 Phase を再実行
  - **要再設計**: `/strategic-ddd` を最初からやり直し
- **リマインド**: 本スキルを新規インストールした場合は `./setup.sh install` を実行してください

## 実装計画ファイル

引数 `$ARGUMENTS` で指定したファイルを参照する。

## 補助ドキュメントへの参照

skill 詳細は同ディレクトリの `reference/*.md` に分離してある。状況に応じて読み込む:

| 補助ドキュメント | 読むタイミング |
|------------------|----------------|
| `./reference/tdd-policy.md` | 新規テストを書く時 / バグ修正ステップを始める時 / TDD ユーザーチェックポイントの判定をする時 |
| `./reference/review-cycle.md` | 各実装ステップ完了後（必須）/ 全体レビュー前 |
| `./reference/agent-selection.md` | Phase 2.5 でエージェント選択する時 / Medium/Large ステップで並列実行する時 |

「念のため全部読む」は禁止。表のトリガー条件に該当する場合のみ読み込む。

## 実行プロセス

### Phase 1: 実装計画の読込

- 実装計画ファイルを読み込み、すべてのコンテキストと要件を理解する
- 必要に応じて Web 検索とコードベース探索を追加で行う
- セマンティックなコード検索には Serena MCP、ライブラリ・フレームワークの使用方法には Context7 MCP を使用する

### Phase 2: 設計と計画

- すべての要件に対応する包括的な計画を作成する
- 複雑なタスクを小さく管理可能なフェーズに分解し、TodoWrite tool で実装計画を作成・追跡する
- 既存コードから従うべき実装パターンを特定する
- TDD で書くべきテストの種類（単体/統合）と失敗パターンをプラン内で明示する
- **各ステップにレビューサイクル（2 回）を組み込んだ計画を立てる**
- plan.md の各ステップに `- [ ]` チェックボックスがない場合は付与する

### Phase 2.5: エージェント選択

プロジェクトタイプと変更内容に基づき、専門エージェントを決定する。詳細は `./reference/agent-selection.md` を参照。

### Phase 3: 実装の実行（ステップごとにレビューサイクル適用）

各ステップで以下のサイクルを実行する。

#### 3.0 コンテキスト再読込

ステップ開始時に以下を再読込し、コンテキストドリフトを防止する:
1. **CLAUDE.md**: プロジェクト規約の最新状態
2. **実装計画ファイル**: 現在のステップの詳細と前後の依存関係
3. **関連仕様ファイル**: openspec/specs/ 配下の該当仕様（存在する場合）
4. **前ステップの出力**: 直前ステップのレビュー結果や修正内容

#### 3.1 ステップ実装

- 実装計画のステップを実行する
- セマンティックなコード検索には Serena MCP を使用する
- 実装は使用モデルに依存しない汎用的な記法で行う

**Small ステップ（変更ファイル < 3）**: メインエージェントが直接 TDD サイクルを実行する（3.2 に進む）。

**Medium/Large ステップ（変更ファイル >= 3）**: `./reference/agent-selection.md` の並列実行プロトコルと結果マージプロトコルに従う。

#### 3.2 TDD サイクル

新規テストを書く場合は `./reference/tdd-policy.md` の TDD サイクル詳細（① Red → ② Red 確認 → ③ Green → ④ Mutation → ⑤ Refactor）に従う。

バグ修正ステップの場合は `./reference/tdd-policy.md` の「バグ修正時のリグレッションテスト必須化」フローに従う。

#### 3.3 / 3.4 レビューサイクル 1 回目 / 2 回目

`./reference/review-cycle.md` のレビュー観点チェックリストと手順に従う。

#### 3.5 ステップ完了確認

- テストがすべて通ることを確認
- TodoWrite で該当ステップを completed にマーク
- plan.md 内の該当ステップ `- [ ]` を `- [x]` に更新
- 次のステップへ進む

### Phase 4: 最終検証

- format、lint を **変更対象ファイルのみ** に実行（L-1: プロジェクト全体への実行は禁止）
- test を実行し、すべての失敗を修正する
- typecheck は自動実行しない（L-2）。ユーザーに手動実行を促すか CI に委ねる
- ドメイン層・複雑ロジックの追加箇所は新規テストが緑になることを確認し、必要に応じて回帰テストも走らせる
- 受入基準が満たされていることを確認する

### Phase 5: AI コメントの整理

- 実装コード内のすべての AI 生成コメントをレビューする
- コードが既に表現している内容を単純に繰り返す冗長なコメントを削除する
- 特定の目的がない限り、コメントアウトされたコードブロックを削除する
- コード構造では表現できない価値あるコンテキストを提供するコメントのみ残す
- 原則: 「コードが言えないことをコメントし、コードが言わないことを単にコメントしない」（参考: https://github.com/97-things/97-things-every-programmer-should-know/tree/master/en/thing_16 ）

### Phase 5.5: ドキュメント同期

Phase 6 の全体レビュー前に、documenter エージェントを起動してドキュメントとコードの整合性を検証する。

1. documenter エージェントが全ステップの変更をまとめて検証
2. plan.md / openspec/specs/ との差分を検出 → 必要なら更新
3. レビューサイクルで発見された問題パターンを `tmp/lessons.md` に記録
4. 知識の圧縮（冗長な記録の統合・要約）

UI 変更がある場合は Phase 6.1 のチェック項目に「WCAG 2.1 AA 準拠」「デザインシステム整合性」を追加。
パフォーマンス最適化が該当する場合は Phase 6.1 に「N+1 クエリ」「不要な再レンダリング」を追加。

### Phase 6: 全体レビューサイクル（最終 2 回）

実装全体に対して **レビュー → 修正** を 2 回実行する。

**レビュー観点（両回共通）**: 全体俯瞰 / ステップ間整合性 / アーキテクチャ / パフォーマンス（N+1、不要な再レンダリング） / セキュリティ / UI 変更時は WCAG 2.1 AA + デザインシステム / ドキュメント (TSDoc) 整合性

**修正実行（両回共通）**: 発見した問題を修正 → 必要ならリファクタ → テスト再実行 → lint/format を **変更対象ファイルのみ** に実行（L-1）→ typecheck はユーザーに手動実行を促す（L-2）

### Phase 7: 完了報告

- すべてのチェックリスト項目が完了したことを確認する
- 実装計画を再度読み、すべてを実装したことを確認する
- レビューサイクルの実施記録を含めて完了ステータスを報告する

**tmp/ クリーンアップ:**
- `./tmp/step-*` ファイルを削除する（一時的なエージェント出力）
- `./tmp/lessons.md` は保持する
- `./tmp/plan.md` は保持する

**報告フォーマット**: 以下の項目を含む `## 実装完了報告` を出力する:

- 実装サマリー（ステップ数、レビューサイクル回数）
- エージェント実行サマリー（使用した専門エージェント、並列実行回数）
- TDD チェックポイント実施記録（ユーザー確認回数、再作成回数）
- 品質確認結果（lint / format / typecheck / test の各チェックリスト）
- レビューで発見・修正した主な問題
- 残存課題（あれば）

# レビュー・修正サイクル詳細（implement skill 補助）

各実装ステップ完了後、**レビュー → 修正** を **1 回必須で実施**する。1 回目で修正が発生した場合のみ、修正後に 2 回目レビューを実施する（修正ゼロなら 1 回で完了）。

## レビューサイクルの手順

```
ステップ実装完了
    ↓
【1 回目レビュー】セルフレビュー実行（必須）
    ↓
 ┌─修正発生あり─→【1 回目修正】指摘事項を修正
 │                  ↓
 │              【2 回目レビュー】修正後の再検証（条件付き）
 │                  ↓
 │              【2 回目修正】残存指摘があれば修正
 │                  ↓
 └─修正発生なし──────────────────→
                  ↓
        実装計画書の TODO リストに完了マークをつけて次のステップへ
```

### 発動条件の判定

「修正発生あり」とは以下のいずれかに該当する場合:

- レビュー観点チェックリストで 1 件以上の問題を検出
- リファクタリングが必要と判断
- テストの追加・修正が必要と判断
- lint / format 違反を発見

「修正発生なし」(= 1 回で完了) は:

- すべてのレビュー観点が問題なし
- 軽微な改善点のみで、修正コストが利益を上回らないと判断

## レビュー観点チェックリスト

各レビューでは以下の観点を確認する:

1. **コード品質**
   - 型安全性（any 型の使用禁止）
   - 命名規則の遵守
   - 単一責任原則の遵守
   - 不要なコメントの削除

2. **エラーハンドリング**
   - サイレント失敗がないか
   - 適切なエラーメッセージ
   - エッジケースの考慮

3. **プロジェクト規約**
   - CLAUDE.md の規約遵守
   - .cursor/rules/ の規約遵守
   - 既存コードパターンとの整合性

4. **テスト**
   - テストカバレッジの確認
   - テストケースの網羅性
   - テストの可読性
   - 偽陽性: 実装を壊してもテストが通り続けないか（アサーション不足の兆候）
   - 偽陰性: 期待値が仕様に基づいているか、過剰な厳密さで脆くなっていないか

## 1 回目レビュー手順（必須）

**レビュー実行**:
```
1. 変更したファイルの差分を確認（git diff）
2. レビュー観点チェックリストに基づいて自己レビュー
3. 問題点をリストアップ
```

**修正実行（修正発生時のみ）**:
```
1. リストアップした問題点を優先度順に修正
2. 修正内容を記録
3. lint/format を変更対象ファイルのみに実行して整合性確認
   ※ プロジェクト全体への lint/format 実行は禁止（L-1）
```

## 2 回目レビュー手順（条件付き: 1 回目で修正が発生した場合のみ）

**レビュー実行**:
```
1. 1 回目の修正結果を確認
2. 見落とした問題がないか再度チェック
3. 新たに発生した問題がないか確認
```

**修正実行（残存指摘があれば）**:
```
1. 残存する問題点を修正
2. 最終的なコード品質を確認
3. lint/format を変更対象ファイルのみに実行
   ※ typecheck は自動実行しない（L-2: 実行に数分かかりタイムアウトの原因になる）
```

## 設計思想（Opus 4.7 baseline）

Opus 4.7 はデフォルトで literal instruction following + self-verification が強化された。4.6 時代の「double-check 必須」scaffold は**逆効果**になりうるため、本サイクルでは:

- 1 回目を必須化（最低限のセーフティネット）
- 2 回目を条件付き化（修正発生時のみ → 無駄な反復を抑制）

これにより、修正が不要な単純ステップは 1 回で完了し、修正が必要な複雑ステップは確実に再検証される、という適応的な品質保証を実現する。

# レビュー・修正サイクル詳細（implement skill 補助）

各実装ステップ完了後、必ず **レビュー → 修正** を **2 回繰り返す** こと。

## レビューサイクルの手順

```
ステップ実装完了
    ↓
【1回目レビュー】セルフレビュー実行
    ↓
【1回目修正】指摘事項を修正
    ↓
【2回目レビュー】再度セルフレビュー実行
    ↓
【2回目修正】残存指摘事項を修正
    ↓

実装計画書に記載しているTODOリストに完了マークをつけて次のステップへ
```

## レビュー観点チェックリスト

各レビューでは以下の観点を確認する：

1. **コード品質**
   - 型安全性（any 型の使用禁止）
   - 命名規則の遵守
   - 単一責任原則の遵守
   - 不要なコメントの削除

2. **エラーハンドリング**
   - サイレント失敗がないか
   - 適切なエラーメッセージ
   - エッジケースの考慮

3. **プロジェクト規約**
   - CLAUDE.md の規約遵守
   - .cursor/rules/ の規約遵守
   - 既存コードパターンとの整合性

4. **テスト**
   - テストカバレッジの確認
   - テストケースの網羅性
   - テストの可読性
   - 偽陽性: 実装を壊してもテストが通り続けないか（アサーション不足の兆候）
   - 偽陰性: 期待値が仕様に基づいているか、過剰な厳密さで脆くなっていないか

## 1 回目レビュー手順

**レビュー実行**:
```
1. 変更したファイルの差分を確認（git diff）
2. レビュー観点チェックリストに基づいて自己レビュー
3. 問題点をリストアップ
```

**修正実行**:
```
1. リストアップした問題点を優先度順に修正
2. 修正内容を記録
3. lint/format を変更対象ファイルのみに実行して整合性確認
   ※ プロジェクト全体への lint/format 実行は禁止（L-1）
```

## 2 回目レビュー手順

**レビュー実行**:
```
1. 1回目の修正結果を確認
2. 見落とした問題がないか再度チェック
3. 新たに発生した問題がないか確認
```

**修正実行**:
```
1. 残存する問題点を修正
2. 最終的なコード品質を確認
3. lint/format を変更対象ファイルのみに実行
   ※ typecheck は自動実行しない（L-2: 実行に数分かかりタイムアウトの原因になる）
```

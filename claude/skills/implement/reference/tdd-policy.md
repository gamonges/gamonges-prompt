# TDD ポリシー（implement skill 補助）

## テスト駆動開発の基本

- すべての実装はテストから書き始め、Red → Green → リファクタのサイクルを徹底する
- ドメイン層（entity/command/repository/value）を変更・追加するときは TDD を必須とする
- 複雑なビジネスロジック、クエリ、あるいは外部 I/F に依存した振る舞いを追加する場合も必ずテストを先に書く
- 既存テストでカバーできない場合は統合テストやエンドツーエンドテストを追加し、仕様を守る証拠として残す

## テスト品質検証（偽陽性・偽陰性の防止）

新規テストを書いた場合、以下の検証を必ず行い、テストが本当に実装を検証していることを担保する。

**偽陽性（False Positive）の検出** — テストが常に通ってしまい、バグを見逃す状態:
- **Red フェーズ検証**: テストを書いた直後（実装前）に実行し、**必ず失敗する**ことを確認する。実装前に通るテストは無価値
- **ミューテーションチェック**: Green 到達後、実装の核心ロジック（条件分岐・計算・戻り値など）を一時的に壊して、テストが**失敗に転じる**ことを確認する。壊しても通るテストはアサーション不足

**偽陰性（False Negative）の検出** — 正しい実装なのにテストが落ちる状態:
- **環境依存の排除**: テスト結果が実行順序・タイミング・外部状態に依存していないか確認する
- **期待値の妥当性**: ハードコードされた期待値が仕様に基づいているか、実装の偶然の出力をコピーしただけでないか確認する
- **過剰な厳密さの回避**: 本質でない詳細（オブジェクトの全プロパティ一致、メッセージ文言の完全一致など）に依存していないか確認する

## バグ修正時のリグレッションテスト必須化

バグ修正ステップでは、リグレッションテストを必須とする。

**検出条件（以下のいずれかに該当）:**
- plan.md のステップ説明に「バグ修正」「不具合修正」「fix」「bug」「regression」を含む
- `./tmp/fix-plan.md` から生成されたステップである
- ユーザーが「バグ修正」と明示した場合

※ 単独の「修正」はバグ修正以外（設計修正、テスト修正等）にも該当するため検出条件に含めない。
  判断が曖昧な場合は AskUserQuestion でユーザーに確認する。

**必須テストフロー:**
1. **再現テスト**: バグを再現するテストを最初に書き、**失敗する**ことを確認
   - テスト名に `regression` または `bug` を含める
   - バグの発生条件を明確にコメントで記述
2. **修正実装**: バグを修正し、再現テストが**通る**ことを確認
3. **回帰テスト**: 修正箇所の周辺機能の既存テストを実行し、デグレがないことを確認
   - 同一モジュール内のテストスイートを全実行
   - 修正ファイルを import しているファイルのテストも実行
4. **ミューテーションチェック**: 修正を元に戻し、再現テストが再び**失敗する**ことを確認

## TDD ユーザーチェックポイント

テスト作成後にユーザー確認を挟み、テストが仕様を正しく表現しているか検証する。

**適用範囲（以下に該当するテストのみ）:**
- ドメイン層（entity/command/repository/value）のビジネスロジックテスト
- 仕様の解釈が曖昧で、複数の実装が考えられるテスト
- ユーザーが明示的にチェックを要求した場合

**適用しない:**
- CRUD の基本操作テスト
- 既存パターンを踏襲するテスト
- リファクタリング時の既存テスト維持

**フロー:**
① テストを作成する（Red フェーズ前）
② AskUserQuestion でテストコードをユーザーに提示する
   提示内容:
   - テストファイルのパスと作成したテストの要約
   - テストが検証する仕様項目
   - 「このテストは仕様を正しく表現していますか？」
③ ユーザーの応答:
   - 承認 → Red フェーズに進む
   - 修正指示 → テストを修正し、再度 ② に戻る

## TDD サイクル詳細（新規テスト作成時）

```
① Red: 仕様に基づいてテストを作成する
   - plan.md / openspec のステップ仕様からテストケースを導出する

   【ユーザーチェックポイント — 適用対象テストのみ】
   TDD ユーザーチェックポイントポリシーの適用対象に該当する場合:
   AskUserQuestion でテストコードをユーザーに提示し、承認を得る。
   非該当の場合はスキップして ② に進む。

② Red 確認: テスト実行 → 失敗を確認
   └─ 通ってしまった場合 → 偽陽性。テストを見直す
      （アサーション漏れ、テスト対象の未呼出、モックの過剰設定など）

③ Green: 最小の実装を書く → 実行 → 成功を確認
   └─ 失敗する場合 → 偽陰性の可能性を検証
      （期待値の妥当性、環境依存、過剰な厳密さを確認）

④ Mutation: 実装の核心を一時的に壊す → テスト実行 → 失敗を確認
   └─ まだ通る場合 → 偽陽性。アサーションを強化する
   └─ 確認後、壊した実装を元に戻す

⑤ Refactor: リファクタ → テスト実行 → 成功を確認
```

**バグ修正ステップの場合:** 上記サイクルの代わりに上記「バグ修正時のリグレッションテスト必須化」フローに従う。

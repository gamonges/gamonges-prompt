name: design
description: 要件・コンテキストから実装計画 plan.md を新規作成する。`/ask` 後の計画策定、新機能の設計、3 ステップ以上の非自明なタスク着手前、`/design` 呼び出しで使用。
---

要件定義やコンテキスト情報をもとに、実装計画を新規作成する。

**重要**: `./tmp/plan.md` またはCursorのplanモードで作成された実装計画書のみ作成・変更する。他のファイルは変更しない。
**規約**: CLAUDE.md の Skills 共通規約に従う
## パラメーター

`$ARGUMENTS` でコンテキストファイルのパスを指定できる。省略時は `./tmp/context.md` を使用する。

## ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| 要件定義・仕様検討 | `/design` 計画作成 | `/revise` 計画修正 → `/implement` 実装 |

## 実行条件

以下のファイルが存在することを確認する:

- `$ARGUMENTS`（指定がない場合は `./tmp/context.md`）— 要件・コンテキスト情報
- `./tmp/research.md`（任意）— `/ask` の調査結果。存在する場合は自動的に参照する
- `openspec/specs/`（任意）— 既存の仕様ファイル。存在する場合は自動的に参照し、整合性を考慮する
- `openspec/config.yaml`（任意）— プロジェクト固有のルール。存在する場合は計画策定に反映する

ファイルが存在しない場合:

- プロセスを即座に停止する
- どの条件が満たされていないかをユーザーに報告する
- 実装を続行しない

## 実行プロセス

### フェーズ 1: コンテキスト読み込み

- [ ] コンテキストファイル（`$ARGUMENTS` または `./tmp/context.md`）を読み込み、要件を把握する
- [ ] ファイル参照（`file_path:line_number` 形式）やコンテキスト情報を抽出する
- [ ] 要件を「機能要件」「非機能要件」「制約事項」に分類する
- [ ] `./tmp/research.md` が存在する場合、調査結果として追加コンテキストに含める
- [ ] `openspec/specs/` が存在する場合、関連する既存仕様を参照し、新計画との整合性を考慮する
- [ ] `openspec/config.yaml` が存在する場合、プロジェクト固有のルールを参照する

### フェーズ 1.5: 既存 plan.md の確認

`./tmp/plan.md` が既に存在する場合、**プロセスを即座に停止**してユーザーに確認する:

- 既存の plan.md の概要（タイトル・ステップ数）を表示する
- 以下の選択肢を提示する:
  1. **上書き**: 既存の plan.md を破棄し、新しい計画で置き換える
  2. **追記**: 既存の plan.md の末尾に新しい計画を追加する（複数機能を1ファイルで管理する場合）
  3. **キャンセル**: 計画作成を中止する

ユーザーの選択に従って後続のフェーズを実行する。

### フェーズ 2: ユーザー確認（必要に応じて）

以下の問題を検出した場合、**プロセスを即座に停止**してユーザーに報告する:

- **曖昧な要件**: 複数の解釈が可能な記述がある場合
  - 具体的にどの要件が曖昧かを提示し、質問する
- **矛盾する要件**: 要件間で整合性がとれない場合
  - 矛盾する要件を並べて提示し、優先度を確認する
- **技術的制約が不明**: 実装方式に影響する制約が特定できない場合
  - 不足している情報を具体的に列挙して確認する

複数の問題がある場合のフォーマット:

```
## 問題 1: [問題の要約]
[該当する要件の引用と、何が問題なのかの説明]

## 問題 2: [問題の要約]
[該当する要件の引用と、何が問題なのかの説明]
```

### フェーズ 3: コードベース調査

サブエージェントを使って**並列に**調査を実施する:

- [ ] **既存パターンの調査**: Serena MCP でセマンティックなコード検索を行い、参考実装やパターンを発見する
- [ ] **影響範囲の分析**: 変更対象となるファイル・モジュールの依存関係を調査する
- [ ] **外部情報の調査**: Web search tool で外部ドキュメントや事例を検索する
- [ ] **ライブラリ情報**: Context7 MCP でライブラリ・フレームワークの使用方法を確認する

### フェーズ 4: 実装計画の策定

- [ ] 要件を実装ステップに分解する
- [ ] 各ステップの影響範囲・変更対象ファイルを特定する
- [ ] テスト方針を明記する（TDD で書くべきテストの種類: 単体 / 統合）
- [ ] **プロジェクト固有規約を参照する**: 以下の階層的フォールバックで規約を取得し、計画策定時の考慮事項として反映する。該当する規約が見つかったら、それ以降のフォールバックは行わない:
  1. **リポジトリ root の `CLAUDE.md`** に `## プロジェクト固有規約` セクション（または `## プロジェクト固有規約（/design 用）`）があれば、そのセクション内容を参照する
  2. **`openspec/config.yaml`** に `rules.design` キーがあれば、その値を参照する（既存の `/spec-propose` の参照パターンと整合）
  3. どちらも見つからなければ、汎用 best practices（DRY / YAGNI / SOLID / 「シンプルさ第一・最小限のインパクト」）を適用する

  **`## プロジェクト固有規約` セクションのフォーマット**（推奨）: H2 ヘッダ + 太字キー + 箇条書きの自由文。例:

  ```markdown
  ## プロジェクト固有規約（/design 用）

  - **アーキテクチャ**: レイヤード, CQRS
  - **エラーハンドリング**: Result 型, ErrorWithDisplayMessages
  - **多言語対応**: en, ja, id, vn, th, zhCN
  - **クロステナントアクセス防止**: 必須
  ```

  代替フォーマットとして、上記セクション内にコードブロック形式で YAML を記述する形式も受け付ける（`architecture: [layered, cqrs]` のような構造化キー）。フォーマット崩れや内容不在の場合は次の階層へフォールバックする。
- [ ] 各ステップの依存関係と実行順序を決定する
- [ ] **要件カバレッジを検証する**: すべての要件が実装ステップでカバーされているか確認し、漏れがあれば追加する
- [ ] **任意セクションの該当判定**: フェーズ 5 のテンプレートには `[該当時]` 指示の任意セクションがあり、以下の判定ルールに従って出力するセクションを決定する。該当しない場合はセクションごと省略する（空セクションを残さない）:
  - **公開インターフェース**: 公開 API・型エクスポート・DB スキーマのいずれかが変わる場合に出力
  - **データモデル変更**: DB マイグレーションが発生する場合に出力
  - **ロールバック方針**: 本番影響のある変更（API 変更・データ移行等）の場合に出力
  - **全体テスト戦略**: E2E / CI パイプライン / テスト環境構築 / テストデータ戦略 など、ステップ単位を横断するテスト事項がある場合に出力（ステップ内の「テスト方針」とは切り分ける）
  - **マイルストーン**: 実装ステップが 10 を超える、または段階リリースの単位がある場合に出力
- [ ] **確信度評価**: 各根本原因 / 主要設計判断について以下を必須記述する
  - `**確信度: high / medium / low**`
  - **low** の場合は「**検証ステップ**」を併記必須（実コード照合、実行確認、調査エージェント起動など）
  - 「low なまま実装に進まない」原則を徹底し、検証ステップの結果で再評価する

### フェーズ 5: plan.md の出力

`./tmp/plan.md` に以下の構成で実装計画を出力する。各セクションの **[必須] / [任意] / [該当時]** 指示に従い、該当しない場合はセクションごと省略する（空セクションを残さない）。該当判定ルールはフェーズ 4 を参照:

````markdown
# 実装計画: [機能名]

## TL;DR
<!-- [任意] 1〜3 行で全体像を要約。/status や PR 本文への転用を意識 -->

## 概要
<!-- [必須] 実装する機能の概要と目的 -->

## 前提条件
<!-- [必須] 実装に必要な前提知識・環境 -->

## アンチゴール
<!-- [必須] このタスクではやらないこと。スコープ境界を明示し、過剰実装と review のノイズ指摘を防ぐ -->
- ...

## ファイル変更マップ
<!-- [必須] 変更ファイルをディレクトリ構成で俯瞰する。ステップ駆動の縦割りでは見えない全体像を提供 -->

凡例: `[NEW]` 新規 / `[MOD]` 編集 / `[DEL]` 削除 / `[REN]` リネーム

```
project-root/
├── src/
│   └── ...                            [MOD]  ステップ N
└── ...
```

### ファイル一覧表
| パス | 種別 | 役割 | ステップ | 概算行数 |
|------|------|------|----------|---------|

## 公開インターフェース
<!-- [該当時] 公開 API・型エクスポート・DB スキーマのいずれかが変わる場合 -->

## データモデル変更
<!-- [該当時] DB マイグレーションが発生する場合 -->

## ステップ依存関係
<!-- [必須] ステップ間の前後関係を ASCII グラフで表現。並列実行可否とブロッカーを明示 -->

```
ステップ 1 ──► ステップ 2 ──► ...
```

## 実装ステップ
<!-- [必須] -->

### ステップ 1: [ステップ名]
- 変更対象ファイル: ...
- テスト方針: ...
- 実装手段: ...（必要に応じて Edit/Write の使い分けや変更単位を明示）
- 詳細: ...

### ステップ 2: [ステップ名]
...

## 受入条件
<!-- [必須] Definition of Done をチェックリスト形式で。/review のチェックリストも兼ねる -->
- [ ] ...

## ロールバック方針
<!-- [該当時] 本番影響のある変更（API 変更・データ移行等）の場合 -->

## 全体テスト戦略
<!-- [該当時] E2E / CI パイプライン / テスト環境構築 / テストデータ戦略 など、ステップ単位を横断するテスト事項がある場合のみ。ステップ内の「テスト方針」と切り分け -->

## 確信度サマリ
<!-- [必須] 各ステップ・設計判断の確信度と検証ステップ -->

| 確信度 | ステップ / 設計判断 | 検証ステップ |
|--------|--------------------|-------------|
| high   | ステップ 1, 2 | — |
| medium | ステップ 3 | コード照合 |
| low    | ステップ 4 | 実行確認、依存ライブラリ動作検証 |

> **low confidence のステップは `/review-plan` で優先確認される**。
> 検証ステップを実施した結果に応じて確信度を再評価し、low のまま実装に進まない。

## オープン質問
<!-- [必須] 未解決事項。空でも見出しを残す（解決済み事項は ✅ 履歴として残してよい）-->
- [ ] ...

## 検討した代替案
<!-- [任意] 採用しなかった案を 1〜2 行で残し、`/review-plan` の「なぜこの設計？」往復を減らす -->

## フォローアップタスク
<!-- [任意] 本タスクのスコープ外で、後で別途起票したい作業 -->
- F-1: ...

## マイルストーン
<!-- [該当時] 実装ステップが 10 を超える、または段階リリースの単位がある場合 -->
- M1 (ステップ 1-N): ...
````

> 状態遷移が複雑な場合は、本コマンド完了後に `/state-machine` を実行して状態遷移図を後付けで追加できる。

#### 記入例（特に抽象的なセクション）

**アンチゴール（必須）**:

```
- 既存の `LegacyService` のリファクタリング — 別 PR で扱う
- UI の i18n 対応追加 — バックエンドのみ対応
- パフォーマンスチューニング — 動作確認のみ
```

**ファイル変更マップ・ディレクトリツリー（必須）**:

小規模変更の例:

```
src/
├── domain/user/User.ts                       [MOD]  ステップ 2
└── domain/user/UserStatus.ts                 [NEW]  ステップ 1
```

大規模変更の例（ステップ範囲表記を活用）:

```
src/
├── domain/user/                              [MOD]  ステップ 2-3
├── application/user/                         [NEW]  ステップ 4-5
├── infrastructure/prisma/migrations/         [NEW]  ステップ 6
└── ui/admin/                                 [MOD]  ステップ 7-8
```

**ステップ依存関係（必須）**:

```
ステップ 1 (enum 定義) ──┬──► ステップ 2 (集約) ──► ステップ 3 (UseCase) ──► ステップ 5 (Controller)
                          └──► ステップ 4 (Migration) ──────────────────────┘
```

**オープン質問（必須・空でも見出しを残すパターン）**:

未解決事項がない場合も以下のように見出しを残し、過去のオープン質問の解決履歴を `✅` で残してよい:

```
## オープン質問

現時点で未解決のオープン質問はありません。/design 実行時に提示された質問はすべて解決済み:

- ✅ XXX の参照優先順位 → A → B の順で確定（ステップ N に反映）
- ✅ ロールバック方針の出力タイミング → 該当時のみ（ステップ 2 に反映）
```

**確信度サマリの low ステップ（必須）**:

low confidence のステップは検証ステップを必ず併記する:

```
| low | ステップ 4（外部 API 連携） | スパイク実装で 1 リクエスト送信し、レート制限とレスポンス形式を確認 |
```

### フェーズ 6: 完了報告

以下の内容をユーザーに報告する:

- 作成した計画の要約（ステップ数、ファイル変更マップの要約）
- 計画策定中に判断した設計上のポイント
- 確信度サマリの medium/low 件数と検証ステップの有無
- 次のステップの案内:
  - `/review-plan` で計画レビュー（推奨、特に medium/low 確信度がある場合）
  - `/revise` で計画修正
  - `/implement` で実装開始

要件定義やコンテキスト情報をもとに、実装計画を新規作成する。

**重要**: `./tmp/plan.md` または Cursor の plan モードで作成された実装計画書のみ作成・変更する。他のファイルは変更しない。
**規約**: CLAUDE.md の Skills 共通規約に従う

## 補助ドキュメントへの参照

| 補助ドキュメント | 読むタイミング |
|------------------|----------------|
| `./reference/phase-details.md` | フェーズ 1-4 の調査・確認手順 / 任意セクション該当判定ルール / 記入例を確認する時 / フェーズ 4 で 6 カテゴリチェックを実施する時 |
| `./reference/acceptance-criteria.md` | フェーズ 5 で受入条件（Acceptance Criteria）を記述する時 / Given/When/Then 形式のテンプレを使う時 |

「念のため全部読む」は禁止。表のトリガー条件に該当する場合のみ読み込む。

## パラメーター

`$ARGUMENTS` でコンテキストファイルのパスを指定できる。省略時は `./tmp/context.md` を使用する。

## ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| 要件定義・仕様検討 | `/design` 計画作成 | `/revise` 計画修正 → `/implement` 実装 |

## 実行条件

以下のファイルを参照する:

- `$ARGUMENTS`（指定がない場合は `./tmp/context.md`）— 要件・コンテキスト情報
- `./tmp/research.md`（任意）— `/ask` の調査結果
- `openspec/specs/`（任意）— 既存仕様
- `openspec/config.yaml`（任意）— プロジェクト固有ルール

コンテキストファイルが存在しない場合: プロセスを即座に停止し、ユーザーに報告する。

## 実行プロセス

### フェーズ 1: コンテキスト読み込み

要件を「機能要件」「非機能要件」「制約事項」に分類し、`./tmp/research.md` / `openspec/specs/` / `openspec/config.yaml` を参照する。詳細は `./reference/phase-details.md`。

### フェーズ 1.5: 既存 plan.md の確認

`./tmp/plan.md` が既に存在する場合、**プロセスを即座に停止**してユーザーに「上書き / 追記 / キャンセル」を確認する。詳細は `./reference/phase-details.md`。

### フェーズ 2: ユーザー確認（必要に応じて）

曖昧な要件 / 矛盾する要件 / 技術的制約が不明 を検出したら停止してユーザーに報告。詳細は `./reference/phase-details.md`。

### フェーズ 3: コードベース調査

調査対象が複数モジュールに渡る・複数の独立した観点を並行調査する場合は、サブエージェントを同一ターンで複数 spawn して並列に調査する。対象が少数の明確なファイルに収まる場合は直接読む。以下の観点を調査する:

- [ ] **既存パターン**: Serena MCP でセマンティック検索
- [ ] **影響範囲**: 変更対象ファイル・モジュールの依存関係調査
- [ ] **外部情報**: Web search で外部ドキュメント・事例検索
- [ ] **ライブラリ情報**: Context7 MCP で使用方法確認

### フェーズ 4: 実装計画の策定

- [ ] 要件を実装ステップに分解
- [ ] 各ステップの影響範囲・変更対象ファイルを特定
- [ ] テスト方針を明記
- [ ] **プロジェクト固有規約を参照**（階層的フォールバック、詳細は `./reference/phase-details.md`）
- [ ] 各ステップの依存関係と実行順序を決定
- [ ] **要件カバレッジを検証**
- [ ] **任意セクションの該当判定**（詳細は `./reference/phase-details.md`）
- [ ] **構造的欠陥防止チェック（6 カテゴリ）**:
  - [ ] **統合点トレース**: skip / 削除 / 短絡提案の副作用を 1 件ずつ列挙
  - [ ] **Decision 間整合性**: Decision を 3 個以上立てたら相互整合性マトリクスを必須化
  - [ ] **並列性マトリクス**: 並列子実行を伴う feature は共有リソース × 並列度を必須記述
  - [ ] **「変更不要」根拠**: 「変更不要」「透過」と書く箇所には grep / read 根拠を併記
  - [ ] **HTTP contract concrete 化**: 抽象 IF を書いたら curl で叩ける具体例を併記
  - [ ] **完了パスツリー**: feature の完了パスを tree で書き出し各葉に AC を割り当て

  > 詳細テンプレートと BAD/GOOD 例は `./reference/phase-details.md` の「構造的欠陥防止チェックリスト」を参照。軽微な計画 / 該当しない場合は各項目「該当なし」と明記して OK。
- [ ] **確信度評価**（high / medium / low + low には検証ステップを必須記述）

### フェーズ 5: plan.md の出力

`./tmp/plan.md` に以下の構成で実装計画を出力する。各セクションの **[必須] / [任意] / [該当時]** 指示に従い、該当しない場合はセクションごと省略する（空セクションを残さない）。該当判定ルールは `./reference/phase-details.md` を参照:

````markdown
# 実装計画: [機能名]

## TL;DR
<!-- [任意] 1〜3 行で全体像を要約。/status や PR 本文への転用を意識 -->

## 概要
<!-- [必須] 実装する機能の概要と目的 -->

## 前提条件
<!-- [必須] 実装に必要な前提知識・環境 -->

## アンチゴール
<!-- [必須] このタスクではやらないこと。スコープ境界を明示し、過剰実装と review のノイズ指摘を防ぐ -->
- ...

## ファイル変更マップ
<!-- [必須] 変更ファイルをディレクトリ構成で俯瞰する。ステップ駆動の縦割りでは見えない全体像を提供 -->

凡例: `[NEW]` 新規 / `[MOD]` 編集 / `[DEL]` 削除 / `[REN]` リネーム

```
project-root/
├── src/
│   └── ...                            [MOD]  ステップ N
└── ...
```

### ファイル一覧表
| パス | 種別 | 役割 | ステップ | 概算行数 |
|------|------|------|----------|---------|

## 公開インターフェース
<!-- [該当時] 公開 API・型エクスポート・DB スキーマのいずれかが変わる場合 -->

## データモデル変更
<!-- [該当時] DB マイグレーションが発生する場合 -->

## ステップ依存関係
<!-- [必須] ステップ間の前後関係を ASCII グラフで表現。並列実行可否とブロッカーを明示 -->

```
ステップ 1 ──► ステップ 2 ──► ...
```

## 実装ステップ
<!-- [必須] -->

### ステップ 1: [ステップ名]
- 変更対象ファイル: ...
- テスト方針: ...
- 実装手段: ...（必要に応じて Edit/Write の使い分けや変更単位を明示）
- 詳細: ...

### ステップ 2: [ステップ名]
...

## 受入条件（Acceptance Criteria）
<!-- [必須] Definition of Done を Given/When/Then または「テストケース名 / 期待出力」形式で。/review のチェックリストを兼ねる。詳細とテンプレは `./reference/acceptance-criteria.md` を参照 -->
- [ ] **AC-1**: {ユーザーが {状況} で {操作} すると、{期待結果} になる}
  - **Given**: ...
  - **When**: ...
  - **Then**: ...
  - **検証ステップ**: テストケース名や手動確認手順

## ロールバック方針
<!-- [該当時] 本番影響のある変更（API 変更・データ移行等）の場合 -->

## 全体テスト戦略
<!-- [該当時] E2E / CI パイプライン / テスト環境構築 / テストデータ戦略 など、ステップ単位を横断するテスト事項がある場合のみ。ステップ内の「テスト方針」と切り分け -->

## 確信度サマリ
<!-- [必須] 各ステップ・設計判断の確信度と検証ステップ -->

| 確信度 | ステップ / 設計判断 | 検証ステップ |
|--------|--------------------|-------------|
| high   | ステップ 1, 2 | — |
| medium | ステップ 3 | コード照合 |
| low    | ステップ 4 | 実行確認、依存ライブラリ動作検証 |

> **low confidence のステップは `/review-plan` で優先確認される**。
> 検証ステップを実施した結果に応じて確信度を再評価し、low のまま実装に進まない。

## オープン質問
<!-- [必須] 未解決事項。空でも見出しを残す（解決済み事項は ✅ 履歴として残してよい）-->
- [ ] ...

## 検討した代替案
<!-- [任意] 採用しなかった案を 1〜2 行で残し、`/review-plan` の「なぜこの設計？」往復を減らす -->

## フォローアップタスク
<!-- [任意] 本タスクのスコープ外で、後で別途起票したい作業 -->
- F-1: ...

## マイルストーン
<!-- [該当時] 実装ステップが 10 を超える、または段階リリースの単位がある場合 -->
- M1 (ステップ 1-N): ...
````

> 状態遷移が複雑な場合は、本コマンド完了後に `/state-machine` を実行して状態遷移図を後付けで追加できる。

記入例（アンチゴール / ファイル変更マップ / ステップ依存関係 / オープン質問 / 確信度サマリ low）は `./reference/phase-details.md` を参照。

### フェーズ 6: 完了報告

以下の内容をユーザーに報告する:

- 作成した計画の要約（ステップ数、ファイル変更マップの要約）
- 計画策定中に判断した設計上のポイント
- 確信度サマリの medium/low 件数と検証ステップの有無
- 次のステップの案内:
  - `/review-plan` で計画レビュー（推奨、特に medium/low 確信度がある場合）
  - `/revise` で計画修正
  - `/implement` で実装開始

#### HTML view 化 (オプション)

完了報告に続けて、以下をユーザーに尋ねる:

> **HTML 化しますか?** (人間レビュア向けのデザイン HTML を生成)

ユーザーが Yes と回答した場合、**Claude は自動実行せず**、次のコマンドを案内する:

```
/html-view ./tmp/plan.md
```

ユーザーが明示的に slash command を入力することで HTML 生成 + ブラウザ自動起動が完了する。

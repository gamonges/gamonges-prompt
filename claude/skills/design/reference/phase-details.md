# /design の各フェーズ詳細手順

> SKILL.md フェーズ 1-4 の調査・確認手順、任意セクション該当判定ルール、確信度評価の指針。

## フェーズ 1: コンテキスト読み込み

- [ ] コンテキストファイル（`$ARGUMENTS` または `./tmp/context.md`）を読み込み、要件を把握
- [ ] ファイル参照（`file_path:line_number`）やコンテキスト情報を抽出
- [ ] 要件を「機能要件」「非機能要件」「制約事項」に分類
- [ ] `./tmp/research.md` が存在する場合、調査結果として追加コンテキストに含める
- [ ] `openspec/specs/` が存在する場合、関連する既存仕様を参照し、新計画との整合性を考慮
- [ ] `openspec/config.yaml` が存在する場合、プロジェクト固有のルールを参照

## フェーズ 1.5: 既存 plan.md の確認

`./tmp/plan.md` が既に存在する場合、**プロセスを即座に停止**してユーザーに確認:

- 既存 plan.md の概要（タイトル・ステップ数）を表示
- 選択肢を提示:
  1. **上書き**: 既存 plan.md を破棄し新計画で置き換え
  2. **追記**: 既存末尾に新計画を追加（複数機能を 1 ファイル管理する場合）
  3. **キャンセル**: 計画作成を中止

ユーザー選択に従って後続フェーズを実行。

## フェーズ 2: ユーザー確認（必要に応じて）

以下を検出した場合、**プロセスを即座に停止**してユーザーに報告:

- **曖昧な要件**: 複数解釈が可能 → どの要件が曖昧かを提示し質問
- **矛盾する要件**: 整合性が取れない → 矛盾する要件を並べて優先度確認
- **技術的制約が不明**: 実装方式に影響する制約が特定できない → 不足情報を列挙

複数の問題がある場合:

```
## 問題 1: [問題の要約]
[該当する要件の引用と、何が問題なのかの説明]

## 問題 2: [問題の要約]
...
```

## フェーズ 3: コードベース調査

サブエージェントを使って**並列に**調査:

- [ ] **既存パターンの調査**: Serena MCP でセマンティック検索、参考実装やパターンを発見
- [ ] **影響範囲の分析**: 変更対象ファイル・モジュールの依存関係を調査
- [ ] **外部情報の調査**: Web search で外部ドキュメントや事例を検索
- [ ] **ライブラリ情報**: Context7 MCP でライブラリ・フレームワークの使用方法を確認

## フェーズ 4: 実装計画の策定

### 基本作業

- [ ] 要件を実装ステップに分解
- [ ] 各ステップの影響範囲・変更対象ファイルを特定
- [ ] テスト方針を明記（TDD で書くべきテストの種類: 単体 / 統合）
- [ ] 各ステップの依存関係と実行順序を決定
- [ ] **要件カバレッジを検証**: すべての要件が実装ステップでカバーされているか確認

### プロジェクト固有規約の参照

階層的フォールバックで規約を取得し、計画策定時の考慮事項として反映する。該当する規約が見つかったら、それ以降のフォールバックは行わない:

1. **リポジトリ root の `CLAUDE.md`** に `## プロジェクト固有規約` セクション（または `## プロジェクト固有規約（/design 用）`）があれば、そのセクション内容を参照
2. **`openspec/config.yaml`** に `rules.design` キーがあれば、その値を参照（既存の `/spec-propose` と整合）
3. どちらも見つからなければ、汎用 best practices（DRY / YAGNI / SOLID / 「シンプルさ第一・最小限のインパクト」）を適用

#### `## プロジェクト固有規約` セクションのフォーマット（推奨）

H2 ヘッダ + 太字キー + 箇条書きの自由文:

```markdown
## プロジェクト固有規約（/design 用）

- **アーキテクチャ**: レイヤード, CQRS
- **エラーハンドリング**: Result 型, ErrorWithDisplayMessages
- **多言語対応**: en, ja, id, vn, th, zhCN
- **クロステナントアクセス防止**: 必須
```

代替: コードブロック内 YAML 形式（`architecture: [layered, cqrs]` のような構造化キー）も受け付ける。フォーマット崩れや内容不在の場合は次の階層へフォールバック。

### 任意セクションの該当判定

フェーズ 5 テンプレートには `[該当時]` 指示の任意セクションがある。以下の判定で出力セクションを決定する。該当しない場合はセクションごと省略する（空セクションを残さない）:

- **公開インターフェース**: 公開 API・型エクスポート・DB スキーマのいずれかが変わる場合に出力
- **データモデル変更**: DB マイグレーションが発生する場合に出力
- **ロールバック方針**: 本番影響のある変更（API 変更・データ移行等）の場合に出力
- **全体テスト戦略**: E2E / CI パイプライン / テスト環境構築 / テストデータ戦略 など、ステップ横断のテスト事項がある場合に出力（ステップ内「テスト方針」と切り分け）
- **マイルストーン**: 実装ステップが 10 を超える、または段階リリースの単位がある場合に出力

### 確信度評価

各根本原因 / 主要設計判断について必須記述:

- `**確信度: high / medium / low**`
- **low** の場合は「**検証ステップ**」を併記必須（実コード照合、実行確認、調査エージェント起動など）
- 「low なまま実装に進まない」原則を徹底し、検証ステップの結果で再評価

## 記入例（特に抽象的なセクション）

### アンチゴール

```
- 既存の `LegacyService` のリファクタリング — 別 PR で扱う
- UI の i18n 対応追加 — バックエンドのみ対応
- パフォーマンスチューニング — 動作確認のみ
```

### ファイル変更マップ・ディレクトリツリー

小規模変更の例:

```
src/
├── domain/user/User.ts                       [MOD]  ステップ 2
└── domain/user/UserStatus.ts                 [NEW]  ステップ 1
```

大規模変更の例（ステップ範囲表記を活用）:

```
src/
├── domain/user/                              [MOD]  ステップ 2-3
├── application/user/                         [NEW]  ステップ 4-5
├── infrastructure/prisma/migrations/         [NEW]  ステップ 6
└── ui/admin/                                 [MOD]  ステップ 7-8
```

### ステップ依存関係

```
ステップ 1 (enum 定義) ──┬──► ステップ 2 (集約) ──► ステップ 3 (UseCase) ──► ステップ 5 (Controller)
                          └──► ステップ 4 (Migration) ──────────────────────┘
```

### オープン質問（空でも見出しを残すパターン）

未解決事項がない場合も見出しを残し、過去のオープン質問の解決履歴を `✅` で残してよい:

```
## オープン質問

現時点で未解決のオープン質問はありません。/design 実行時に提示された質問はすべて解決済み:

- ✅ XXX の参照優先順位 → A → B の順で確定（ステップ N に反映）
- ✅ ロールバック方針の出力タイミング → 該当時のみ（ステップ 2 に反映）
```

### 確信度サマリの low ステップ

low confidence のステップは検証ステップを必ず併記する:

```
| low | ステップ 4（外部 API 連携） | スパイク実装で 1 リクエスト送信し、レート制限とレスポンス形式を確認 |
```

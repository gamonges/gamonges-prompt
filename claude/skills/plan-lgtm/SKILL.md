name: plan-lgtm
description: review-plan と revise を総合判定が LGTM になるまで自動的に繰り返す。計画レビューの反復・修正サイクルの自動化、`/plan-lgtm` 呼び出しで使用。
---

`review-plan ⇄ revise` の LGTM ループを 1 コマンドで実行する。計画ファイルに対して「レビュー → 判定 → 必要なら修正 → 再レビュー」を、総合判定が `LGTM` になるまで自動的に繰り返す。

**重要**: 本スキル自体はソースコードを変更しない。内部で呼び出す `review-plan`/`revise` がそれぞれの既定出力（`./tmp/plan-review.md` の生成、対象計画ファイルの上書き）を行う。
**規約**: CLAUDE.md の Skills 共通規約に従う

## パラメーター

`$ARGUMENTS` で対象計画ファイルのパスを指定できる。省略時は `./tmp/plan.md` を使用する（`revise`/`review-plan` と同じ「ファイルパス」規約に統一）。Cursor 環境の場合は plan モードで作成した `***.plan.md` を対象ファイルパスとして指定できる。

## ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| `/design` 計画作成 or `/fix` 修正計画作成 | `/plan-lgtm` LGTM ループ | `/implement` 実装開始 |

## 実行条件

以下のファイルが存在することを確認する:

- `$ARGUMENTS`（指定がない場合は `./tmp/plan.md`。`$ARGUMENTS` が空文字列の場合も「指定なし」として扱い `./tmp/plan.md` にフォールバックする）— 対象計画ファイル

存在しない場合:

- プロセスを即座に停止する
- 対象ファイルが存在しない旨をユーザーに報告する
- ループを開始しない

## 実行プロセス

### フェーズ 1: 初期化

- [ ] 対象計画ファイルパスを確定する（`$ARGUMENTS` または `./tmp/plan.md`）
- [ ] 反復カウンタを 0 に設定する
  - **反復カウンタの定義**: 対象計画ファイルの更新を伴う `revise` 呼び出し成功回数を数える（`review-plan` の呼び出し回数ではない。`revise` が対象ファイルを更新しないまま終わった呼び出しは含まない。判定方法はフェーズ2手順4を参照）。初回の `review-plan` 呼び出しはカウントに含めない。`revise` が理由あってプロセスを一時停止し応答後に再開した場合も、同一呼び出しとして1回のみカウントし、追加増分は行わない
  - **上限 N = 3**: `revise` を最大 3 回まで呼び出す。値を変更したい場合はこのファイルの本節の数値を直接編集する

### フェーズ 2: ループ本体

本フェーズは条件分岐を含む順序依存の判定フローであり、独立した確認項目の列挙ではないため `- [ ]` 形式を用いない。

各反復で以下を実施する:

1. `review-plan` 呼び出し前の `./tmp/plan-review.md` の mtime（存在しない場合は無し）を記録した上で、`Skill` ツールで `review-plan`（対象計画ファイルパスを引数として渡す）を呼び出す
2. 呼び出し後、`./tmp/plan-review.md` の mtime が更新されている（または新規作成された）ことを確認する。更新されていない場合は `review-plan` が正常完了しなかったとみなし、プロセスを停止してユーザーに報告する
3. `./tmp/plan-review.md` を Read ツールで読み、「`**総合判定**:` 行」の直後の値を**文字列一致で**判定する（LLM の雰囲気判断に頼らない）。値は `LGTM` / `要修正` / `要再設計` のいずれか
4. 判定に応じて分岐する:
   - **`LGTM`** → ループを抜けてフェーズ 3（完了報告）へ進む
   - **`要修正` かつ 反復カウンタ < N** → `Skill` ツールで `revise` を呼び出す。引数は**明示的に 2 つとも渡す**（第 1 引数 `./tmp/plan-review.md`、第 2 引数に本スキルが受け取った対象計画ファイルパス）。`revise` の対象ファイル自動判別は `plan.md`/`fix-plan.md` の 2 パターンしか認識しないため、任意ファイルパスに対応するには自動判別に頼らない。呼び出し前に対象計画ファイルの mtime を記録し、呼び出し後に更新されていることを確認する。更新されていない場合は `revise` が完了しなかったとみなし、ループを中断してユーザーに状況（何反復目で・どちらの停止経路か）を報告する。更新を確認できた場合のみ反復カウンタを +1 して手順 1 に戻る
   - **`要修正` かつ 反復カウンタ ≥ N** → ループを中断する。反復カウンタが 0 の場合（N=0 設定等でまだ一度も `revise` の更新成功を記録していない場合）は「`revise` を1回試みる（上限を無視して実行する） / このまま中断する」を、反復カウンタが 1 以上の場合は「反復を継続する / ここで打ち切りユーザーが手動介入する / 中断する」を `AskUserQuestion` で確認する。「反復を継続する」または「1回試みる」が選択された場合、上限 N を +1 した上で手順 1 に戻る。ただし延長は本ループにつき1回のみとし、再度上限に達した場合は自動延長せず中断する
   - **`要再設計`** → ループを即座に中断する。対象計画ファイルパスの basename が `fix-plan.md` で終わる場合は「`/fix` で修正計画を作り直す / このまま続行を試みる / 中断する」を、それ以外（`plan.md` 系。一致しない任意パスもここにフォールバックする）は「`/design` で再設計する / このまま続行を試みる / 中断する」を `AskUserQuestion` で確認する。「このまま続行を試みる」が選択された場合、`要修正` と同等に扱って上記の `revise` 呼び出し分岐に進む（反復カウンタは通常どおり加算）
   - **上記いずれとも一致しない場合** → プロセスを停止し、`./tmp/plan-review.md` の内容と共にユーザーに報告する
5. `revise` がフェーズ3のユーザー確認（曖昧なフィードバック・矛盾する要求・技術的制約の不足の検出）でプロセスを停止した場合は、ユーザーの回答を待って `revise` を再度呼び出し、通常どおり処理を継続する（本スキル側で新規に肩代わりする実装は不要。`revise` 既存の契約をそのまま活かす）。この場合も `revise` の呼び出しは1回として数え、追加増分は行わない

### フェーズ 3: 完了報告

以下の内容をユーザーに報告する:

- 反復回数（対象計画ファイルの更新を伴う `revise` 呼び出し成功回数）
- 最終判定（`LGTM` / 上限到達で中断 / 要再設計で中断 / 継続選択によりループ再開→最終的に {`LGTM` / 上限到達で中断 / 要再設計で中断 / ユーザー中断} / ユーザー中断）
- `revise` を挟んだ場合は、各反復での主な修正点の要約
- 次のステップの案内: `LGTM` の場合は `/implement {対象計画ファイルパス}`、中断の場合はユーザーの選択に従う

## 注意事項

本スキルはコード実装を持たないため、反復上限の遵守はモデル自身の instruction-following に依存する（ハード保証ではない）。暴走が観測された場合は DIY Stop hook や Workflow ツールでの構造化実装への切替を検討する。

N=3 をフル実行した場合、`review-plan`（3周レビュー＋コードベース調査＋サブエージェント連携を伴う重い処理）を最大4回・`revise` を最大3回実行するため、トークン・時間コストは相当な規模になる。

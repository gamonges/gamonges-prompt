`review-plan ⇄ revise` の LGTM ループを 1 コマンドで実行する。計画ファイルに対して「レビュー → 判定 → 必要なら修正 → 再レビュー」を、総合判定が `LGTM` になるまで自動的に繰り返す。

**重要**: 本スキル自体はソースコードを変更しない。内部で呼び出す `review-plan`/`revise` がそれぞれの既定出力（`./tmp/plan-review.md` の生成、対象計画ファイルの上書き）を行う。
**規約**: CLAUDE.md の Skills 共通規約に従う

## パラメーター

`$ARGUMENTS` で対象計画ファイルのパスを指定できる。省略時は `./tmp/plan.md` を使用する（`revise`/`review-plan` と同じ「ファイルパス」規約に統一）。

## ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| `/design` 計画作成 or `/fix` 修正計画作成 | `/plan-lgtm` LGTM ループ | `/implement` 実装開始 |

## 実行条件

以下のファイルが存在することを確認する:

- `$ARGUMENTS`（指定がない場合は `./tmp/plan.md`）— 対象計画ファイル

存在しない場合:

- プロセスを即座に停止する
- 対象ファイルが存在しない旨をユーザーに報告する
- ループを開始しない

## 実行プロセス

### フェーズ 1: 初期化

- [ ] 対象計画ファイルパスを確定する（`$ARGUMENTS` または `./tmp/plan.md`）
- [ ] 反復カウンタを 0 に設定する
  - **反復カウンタの定義**: `revise` の呼び出し回数を数える（`review-plan` の呼び出し回数ではない）。初回の `review-plan` 呼び出しはカウントに含めない
  - **上限 N = 3**: `revise` を最大 3 回まで呼び出す。値を変更したい場合はこのファイルの本節の数値を直接編集する

### フェーズ 2: ループ本体

各反復で以下を実施する:

1. `Skill` ツールで `review-plan`（対象計画ファイルパスを引数として渡す）を呼び出す
2. `./tmp/plan-review.md` を Read ツールで読み、「総合判定」フィールドの値を**文字列一致で**判定する（LLM の雰囲気判断に頼らない）。値は `LGTM` / `要修正` / `要再設計` のいずれか
3. 判定に応じて分岐する:
   - **`LGTM`** → ループを抜けてフェーズ 3（完了報告）へ進む
   - **`要修正` かつ 反復カウンタ < N** → `Skill` ツールで `revise` を呼び出す。引数は**明示的に 2 つとも渡す**（第 1 引数 `./tmp/plan-review.md`、第 2 引数に本スキルが受け取った対象計画ファイルパス）。`revise` の対象ファイル自動判別は `plan.md`/`fix-plan.md` の 2 パターンしか認識しないため、任意ファイルパスに対応するには自動判別に頼らない。呼び出し後、反復カウンタを +1 して手順 1 に戻る
   - **`要修正` かつ 反復カウンタ ≥ N** → ループを中断する。`AskUserQuestion` で「反復を継続する / ここで打ち切りユーザーが手動介入する / 中断する」を確認する
   - **`要再設計`** → ループを即座に中断する。`AskUserQuestion` で「`/design` で再設計する / このまま続行を試みる / 中断する」を確認する
4. `revise` 自身が曖昧なフィードバック等の理由で `AskUserQuestion` により停止した場合は、その回答を待って通常どおり処理を継続する（本スキル側で新規に肩代わりする実装は不要。`revise` 既存の契約をそのまま活かす）

### フェーズ 3: 完了報告

以下の内容をユーザーに報告する:

- 反復回数（`revise` を呼び出した回数）
- 最終判定（`LGTM` / 上限到達で中断 / 要再設計で中断 / ユーザー中断）
- `revise` を挟んだ場合は、各反復での主な修正点の要約
- 次のステップの案内: `LGTM` の場合は `/implement`、中断の場合はユーザーの選択に従う

## 注意事項

本スキルはコード実装を持たないため、反復上限の遵守はモデル自身の instruction-following に依存する（ハード保証ではない）。暴走が観測された場合は DIY Stop hook や Workflow ツールでの構造化実装への切替を検討する。

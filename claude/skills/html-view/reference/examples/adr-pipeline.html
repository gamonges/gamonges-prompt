文脈

背景

dresscode-data-connector の既存 API パイプライン（ENTRA / M365 / BOX / HRMOS）は、FetchPage Lambda を nextLink で直列ループする ApiStateMachine を共通利用している。新規追加する GOOGLE_DRIVE × FILE は以下のドメイン特性により、既存 sequential パターンでは現実的な時間で完了しない。

  • 対象規模: 500 Drive 程度 / 1 Drive あたり最大 30 分（実測ベース）
  • Drive 単位の独立性: My Drive はオーナー毎、Shared Drive は driveId 毎に認証 (DwD impersonation) と quotas が分離しており、並列化しても互いに干渉しない
  • API 制約:
    • files.list の pageSize 最大値は 100（Box の 1000 と異なる）
    • Drive 横断のページネーションは存在しない（1 Drive 内のみ）
    • permissions.list はファイル単位 API のため「ファイル数 × 1 呼び出し」が必要
  • 基盤制約: Lambda 10 分タイムアウト / Step Functions Standard 子実行 25,000 遷移上限 / state 256KB 上限
  • 取り込みモード: 初期実装は FULL のみ（INCREMENTAL は将来対応）

直列実行した場合、500 Drive × 30 分 = 250 時間となり、リトライ・運用に耐えない。かつ、既存 4 サービスのパイプラインを 1 byte も壊さずに拡張することが必須条件（CI の CFN snapshot diff で担保）。

結論

チームに特に共有したい 2 点

1. 全ファイル・フォルダの収集方法 — Drive 単位のフラット files.list で 1 Drive 内の ファイル + フォルダを区別なく一気に列挙し、収集後に folderMeta から FolderPathCache を O(N) メモ化で構築する。再帰的にフォルダを掘る従来パターンを採用しなかった。

2. Distributed Map による Drive 単位並列 Fetch — 新規 construct DistributedMapApiStateMachine を導入し、1 子実行 = 1 Drive で FetchPage Lambda を並列に走らせる。Lambda 10 分タイムアウトと Step Functions 25,000 遷移上限の双方を 1 つの設計で吸収する。

その他の判断（Claim-Check / handler dual-mode / ファクトリー / CDK 循環依存回避）はこの 2 点を成立させるための補助である。

重点 1: フラット files.list + 後段 FolderPathCache で全件収集

何を採用したか

1 Drive あたり 1 種類の API 呼び出し（files.list）で ファイルとフォルダを区別なくフラットに全件列挙し、すべて集めた後で folderMeta Map から FolderPathCache を構築する。フォルダを再帰的に掘り下げて子を辿る古典パターンは採用しなかった。

実装の中心

lambda/data-pipeline/api/ingestion/google-drive/_lib/google-drive-api/files.ts:

export async function listDriveItems(
  client: GoogleDriveApiClient,
  options: { driveId?: string; ownerEmail?: string; pageToken?: string },
) {
  const params: Record<string, string> = {
    fields: DRIVE_FILE_FIELDS_PARAM,
    pageSize: '100',                 // Google Drive API の最大値
    supportsAllDrives: 'true',
  };
  if (options.driveId) {
    // Shared Drive: 内側全件を 1 リストで取得
    params.corpora = 'drive';
    params.driveId = options.driveId;
    params.includeItemsFromAllDrives = 'true';
    params.q = 'trashed=false';
  } else {
    // My Drive: オーナーのファイルのみ
    params.q = "'me' in owners and trashed=false";
  }
}

explore フェーズの handler 側ループでは、ファイル / フォルダの両方を pendingFileIds に積み、フォルダだけ folderMeta にも保存する：

for (const item of result.files) {
  if (item.mimeType === FOLDER_MIME_TYPE) {
    folderMeta.set(item.id, { name: item.name, parentId: item.parents?.[0] });
  }
  // ファイル・フォルダ両方を Data Lake 保存対象にする
  pendingFileIds.push(item.id);
  exploreCache.set(item.id, item);
}

全件列挙が終わったタイミングで FolderPathCache を一括構築する。フラット list では 子フォルダが親より先に出現することがあるため、メモ化付き再帰でパス解決を行う（計算量 O(N)、各フォルダは 1 度しか resolve されない）。

My Drive と Shared Drive で driveKey を揃える

Google Drive API は My Drive のトップに置かれたファイルの親として 'root' を返すため、My Drive 用の driveKey も 'root' に揃える（'user:{email}' などにすると親解決が崩れる）。Shared Drive は driveId をそのまま使う。 

const driveKey = driveId ?? 'root';
const driveName = currentDrive.type === 'SHARED_DRIVE' ? currentDrive.driveName : 'My Drive';
const folderPathCache = buildFromFlatList(folderMeta, driveKey, driveName);

permissions.list だけはファイル単位

files.list は権限を返さないため、process フェーズで 500 件単位のバッチを Promise.allSettled + GOOGLE_DRIVE_BATCH_CONCURRENCY=15 で並列実行する。403/404 は空配列を返して skip、それ以外は warn ログのみで握りつぶさず例外化する。

重点 2: Distributed Map による Drive 単位並列 Fetch

何を採用したか

新規 construct DistributedMapApiStateMachine を導入し、AWS Step Functions Distributed Map で Drive 単位に水平並列化する。1 子実行 = 1 Drive で FetchPage Lambda を回し、Drive 内のページネーションは子実行内ループで処理する。

itemReader / itemSelector の要点

lib/stack/data-pipeline/constructs/distributed-map-api-state-machine.ts:

const distributedMap = new sfn.DistributedMap(this, `DistributedMap_${constructId}`, {
  itemReader: new sfn.S3JsonItemReader({
    bucket: workBucket,
    key: sfn.JsonPath.stringAt('$.driveListS3Key'),   // 前段が S3 に退避した driveList を入力
  }),
  mapExecutionType: sfn.StateMachineType.STANDARD,
  maxConcurrency: orchestrationStrategy.maxConcurrency,            // 5
  toleratedFailurePercentage: orchestrationStrategy.toleratedFailurePercentage ?? 0,
  itemSelector: {
    'tenantId.$': '$.tenantId',
    /* ... */
    'driveListS3Key.$': '$.driveListS3Key',
    'driveIndex.$': '$$.Map.Item.Index',              // Map が割り当てる Drive 番号
    driveMode: 'PROCESS_SINGLE_DRIVE',
    nextLink: null,
  },
  resultPath: sfn.JsonPath.DISCARD,
});

dual-mode handler

同じ FetchPage Lambda が 3 つのモードを driveMode で演じ分けることで、列挙ロジックと処理ロジックを 1 つのコードベースで保つ：

driveMode役割
未指定全 Drive を 1 実行で直列処理（既存 sequential 互換、障害時のフォールバック）
'ENUMERATE_ALL_DRIVES'drives 列挙だけ実行し driveListS3Key を Distributed Map に渡す
'PROCESS_SINGLE_DRIVE'Map 子実行が driveIndex 指定で起動。drives フェーズを飛ばして explore から開始し、完了時に per-drive S3 をクリーンアップ

ENUMERATE_ALL_DRIVES では pagination 中も完了時も driveListS3Key を必ず出力（pagination 中は null）。Step Functions の resultSelector が JsonPath でこのフィールドを参照するため、フィールド未出力 (undefined) だと States.ResultPathMatchFailure で実行が失敗する。

タイムアウト階層と並列度

レイヤータイムアウト備考
メイン Pipeline SM8 時間3h → 8h に延長
Distributed Map SM6 時間500 Drive × 30min / maxConcurrency=5 の実測ベース
FetchPage Lambda10 分MAX_BATCH_DURATION_MS は 9 分（60s バッファ）
Drive 並列度5My Drive / Shared Drive 共通。将来は分離検討（F-3）

補助的な設計判断

重点 1・2 を支えるための判断。優先度は低いが、再現性のために残す。

補助 1: Claim-Check パターンで中間データを S3 退避

_lib/exploration/claim-check.ts を新設し、driveList / pendingFileIds / exploreCache / folderPathCache / folderMeta を Work Bucket に書き出し、state には S3 キーのみ持つ。キー構造は {tenantId}/{orgId}/{taskId}/[drive-{index}/]{name}.json で、driveIndex 付与で Distributed Map 子実行同士の S3 衝突を防ぐ。driveList だけは task レベル共有のため driveIndex を付けない。

理由: Step Functions state 256KB 上限を超える中間データを Lambda 跨ぎで持ち回すため。

補助 2: handler の Phase Machine + driveMode 化

handler は 3 フェーズ Phase Machine (drives → explore → process) を nextLink (base64url JSON) に詰めて Step Functions に戻す。各フェーズ内で isApproachingTimeout() を都度チェックし、9 分超過直前に中間データを S3 退避してから return する。Lambda タイムアウトを自然なシャーディング境界として使うイディオム。dual-mode との合わせ技で、列挙だけ / 1 Drive だけ / 全 Drive 直列 の 3 用途を 1 つの handler でカバーする。

補助 3: ファクトリーパターン + 共通部品抽出（CFN diff ゼロ）

api-state-machine-factory.ts で ApiPipelineConfig.orchestrationStrategy (sequential / distributed-map) を切り替え、_lib/api-state-machine-parts.ts に共通の Task / Map / Fail 部品を抽出。共通部品は scope と child id を呼び出し側から引数で受ける設計（関数内部で Construct を生成しない）。

理由: 既存 ENTRA / M365 / BOX / HRMOS の CFN snapshot を 1 byte も変えずに済ませる。scope/id を内部生成すると論理 ID が変わり既存リソース replacement になる。

補助 4: CDK Token 循環依存を formatArn で回避

Distributed Map の IAM 権限付与で stateMachine.stateMachineArn（CDK Token）を Role policy に渡すと StateMachine → Role → StateMachine の CFN 循環依存になる。stack.formatArn({service:'states', resource:'stateMachine', resourceName:<name>, arnFormat: ArnFormat.COLON_RESOURCE_NAME}) で 既知の name から static ARN 文字列を組み立てて Token 参照を切る。

理由: CDK Token は合成時にプレースホルダー文字列のままで、.split() しても undefined を返すためビルド時に気付けない。Lambda / DynamoDB / SQS の自己参照でも同じ罠が再発する。

補助 5: 細かな CDK / Step Functions の罠
  • mapExecutionType と itemProcessor.executionType の両方が必要（CDK は warn を出すが synth validation 上は必須）
  • S3JsonItemReader はルート配列 JSON しか読まないため、DriveList 型を {drives: DriveEntry[]} → DriveEntry[] に変更
  • files.list の pageSize 最大値は 100（Box の 1000 と勘違いしない）

結果

正の影響

  • 500 Drive 規模で 直列 250h → 並列 50h 以内の見込みになり、メイン Pipeline SM 8 時間ウィンドウ × 複数回のスケジュール実行で全量取り込みが現実的になった
  • 既存 4 サービス（ENTRA / M365 / BOX / HRMOS）の CFN snapshot を 1 byte も変えずに済んだ。ファクトリーパターンと共通部品抽出により破壊変更ゼロを CI で担保
  • explore を フラット list + 後段 FolderPathCache にしたことで、Lambda 中断時の再開コストが最小化された（folderMeta を S3 から読み戻すだけ）
  • handler の dual-mode 設計により、開発時は driveMode 未指定で直列実行による手動デバッグが可能、本番では Distributed Map で並列、という local→prod の経路差を最小化
  • Box の process フェーズも同じファクトリーで Distributed Map 化可能（F-2 フォローアップ）

負の影響 / 既知の制限

  • driveList.json のクリーンアップは lifecycle 任せ: 子実行ごとの cleanupSingleDriveArtifacts では driveListS3Key を削除できない（他の子実行が読み込み中の可能性）。Work Bucket の 7 日 lifecycle に依存（F-7 で全 Map 完了後の明示削除を検討）
  • INCREMENTAL 未対応: Phase 4 で Drive Changes API を使った差分取り込みを実装予定（F-5）
  • My Drive / Shared Drive で共通の maxConcurrency=5: 認証単位の API クォータが異なるため本来は別値が望ましい（F-3）
  • toleratedFailurePercentage=0: 初期は厳しめで運用。1 Drive 失敗で Map 全体が失敗するため運用負荷あり（F-6 で ResultWriterV2 集約と percentage 引き上げ）
  • handler の認知負荷が増えた: 3 モード分のテストが必要になり、handler.test.ts に 7 ケース追加

チーム共有時に強調したい教訓

  1. Drive のように「認証単位で独立した塊」を全件収集するときは、まずフラット list で全件取って後段で構造化する。再帰探索は中断・再開・並列化のすべてで不利
  2. Step Functions Distributed Map + S3JsonItemReader は Step Functions の 25K / 256KB 制約を一気に解消するための定番パターン。今後 Box の process フェーズなど同型の問題には迷わず採用する
  3. S3JsonItemReader を使うなら型をルート配列に揃える（{drives: []} 形式は読めない）
  4. CDK Role policy に SM の ARN を入れる時は formatArn で static 化する
  5. CDK construct を共通化するなら scope/id を引数で受ける（CFN diff ゼロの条件）
  6. mapExecutionType と itemProcessor.executionType は警告が出ても両方必須
  7. Step Functions の resultSelector が参照するフィールドは handler が pagination 中も null 出力する（undefined はエラー）
  8. Google Drive API の pageSize 最大値は 100。Box の 1000 と勘違いしない

代替案

重点 1（収集方法）の代替案

採用しなかった: フォルダ単位の再帰探索

各フォルダに対して files.list?q='{folderId}' in parents を投げ、子フォルダが見つかったら再帰的に潜る古典パターン。

観点再帰探索フラット list（採用）
API 呼び出し回数フォルダ数 × files.list で O(F)Drive あたり N/100 回（1 種類のクエリ）
ページネーション境界フォルダごとに別 token を持ち回し state 肥大化Drive 単位の listPageToken 1 本
Lambda 中断耐性「どのフォルダを掘っている途中か」を保持する必要folderMeta を S3 退避するだけで再開可
横に広い Drive浅いが大量のフォルダで呼び出しが爆発一定
深いフォルダ木の深さに応じて呼び出しが増える深さに非依存

要点は 「認証単位 (Drive) でフラットに全件取り、構造化は後段でやる」。Drive API は親情報 (parents[]) を最初から返してくれるので、わざわざフォルダ単位で再帰する必要がない。

重点 2（並列化）の代替案

採用しなかった: Inline Map

sfn.Map で $.chunks 等 state 内配列を items とする方式。

評価: items が state に乗るため、500 Drive 規模の DriveList JSON が Step Functions の 256KB 上限を超える可能性が高い。Distributed Map の S3JsonItemReader なら配列を S3 から読むためサイズ無制限。

採用しなかった: 並列 Lambda 自作 (Promise.allSettled で Lambda を fan-out)

評価: 失敗の集計 / 再試行 / 観測性をすべて自前実装になり、Step Functions と二重管理になる。Distributed Map なら toleratedFailurePercentage で部分失敗を構造化でき、CloudWatch との統合も標準。

採用しなかった: EventBridge による fan-out

EventBridge イベントで Drive 単位の Lambda を非同期起動する方式。

評価: スケジューラと Pipeline の状態が分離し、メイン Step Functions から完了監視ができない。失敗時の retry も EventBridge 側の独立した設定になり、Pipeline 全体の Atomicity が崩れる。

採用した: Distributed Map

  • items を S3 上のルート配列 JSON から読めるためサイズ無制限
  • 子実行は STANDARD で 25K 遷移上限が独立
  • toleratedFailurePercentage で部分失敗を構造化
  • メイン Step Functions からの観測性がそのまま使える

handler 設計の代替案

採用しなかった: 列挙用 Lambda と処理用 Lambda の分離

評価: 認証 / 共通ロジック / テストが二重化する。driveMode で分岐する dual-mode の方が 同じテストでカバーできる。

採用した: 単一 handler の dual-mode 化

  • 1 つの handler が 3 モード（未指定 / ENUMERATE_ALL_DRIVES / PROCESS_SINGLE_DRIVE）を演じ分け
  • 開発時は driveMode 未指定で直列実行による手動デバッグが可能
  • テストは handler.test.ts 1 ファイルでカバー

P1ステップ 1: folder-path-cache.ts に循環検出を追加

種別: バグ修正 出典: unified.md Critical #3 対象: lambda/data-pipeline/api/ingestion/google-drive/_lib/exploration/folder-path-cache.ts:81-108

現状の問題

buildFromFlatList 内の再帰関数 resolve(folderId) は memoization のみで循環検出がない。Google Drive API が A→B→A のように parents 循環を返した場合 (API バグ・破損データ・競合) に再帰スタックがオーバーフローし Lambda がクラッシュする。Drive ごとに数十万フォルダを扱う想定なので発火確率は無視できない。

function resolve(folderId: string): string {
  const cached = cache.paths[folderId];
  if (cached !== undefined) return cached;
  // ...
  const parentPath = resolve(folder.parentId);  // ← in-progress マーカーなし
  const fullPath = `${parentPath}/${folder.name}`;
  cache.paths[folderId] = fullPath;
  return fullPath;
}

修正方針

関数スコープに inProgress = new Set<string>() を保持し、再帰呼び出し前に追加・完了時に削除する try/finally パターン。循環検出時は warn ログ + フォルダ名のみで graceful degradation。

const inProgress = new Set<string>();

function resolve(folderId: string): string {
  const cached = cache.paths[folderId];
  if (cached !== undefined) return cached;
  if (folderId === driveKey) return driveName;

  if (inProgress.has(folderId)) {
    const folder = folders.get(folderId);
    const fallback = folder?.name ?? folderId;
    cache.paths[folderId] = fallback;
    logger.warn('folder-path-cache: cycle detected', { folderId, fallback });
    return fallback;
  }
  // ...
  inProgress.add(folderId);
  try {
    const parentPath = resolve(folder.parentId);
    // ...
  } finally {
    inProgress.delete(folderId);
  }
}

P1ステップ 3: process フェーズループ内に isApproachingTimeout チェックを追加

種別: バグ修正 出典: PR レビュー gemini #1 対象: handler.ts:475-516

現状の問題

process フェーズのバッチループ for (let i = 0; i < batch.length; i += batchConcurrency) に timeout チェックがない。500 アイテムバッチ × API 応答遅延で 10 分上限を超えると、state.totalProcessedFiles の加算と storePendingFileIds 退避が走らず再実行時に同じバッチを丸ごと再処理する。drives / explore フェーズには既存チェックがあるのに process だけ抜けている。

修正方針

バッチループ先頭で isApproachingTimeout() を判定し、true なら 未処理 chunk を pendingFileIds 先頭に巻き戻し て break。ループ後の既存ロジック (storePendingFileIds + buildPaginationOutput) で次回再開状態が自然に保存される。

const items: GoogleDriveFileItem[] = [];
for (let i = 0; i < batch.length; i += batchConcurrency) {
  if (isApproachingTimeout()) {
    pendingFileIds.unshift(...batch.slice(i));
    logger.info('Process phase approaching timeout, suspending batch', {
      processedInBatch: items.length,
      remainingInBatch: batch.length - i,
    });
    break;
  }
  const chunk = batch.slice(i, i + batchConcurrency);
  // ... 既存の Promise.allSettled ロジック (変更なし)
}

P2ステップ 6: StubTokenProvider にプロダクションガードを追加

種別: セキュリティ強化 出典: PR レビュー greptile #8 対象: _lib/google-drive-api/stub-token-provider.ts

現状の問題

handler.ts:183 で new StubTokenProvider() を無条件に生成。GOOGLE_DRIVE_STUB_TOKEN 環境変数から固定トークンを読み込み、本番環境に誤って有効なトークンがセットされた場合、全ユーザーが同一認証情報で処理されるリスク。Phase 2 で WifTokenProvider に差し替え予定だが、過渡期のガードがない。

修正方針

export class StubTokenProvider implements TokenProvider {
  constructor() {
    const stage = process.env.STAGE ?? process.env.NODE_ENV;
    if (stage === 'prd' || stage === 'production') {
      throw new Error(
        'StubTokenProvider must not be used in production. Use WifTokenProvider instead.',
      );
    }
    logger.warn('Using StubTokenProvider (Phase 1 only).', { stage });
  }
}

P3ステップ 10: PR レビュー gemini #3 への却下返信

種別: コミュニケーション 対象: なし (PR コメント返信のみ)

却下根拠付き返信

gemini 指摘: 「My Drive の場合、driveKey にエイリアス 'root' を使用しているが、API レスポンスの parents フィールドには実際のフォルダ ID が含まれるため一致しない可能性」

反論根拠 (tmp/lessons.md 2026-05-21):

Google Drive API は My Drive ルート配下のアイテムについて parents: ['root'] を返す。driveKey='root' で実装することで folder-path-cache の親解決が正しく機能する。'user:{email}' を driveKey にすると parents: ['root'] と一致しないため逆に解決失敗する。

返信文 (案):

ご指摘ありがとうございます。本実装は実機検証済みで、Google Drive API は
My Drive ルート配下のアイテムに対して `parents: ['root']` を返すため、
`driveKey='root'` で文字列比較が成立します。

`'user:{email}'` を driveKey にすると API レスポンスの 'root' と
一致せず逆に path resolution が失敗するため、現実装が正しいです。

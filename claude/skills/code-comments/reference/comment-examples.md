# コメント例集（BAD / GOOD）

> `code-comments/SKILL.md` 第 1 部から参照される詳細例。SKILL.md 本体の最小例を超えて、
> 実コードに近い形で示す。言語は例示用で、原則は言語非依存。

## 残すべき 7 分類の詳しい例

### 1. Why（なぜこのアプローチか）

```ts
// BAD: 何をしているかの言い換え（コードを読めば分かる）
// retries を 3 に設定
const retries = 3

// GOOD: なぜ 3 なのか（コードから読めない判断）
// 決済 API の SLA は 99.9%。3 回で実質 99.9999% をカバーでき、それ以上は
// レイテンシ悪化が利益を上回るため 3 に固定。
const retries = 3
```

### 2. Why not（却下した代替）

```ts
// GOOD: 自然に見える別解をなぜ採らないか
// ここで Promise.all を使わない: 取引先 API は同一トークンの並列呼び出しを
// 429 で弾くため、直列 + backoff が必須（並列化は contract 違反）。
for (const id of ids) await fetchWithBackoff(id)
```

### 3. 不変条件・契約

```ts
// GOOD: 呼び出し側が守るべき前提（破ると静かに壊れる）
// 事前条件: items は updatedAt 昇順でソート済みであること。
// 本関数は最後の要素を「最新」とみなし、再ソートしない。
function pickLatest(items: Item[]): Item { /* ... */ }
```

### 4. 危険・落とし穴

```ts
// GOOD: 順序依存の警告
// 順序厳守: setAuthHeader は initSession が張る cookie に依存する。
// 入れ替えると未認証リクエストになる。
initSession()
setAuthHeader()
```

### 5. 非自明な意図

```ts
// GOOD: 複雑な式が何を達成したいか
// 月末日の繰り上げを吸収する: 1/31 に 1 か月足して 3/3 になる JS Date の
// 仕様を避け、「翌月同日、無ければ月末」に正規化する。
const next = normalizeMonthEnd(addMonths(d, 1))
```

### 6. 根拠への参照（経緯ではなく根拠）

```ts
// GOOD: 仕様・issue へのアンカー（grep 可能）
// 上限は spec 由来: SPEC-1042 §3.2（添付は 25MB まで）。issue: #874
const MAX_ATTACHMENT = 25 * 1024 * 1024
```

### 7. 意図的な逸脱

```ts
// GOOD: バグに見えるが意図的
// intentional: ここでは await しない。通知は best-effort で、失敗しても
// 本処理をブロックしない（送信失敗は logger 経由で監視）。
void notifyAsync(user)
```

## レビュー指摘の蒸留 before / after

レビュー指摘 → コードに残すのは「指摘が明らかにした制約」だけ。経緯は書かない。

| レビュー指摘（素材）| ❌ 経緯の転写 | ✅ 制約への蒸留 |
|--------------------|--------------|----------------|
| 「null のとき落ちる」| `// レビュー指摘で null チェック追加` | `// fetchUser() は 404 で null を返す（throw しない）ため分岐必須` |
| 「ここ N+1 では？」| `// レビューで N+1 指摘されたので join に変更` | `// user ごとに query すると N+1。1 クエリで JOIN 取得（件数は最大 1 万）` |
| 「マジックナンバー」| `// 800 はマジックと言われたので定数化` | `// 800ms = 下流 LB のアイドルタイムアウト 1s から安全マージン 200ms を引いた値` |
| 「型が any」| `// any を直すよう言われた` | （コメント不要。型で表現する＝コードで語れる）|

最終行が重要: **蒸留した結果「コードで表現できる」と分かったら、コメントではなくコードを直す**。

**規約**: CLAUDE.md の Skills 共通規約に従う

> **SSOT 宣言**: コードコメントの方針は本スキルが single source of truth。design / fix / implement など他スキルはここを名指し参照し、方針を再宣言しない。方針を更新する場合は本ファイルのみを変更する。

> **`review-comments` との違い**: `review-comments` は PR 上のレビューコメントの返信・resolve を扱う。本スキルは**コード内に残すコメントの書き方**を扱う。別ドメイン。

本スキルは 2 部構成:

- **第 1 部（方針）**: 他スキルから名指し参照される「どんなコメントを残すか」の核。
- **第 2 部（単独実行）**: `/code-comments` で既存コードのコメントを方針に照らして見直す。

---

## 第 1 部: コメント方針（被参照の核）

### 大原則

- **経緯（history）は VCS、判断と制約（why & invariants）はコード**。「いつ誰が何を言ったか」はコミット / PR / issue に委ね、コードには書かない。
- **コードが言えないことだけをコメントする**。コードを読めば分かることは書かない（参考: 97 Things — Comment Only What the Code Cannot Say https://github.com/97-things/97-things-every-programmer-should-know/tree/master/en/thing_16 ）。

### 残すべきコメント 7 分類

| # | 種類 | 内容 | 最小例 |
|---|------|------|--------|
| 1 | Why（判断理由）| なぜこのアプローチか | `// 線形探索で十分。要素数は常に数十件で hash 化コストの方が高い` |
| 2 | Why not（却下案）| 一見良い別解を採らない理由 | `// Promise.all は使わない。API がレート制限で並列不可` |
| 3 | 不変条件・契約 | 常に成り立つべき前提 | `// 呼び出し側でソート済みを保証。ここで再ソートしない` |
| 4 | 危険・落とし穴 | 触ると壊れる / 順序依存 | `// この 2 行は順序を入れ替えると初期化前参照になる` |
| 5 | 非自明な意図 | 複雑な処理が達成したいこと | `// 閏秒をまたぐ場合に時刻が巻き戻るのを補正` |
| 6 | 根拠への参照 | 仕様 / RFC / issue（経緯でなく根拠）| `// RFC 7231 §6.4 に従いリダイレクトは最大 5 回` |
| 7 | 意図的な逸脱 | バグに見えるが意図的 | `// intentional: 例外を握りつぶす。リトライは上位層の責務` |

共通項: いずれも**コードを読んでも復元できない情報**（外部制約・判断の背景・人間の意図）。

### AI エージェントに特に効くコメント

AI は「離れた依存を見落とす」「親切に勝手に直す」「セッションをまたぐと判断履歴を失う」。次が増幅して効く:

- **連動性の明示**: `// この enum を追加したら registry.ts のディスパッチ表も更新すること`
- **「直すな」の明示**: `// DO NOT refactor into async. 同期実行であることに別スレッドが依存`
- **検索可能なアンカー**: `// spec: SPEC-1042 / issue: #381`（grep で根拠を辿れる）
- **外部世界の制約**: `// Safari < 16 は structuredClone 非対応のため手動 deep copy`（コードベースに無い知識）

### レビュー指摘の「蒸留」

レビュー指摘はコメントの**素材**だが、そのまま転写しない。指摘が明らかにした「コードから読めない理由・制約」に蒸留してから残す。指摘の経緯（誰が・いつ・何を）は PR / コミットに委ねる。多くのレビュー指摘は「暗黙の前提がコードに書かれていなかった」ことが原因であり、その前提こそが残す素材になる。

- ❌ 経緯の転写: `// レビューで指摘されたので null チェックを追加`
- ✅ 理由への蒸留: `// 上流の fetchUser() は 404 時に null を返す（例外でない）ため null 分岐が必須`

### アンチパターン（削除対象）

- **レビュー対応の経緯**: `// PR #123 の指摘で追加` — VCS / PR で追跡可能。やがて文脈を失い comment rot する。
- **コードの言い換え（noise）**: `// i を 1 増やす` — トークンの無駄。コードとの乖離リスク（AI には誤誘導にもなる）。
- **コメントアウトされた旧コード**: 復活可否が不明になる。VCS に委ねる。

### 判定ヒューリスティック

| 問い | Yes なら | No なら |
|------|---------|---------|
| 1 年後・別文脈でも真か？ | 残す候補（不変条件・制約）| 経緯の可能性大 → 残さない |
| コードを読めば分かるか？ | 消す（コードで語れ）| 残す候補 |
| VCS / PR / issue で代替できるか？ | コードに書かない（必要ならリンク）| 残す候補 |
| 知らずに変更したら壊れるか？ | **必ず残す**（警告・不変条件）| 優先度低 |

ひとこと: **経緯は VCS、判断と制約はコード**。

> 言語別の詳しい BAD/GOOD 例・蒸留の追加例は `./reference/comment-examples.md` を参照（より深く理解したい時）。

## 第 2 部: 単独実行モード（`/code-comments`）

**重要**: ソースコードは変更しない。見直し結果は `./tmp/comment-review.md` に出力する。

### パラメーター

`$ARGUMENTS` に対象ファイル / ディレクトリを指定できる。省略時は `git diff`（unstaged + staged）を対象とし、それも空なら `git diff HEAD~1`（直近コミット）を対象とする。

### ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| 実装 / `/fix` 後 | `/code-comments` コメント見直し | `/implement`（修正適用）/ `/review`（深いレビュー）|

### 実行プロセス

1. 対象（`$ARGUMENTS` → `git diff` → `git diff HEAD~1`）のコメントを収集する。対象が空なら理由を報告して停止する。
2. 各コメントを第 1 部の「残すべき 7 分類」「判定ヒューリスティック」に照らして評価する。
3. アンチパターン該当（経緯転写 / 言い換え / コメントアウト旧コード）を抽出する。
4. レビュー指摘由来のコメントは蒸留の要否を判定する（蒸留した結果コードで表現できるなら、コメントではなくコード修正を提案）。
5. 指摘を `./tmp/comment-review.md` に `file_path:L{number}` 形式で出力する（残す / 削除 / 蒸留提案の区分付き）。
6. 深い設計レビューが必要な場合は `/review` を案内する。

> 単独実行は提案の出力に留める。実際の編集は `/implement` などコード変更スキルが行う。

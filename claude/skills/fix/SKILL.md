実装後のレビューで発見した問題や修正項目をもとに、修正実装計画を生成する。

**重要**: `./tmp/fix-plan.md` もしくは Cursor の plan モードで作成した `***.plan.md` のみを作成・変更する。ソースコードは変更しない。
**規約**: CLAUDE.md の Skills 共通規約に従う

## 補助ドキュメントへの参照

| 補助ドキュメント | 読むタイミング |
|------------------|----------------|
| `./reference/category-examples.md` | 修正項目の収集（fixes.md / PR レビューコメント / review.md）と分類をする時 |
| `./reference/fix-plan-template.md` | フェーズ 5 で fix-plan.md を出力する時 |

「念のため全部読む」は禁止。表のトリガー条件に該当する場合のみ読み込む。

## ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| `/implement` 実装完了 + `/review` or 手動レビュー | `/fix` 修正計画作成 | `/implement ./tmp/fix-plan.md` 修正実行 |

## パラメーター

`$ARGUMENTS` には以下のいずれかを指定できる。省略時は `./tmp/fixes.md` を使用する。

- **修正項目ファイルのパス**: `./tmp/fixes.md` 等のローカルファイル
- **PR の URL または番号**: `https://github.com/org/repo/pull/123` または `123`

`$ARGUMENTS` の種別判定と PR レビューコメントの取得詳細は `./reference/category-examples.md` を参照。

`$ARGUMENTS` の種別に関わらず、`./tmp/review/unified.md` が存在すれば併せて参照する。

## 実行条件

以下のいずれかが必須:

- `$ARGUMENTS` で指定されたファイル、または `./tmp/fixes.md` が存在する
- `$ARGUMENTS` が PR URL/番号であり、該当 PR が存在しアクセス可能
- `./tmp/review/unified.md` が存在する

いずれも該当しない場合: プロセスを即座に停止し、満たされていない条件を具体的にユーザーへ報告する。

`./tmp/plan.md` の存在（ギャップ分析に使用）: 存在しない場合は警告を出すが処理続行。

## 実行プロセス

### フェーズ 1: 入力の収集と前処理

詳細手順は `./reference/category-examples.md` を参照。骨格:

1. `$ARGUMENTS` の種別判定（PR URL / 番号 / ファイルパス / デフォルト）
2. 修正項目の読み込み（fixes.md / PR レビューコメント / review/unified.md / plan.md）
3. PR レビューコメントの取得（GraphQL + bot フィルタ + outdated 扱い）
4. 現在の実装状態の把握（`git diff HEAD --stat` / `git log --oneline -10`）
5. 修正項目の統合と分類（バグ修正 / 実装漏れ / 品質改善 / テスト追加）

### フェーズ 1.5: 既存 fix-plan.md の確認

`./tmp/fix-plan.md` が既に存在する場合、**分析を開始する前に**ユーザーに以下を確認する:

- **上書き（デフォルト）**: 実行前に既存 `./tmp/fix-plan.md` を `fix-plan.archived-{YYYYMMDD-HHmm}-{変更理由の要約}.md` にコピーして旧版を保存してから、新計画で完全置換する。archive ファイル冒頭に「なぜ新計画に置き換えるか」を1行で記す
- **追記**: 既存の末尾に新規ステップを追加（ステップ番号は既存最大番号の続き）。複数レビューラウンドにまたがって既存 fix-plan を育てていく場合はこちらを選ぶ

ユーザーの回答を得るまで次のフェーズに進まない。追記が選択された場合は、フェーズ 3 の調査で既存内容を参照し重複ステップ生成を避ける。

### フェーズ 2: ユーザー確認（必要に応じて）

修正意図不明 / 矛盾 / 影響範囲不明 を検出したら停止してユーザーに報告。詳細フォーマットは `./reference/category-examples.md` の「ユーザー確認が必要な状況」を参照。

### フェーズ 3: コードベース調査

修正項目が複数あり相互に独立している場合は、サブエージェントを同一ターンで複数 spawn して並列に調査する。修正項目が少数、または 1 ファイルで完結する場合は直接調査する。各項目で以下を確認する:

- [ ] **該当コードの確認**: 指摘されたファイル・行番号の現在のコードを読む
- [ ] **関連コードの調査**: Serena MCP でセマンティック検索を行い、修正の影響範囲を特定
- [ ] **既存パターンの確認**: 修正後のコードが従うべき既存実装パターンを特定
- [ ] **外部情報の調査**: Web search で外部ドキュメント・事例を検索
- [ ] **ライブラリ情報**: Context7 MCP でライブラリ・フレームワークの使用方法を確認
- [ ] **テスト状況の確認**: 修正対象のテストカバレッジ状況を確認

### フェーズ 4: 修正計画の策定

各修正項目について:

- [ ] 修正の具体的な手順を決定
- [ ] 修正による影響範囲を評価
- [ ] TDD 方針を決定（バグ修正・実装漏れには必ずテストを先に追加）
- [ ] 修正の依存関係と実行順序を決定
- [ ] 優先度（P1-P4）を決定（詳細は `./reference/category-examples.md`）
- [ ] **要件カバレッジを検証**: すべての修正項目が修正ステップでカバーされているか確認
- [ ] **コメント方針**: レビュー指摘に対応するコメントを fix-plan に記す場合は `code-comments` スキルに従い、経緯ではなく蒸留した制約・理由を「残すコメント」欄に書く

### フェーズ 5: 修正計画の出力

フェーズ 1.5 で選択された追記/上書き方針に従い、`./reference/fix-plan-template.md` のテンプレートで `./tmp/fix-plan.md` に出力する。

Cursor 環境の場合は `.cursor/plan.md` または指定された plan ファイルに修正ステップとして書き出す。

### フェーズ 6: 完了報告

`./reference/fix-plan-template.md` の完了報告フォーマットに従って、以下を報告する:

- 修正項目の総数と優先度別の件数
- 入力ソース（fixes.md / review.md / PR コメント / ギャップ分析）の取込み結果
- 最優先で対応すべきステップ（P1 Critical の一覧）
- 次のステップ案内（`/implement ./tmp/fix-plan.md` または `/revise`）

#### HTML view 化 (オプション)

完了報告に続けて、以下をユーザーに尋ねる:

> **HTML 化しますか?** (人間レビュア向けのデザイン HTML を生成)

ユーザーが Yes と回答した場合、**Claude は自動実行せず**、次のコマンドを案内する:

```
/html-view ./tmp/fix-plan.md
```

ユーザーが明示的に slash command を入力することで HTML 生成 + ブラウザ自動起動が完了する。

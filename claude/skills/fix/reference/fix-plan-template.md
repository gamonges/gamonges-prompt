# fix-plan.md 出力テンプレート

> SKILL.md フェーズ 5 で生成する `./tmp/fix-plan.md` のテンプレート。
> Cursor 環境の `.cursor/plan.md` 等もこのフォーマットを踏襲する。

## Claude Code 出力フォーマット

```markdown
# 修正計画: [対象機能名]

## 概要
[修正の背景と目的]

## 前提条件
[修正に必要な前提知識・環境]

## 修正サマリー

| 優先度 | 件数 |
|--------|------|
| P1 Critical | X件 |
| P2 High | Y件 |
| P3 Medium | Z件 |
| P4 Low | W件 |

## 実装ステップ

### ステップ 1: [修正タイトル] [P1]
- **種別**: バグ修正 / 実装漏れ / 品質改善 / テスト追加
- **出典**: [fixes.md / review.md Critical C-1 / ギャップ分析]
- 変更対象ファイル: `src/path/to/file.ts:L42-50`
- テスト方針: [追加・修正すべきテストケース]
- 詳細: [現在の問題と具体的な修正内容]
- 影響範囲: [修正による波及範囲]

### ステップ 2: [修正タイトル] [P2]
...

## テスト方針
[全体のテスト戦略と修正後の動作確認方法]

[修正全体が影響を与えるモジュール・機能]
```

## Cursor 出力フォーマット

plan モードで実行し、`.cursor/plan.md` または指定された plan ファイルに修正計画として書き出す。各修正項目をステップとして構造化し、Cursor の実装フローに乗せられる形式にする。

既存の plan ファイルがある場合は修正ステップを追記、なければ上記 Claude Code と同じフォーマットで新規作成。

## TDD 方針の組込み

- **バグ修正**: 必ず先に失敗するテストを書く（リグレッションテスト）
- **実装漏れ**: 期待動作のテストを先に書く
- **品質改善**: テスト改変が必要なら明示
- **テスト追加**: テスト自身がステップになる

## 完了報告フォーマット

```
修正計画を生成しました: ./tmp/fix-plan.md

修正サマリー:
- P1 Critical: X件（即時対応）
- P2 High: Y件
- P3 Medium: Z件
- P4 Low: W件

最優先対応:
1. ステップ 1: [概要] (src/path/file.ts:L42)
2. ステップ 2: [概要] (src/path/file.ts:L100)

次のステップ:
- /implement ./tmp/fix-plan.md — 修正を実行する
- /revise ./tmp/fix-plan.md — 修正計画を調整する（省略可）
```

## 注意事項

- `./tmp/fix-plan.md` の差分はコミットしない（`tmp/` 配下はリポジトリ管理対象外）
- Cursor 環境の `.cursor/plan.md` の扱いはユーザー判断
- 既存 fix-plan.md がある場合の追記/上書き判断は SKILL.md フェーズ 1.5 を参照

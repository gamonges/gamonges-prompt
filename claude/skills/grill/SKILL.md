計画・設計・前提を、共有理解に達するまで**一問ずつ**詰める。意思決定ツリーを上流から辿り、各分岐でまず「決めるために必要な情報」を洗い出し、前提が揃った分岐だけを決定へ進める。

**規約**: CLAUDE.md の Skills 共通規約に従う。

**出力先**: 進行中の状態は `./tmp/grill.md` に集約する（コミット対象外）。ソースコードは変更しない。

## 入力

`$ARGUMENTS` で grill 対象のコンテキストファイルを指定できる（**任意**）。省略時は `./tmp/plan.md` → `./tmp/context.md` の順で存在するものを初期材料に使う。いずれも無ければ、最初の1問で「何を grill するか」を聞いて開始する（`/design` のような即停止はしない）。

既存の `./tmp/grill.md` がある場合は中身を読み、**同一題材なら resume（in-place で継続）／別題材なら上書き前に1問確認**する。不在なら新規作成する。

## コア原則

- 質問は一度に1つだけ出す。まとめて複数問わない。
- ユーザーに聞く前に、repo / docs / diff / config で分かることは自分で探索して確認する。単一の探索や1ファイルで完結する確認は直接行い、複数の独立した確認が要る場合は同一ターンで並列に調べる。
- **情報のブレ防止**: 確定した事実は `./tmp/grill.md` の「確定事項」を単一の真実源とし、以後はそこを参照・更新する。会話の流れで再解釈しない。
- **コンテキスト肥大化の防止**: チャット出力は「直前に確定した1行」＋「次の1問」に絞る。全体像は毎ターン再掲せず `./tmp/grill.md` に外出しする。
- あいまいな回答はそのまま進めず、より具体的な次の1問で絞る。
- おすすめ回答は押し付けず、判断理由を短く添える。
- 設計の選択肢を複数提示するときは、各案に **pros/cons を表で必ず併記**する。選ばせるのが目的でなく、判断に必要な情報を揃えるのが目的。
- `AskUserQuestion` ツールは設計案を選ばせる用途には使わない。タスク所有者の振り分け・優先度確認など非設計の分岐に限って使う。

## フェーズ遷移（per-branch gate）

各意思決定ノードで `Phase 1 → gate → Phase 2` を辿る。

- **Phase 1（情報の洗い出し）**: 「この分岐を決めるには何を知る必要があり／誰が持ち／どう計測・取得するか」を問う。
- **gate**: 決定に必要な前提事実がその場で揃ったかを判定する。
- **揃った場合 → Phase 2（決定前進）**: 推奨する設計判断を「おすすめ回答」として提示する。
- **揃わない場合 → 保留**: 「情報要求マップ」に記録し、その分岐の決定は保留したまま次の分岐へ進む。

前提が揃う前に決定を迫らない。

gate の結果は「直前の確定1行」に畳み込む。前進時は確定行に含め、保留時は「〈不足情報〉待ちで保留 → 情報要求マップに記録」の1行だけ示す。「Phase 1 に入ります」等のフェーズ実況はしない。

## 質問フォーマット

```
質問: …

おすすめ回答: …
```

- **Phase 1 のおすすめ回答** = まず取るべき情報（項目 × 取得元 × 計測・取得方法）。決定ではない。
- **Phase 2 のおすすめ回答** = 推奨する設計判断とその根拠。複数案があれば pros/cons 表を添える。

必要なら、質問の前に1–2行だけ現在の整理を添えてよい。

## 観点（条件付きトリガー）

変更内容が実際に触れる観点だけを質問として立てる。純技術的な内部変更で、業務観点を機械的に全件問わない。**業務・技術のどちらを強く掘るかは変更の性質で都度判断する**（優先度を固定しない）。

- **業務・運用**: 顧客影響 / 権限 / データ移行 / 既存顧客 / 請求・会計・監査 / サポート・運用負荷。
- **セキュリティ・個人情報**: 認証・認可 / 個人情報（PII）の取り扱い / データ保護・漏洩時の影響。
- **技術**: API / data flow / state / failure mode / migration / rollout / test / 可観測性（監視・ログ・アラート）/ パフォーマンス・スケール。
- **整合性**: 既存コードベースの責務分界・命名・データモデル・運用パターンとの整合。技術的に実装可能でも、顧客体験・業務フロー・既存運用に対して不自然なら、その違和感を質問として扱う。
- **仮説の検証**: ユーザーの仮説について、根拠が薄い点・反例・未検証の前提・代替案を一問ずつ確認する。

## 成果物: `./tmp/grill.md`（OKF 準拠・単一ファイル）

[Open Knowledge Format](https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing) の単一概念ドキュメントとして生成する。先頭に YAML frontmatter を置き、本文に4ブロックを持つ。毎ターン in-place で更新し、履歴（`log.md` 相当）は貯めない。

frontmatter（`type` のみ OKF 必須、他は任意）:

```yaml
---
type: Grill Session
title: <題材>
description: <このセッションで詰める対象の要約>
tags: [grill, ...]
timestamp: <最終更新日時 ISO8601>   # in-place 更新のたびに現在時刻へ上書き
---
```

本文の4ブロック:

1. **確定事項** — 決定した分岐とその根拠。以後の単一の真実源。
2. **情報要求マップ** — 未取得の分岐。`項目 × 取得元（redash / PdM / インフラチーム / … ） × 計測・取得方法` の表。
3. **分岐ツリー** — 残っている分岐と依存関係、現在地マーク。
4. **未決事項** — 保留中の分岐と保留理由。

コード参照や概念間リンクは `file_path:L{number}` / markdown リンクで記す。

## ワークフロー

1. ユーザーの計画・設計・前提を短く要約し、未確定の意思決定を洗い出して `./tmp/grill.md` を初期化する（既存ファイルの扱いは「入力」節を参照）。
2. repo / docs / diff / config で答えられる事実は、質問せず探索して埋める。
3. 分岐を依存関係で整理し、上流の決定から順に着手する。
4. 各分岐で Phase 1 → gate → Phase 2 を一問ずつ進める。回答を受けたら、確定したこと・新しく分岐したことを `./tmp/grill.md` に反映し（in-place 更新）、チャットには「直前の確定1行 ＋ 次の1問」だけを返す。
5. 終了時は「確定事項」と「未決事項」を分けて簡潔にまとめ、末尾に次工程「`/design tmp/grill.md`」を案内する（`tmp/grill.md` はそのまま `/design` の `$ARGUMENTS` になる）。

## 終了・継続の判断

次の優先順位で判断する:

1. **ユーザーの明示中断が最優先** — 「まとめて」「stop」等が来たら、その時点で即「確定事項／未決事項」を出力して終了する。
2. **達成条件があれば満たすまで継続** — 会話でユーザーが明示した達成条件、または `./tmp/goal.md`（あれば読む）が満たされるまで grill を続け、skill 起点の停止提案は抑制する。特定の `/goal` 実装には密結合せず、「達成条件があれば従う」疎結合の契約とする。
3. **達成条件がなければ自発的に停止提案** — 主要な分岐が解決した、または残る分岐がすべて外部情報待ちでブロックされたら、停止してまとめることを提案する。「主要な分岐」＝他の分岐や実装方針への依存・影響が大きい分岐。leaf の細部詰めはしない。ハードな質問数上限は設けない。

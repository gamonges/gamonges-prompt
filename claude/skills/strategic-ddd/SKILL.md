`/design`（戦術設計・実装計画）の前工程。具体的な物理設計に入る前に、ドメインの概念レベルの整理をユーザーとの対話を通じて行う。

**出力**: `./tmp/strategic-ddd.md`（Phase ごとに段階的に追記）
**入力**: 機能要件（チケット / FigJam / デモ画面）+ 既存コードベース
**規約**: CLAUDE.md の Skills 共通規約に従う
## ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| `/ask` 調査 | `/strategic-ddd` 概念設計 | `/design` 実装計画 |

## 前提条件

- `./tmp/research.md` が存在する場合、調査結果を入力として参照する
- プロジェクトのアーキテクチャルール（`.cursor/rules/architecture.mdc` 等）を読み、レイヤードアーキテクチャ・CQRS パターンを把握する
- FigJam URL が提供された場合、コンテキスト境界図の入力として活用する

## 進行方式

このスキルは**対話的な壁打ちガイド**として動作する。以下のルールに従う。

### 1. Phase ごとの実行

1回の応答で **1 Phase のみ** 実行し、ユーザーの応答を待つ。全 Phase を一括実行しない。

### 2. 各 Phase の進行テンプレート

各 Phase では以下の順序で進行する:

1. **前 Phase からの引き継ぎ**: 前の Phase で合意した内容を簡潔に要約し、今回の Phase との接続を示す
2. **仮説の提示**: コードベース分析・要件から仮説を立てて提示する
3. **選択肢の提示**: 2〜3個の選択肢とトレードオフを提示する
4. **問いかけ**: ユーザーのドメイン知識を引き出す質問をする（後述の壁打ちポイント参照）
5. **合意の確認**: ユーザーの応答を受けて合意内容を確定し、`./tmp/strategic-ddd.md` に追記する

### 3. 段階的出力

- Phase 1 完了時: `./tmp/strategic-ddd.md` を新規作成（ヘッダー + Phase 1 の合意内容）
- Phase 2〜6 完了時: 既存ファイルに該当セクションを追記
- 中断しても、完了した Phase までの成果が残る

### 4. 状態管理と復帰

`./tmp/strategic-ddd.md` が既に存在する場合、最後に記述された Phase の次から再開する。ユーザーに「Phase N まで完了しています。Phase N+1 から再開します」と伝える。

### 5. ナビゲーション

- ユーザーが「次へ」「進めて」等と応答 → 次の Phase に進む
- ユーザーが「戻って」「Phase N をやり直し」→ 該当 Phase を再実行し、`strategic-ddd.md` の該当セクションを更新する
- ユーザーが短く「OK」「それでいい」→ 深掘りをスキップして次の Phase へ進む

## ワークフロー

### Phase 1: 問題空間のスコーピング

対象機能が扱うサブドメインを分類し、設計のスコープを明確にする。

| 分類 | 判断基準 |
|------|---------|
| **コアドメイン** | 本機能の価値の核心。独自のビジネスロジックがある |
| サポーティング | 既存基盤の利用。新規ロジックは薄い |
| ジェネリック | 汎用的な処理（集計、通知等） |

**中心となるドメイン問い**を1文で言語化する。例: 「あるアカウントが、どの程度の確信度で "休眠" と言えるか？」

#### 壁打ちポイント

- **オープンQ**: 「この機能が解決しようとしている、最も根本的なビジネス課題は何ですか？」
- **仮説挑戦**: 「これをコアドメインに分類しましたが、既存の○○で代替できる可能性はありますか？」
- **具体例要求**: 「この機能がないことで、実際にどんな業務上の困りごとが発生していますか？」

### Phase 2: ドメインイベントの洗い出し

**前 Phase からの引き継ぎ**: Phase 1 で定義したスコープとドメイン問いを起点に、「このドメインで何が起きるか」を洗い出す。

イベントストーミングの手法を用いて、ドメインで発生するイベントを時系列に並べる。

`| イベント | 発生条件 | 消費者 | 副作用 |` のテーブルで記述する。

- 消費者と副作用を必ず明記する
- 既存の EventBridge → SQS パターンとの整合を確認する
- まず網羅的に洗い出し、後で整理する（この段階では完璧を目指さない）

#### 壁打ちポイント

- **オープンQ**: 「この業務フローで、最も頻繁に発生するイベントと最も例外的なイベントはどれですか？ それによって設計の優先度が変わります」
- **仮説挑戦**: 「○○イベントを挙げましたが、これは本当に独立したイベントですか？ △△の一部ではないですか？」
- **具体例要求**: 「このイベントが発生した後、実際に誰が何をする必要がありますか？」

### Phase 3: コマンドと集約の発見

**前 Phase からの引き継ぎ**: Phase 2 で洗い出したイベントを元に、各イベントを引き起こすコマンドを特定し、コマンドの対象となる集約を発見する。

#### 3.1 コマンドの特定

`| コマンド | 発行者 | 対象 | 意味 |` のテーブルで記述する。

コマンドハンドラは 1 Handler = 1 Execute パターン（既存規約）。

**「動詞か名詞か」テスト**: 「除外する」「停止する」は動詞（Command の結果）。Command 結果を別 Entity にせず、対象 Entity のフィールドとして表現するのが本プロジェクトの標準パターン。

参照実装:
- `ShadowItApp.deletedAt` — 手動削除マーカーを Entity フィールドに保持
- `AppIntegrationFeature.forceRefresh` — 管理者オーバーライドを Entity フィールドに保持

#### 3.2 集約の特定

コマンドの対象（= 集約候補）を以下の3つの基準で判定する:

1. **独立した同一性（ID）を持つか** — PK が他 Entity と同じなら集約ではない
2. **不変条件を守る責務があるか** — 単なるデータ保存先は集約ではない（永続化モデル）
3. **独立したライフサイクルか** — バッチ再計算で上書きされるなら独立していない

#### 3.3 概念の分類

特定した概念を以下の5分類に振り分ける:

| 分類 | 判断基準 |
|------|---------|
| **集約ルート** | 独立した同一性 + 不変条件 + 独立ライフサイクル |
| **永続化モデル** | 同一性あるが集約の不変条件を持たない。計算結果の保存先 |
| **Domain Service** | ステートレスな判定ロジック。namespace + 純粋関数（ShadowEvaluationService パターン） |
| **Value Object** | 属性の組み合わせで定義。Prisma enum として実装されることが多い |
| **ReadModel** | 永続化しない。QueryService 層で導出。UI 表示用の合成概念 |

**MECE チェック**: enum を設計したら、その値が MECE（相互排他・網羅的）かを検証する。MECE にならない場合（複数の直交する軸がある場合）、ReadModel での導出を検討する。

#### 壁打ちポイント

- **オープンQ**: 「この集約候補の中で、最もビジネスルールが複雑なものはどれですか？」
- **仮説挑戦**: 「X を集約として提案しましたが、X なしで同じ不変条件を守れる別の方法はありますか？ もしあれば永続化モデルかもしれません」
- **具体例要求**: 「この不変条件が破られた場合、実際にどんな問題が起きますか？」

### Phase 4: ユビキタス言語の整理

**前 Phase からの引き継ぎ**: Phase 2-3 の議論で使用した用語を整理し、チーム内で共通認識を持てる用語集を作る。

Phase 2（イベント）と Phase 3（コマンド・集約）の議論で自然に出てきた用語を整理する。先に辞書を作るのではなく、議論を通じて浮かび上がった用語を定義する。

- 既存コードベースの用語との差異を確認する（例: `INACTIVE` vs `DORMANT` の違い）
- 曖昧な用語・同義語を整理する
- 各用語は `| 用語 | 定義 | 補足 |` のテーブルで記述する

**チェック**: 既存概念（CategorizedAppUser, Worker, AppUser 等）と新規概念の関係を明確にする。

#### 壁打ちポイント

- **オープンQ**: 「ここまでの議論で使った用語の中で、チーム内で解釈が分かれそうなものはありますか？」
- **仮説挑戦**: 「○○と△△は同じ概念を指していませんか？ それとも意図的に区別していますか？」
- **具体例要求**: 「この用語を使って、ビジネスルールを1文で説明してもらえますか？ 不自然なら用語の定義を見直す必要があります」

### Phase 5: コンテキスト境界の特定

**前 Phase からの引き継ぎ**: Phase 2-3 で洗い出したイベント・コマンド・集約の塊から、コンテキスト境界を引く。Phase 4 で整理した用語の「言葉が変わるポイント」が境界のヒントになる。

#### 5.1 既存コンテキストの特定

本プロジェクトの主要コンテキスト:
- **it-force**: Software, AppUser, AppIntegration, ShadowIt, Access（TraceableApp, VisitHistory）
- **hr-force**: Worker, Person, WorkerTenure（BiTemporal）
- **shared-kernel**: Prisma 拡張、共通型

#### 5.2 新規コンテキストの定義

新機能が既存コンテキスト内に収まるか、新しいコンテキストが必要か判断する。Phase 2-3 で発見したイベントとコマンドの塊が、既存コンテキストのどこに属するかを検討する。

#### 5.3 関係パターンの選定

| パターン | 使用場面 | 本プロジェクトでの例 |
|---------|---------|-------------------|
| **Conformist** | Upstream のモデルをそのまま利用。同一チーム・安定した Upstream | Adapter 経由のモジュール間参照 |
| **Published Language** | Upstream が明示的な公開インターフェースを提供 | DomainEvent の型定義 |
| **Shared Kernel** | 共有部分が小さく安定。両チームが合意 | appUserId 等の ID 共有 |
| **ACL（Anti-Corruption Layer）** | 外部モデルが自ドメインと大きく異なる | 将来の外部 HR 連携等 |

ASCII テキスト図でコンテキストマップを描く（FigJam 転記用）。

#### 壁打ちポイント

- **オープンQ**: 「このイベントの塊をグループ分けするとしたら、どこに線を引きますか？」
- **仮説挑戦**: 「○○コンテキストと△△コンテキストを分けましたが、統合した方がシンプルになりませんか？ 分ける必然性は何ですか？」
- **具体例要求**: 「このコンテキスト境界を越える通信が発生するシナリオを具体的に教えてください」

> コンテキスト境界を横断する関係の結合品質（強度×距離×変動性）を確認したい場合は、`/coupling-precheck` を実行できる（本スキルは境界・概念の設計、`coupling-precheck` はその結合品質を扱う）。

### Phase 6: 未決定事項の整理

**前 Phase からの引き継ぎ**: Phase 1-5 の議論で浮かび上がった未解決の論点を整理し、戦術設計（`/design`）に引き継ぐ。

戦術設計（`/design`）に先送りする項目を明示する。

`| # | 項目 | 選択肢 | 判断基準 | 現時点の方針 |` のテーブルで記述する。

特に以下の観点を漏らさない:
- パフォーマンス設計（バッチのトランザクション範囲、ReadModel の導出コスト）
- コンテキスト境界を越える参照のコスト（他 Force のサービス呼び出し）
- 冪等性・エラー隔離の戦略

#### 壁打ちポイント

- **オープンQ**: 「ここまでの議論で、最も判断に迷った点はどこですか？」
- **仮説挑戦**: 「この項目を未決定としましたが、今の時点で仮決定しておいた方が `/design` がスムーズに進みませんか？」
- **具体例要求**: 「この未決定事項が解決しないまま実装に進んだ場合、最悪のシナリオは何ですか？」

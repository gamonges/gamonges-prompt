`/design`（戦術設計・実装計画）の前工程。具体的な物理設計に入る前に、ドメインの概念レベルの整理を行う。

**出力**: `./tmp/strategic-ddd.md`
**入力**: 機能要件（チケット / FigJam / デモ画面）+ 既存コードベース
**IMPORTANT**: Always respond in Japanese.

## ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| `/ask` 調査 | `/strategic-ddd` 概念設計 | `/design` 実装計画 |

## 前提条件

- `./tmp/research.md` が存在する場合、調査結果を入力として参照する
- プロジェクトのアーキテクチャルール（`.cursor/rules/architecture.mdc` 等）を読み、レイヤードアーキテクチャ・CQRS パターンを把握する
- FigJam URL が提供された場合、コンテキスト境界図の入力として活用する

## ワークフロー

### Phase 1: ドメインの問題空間

対象機能が扱うサブドメインを分類する。

| 分類 | 判断基準 |
|------|---------|
| **コアドメイン** | 本機能の価値の核心。独自のビジネスロジックがある |
| サポーティング | 既存基盤の利用。新規ロジックは薄い |
| ジェネリック | 汎用的な処理（集計、通知等） |

**中心となるドメイン問い**を1文で言語化する。例: 「あるアカウントが、どの程度の確信度で "休眠" と言えるか？」

### Phase 2: ユビキタス言語

対象機能の主要用語を定義する。

- 既存コードベースの用語との差異を確認する（例: `INACTIVE` vs `DORMANT` の違い）
- 曖昧な用語・同義語を整理する
- 各用語は `| 用語 | 定義 | 補足 |` のテーブルで記述する

**チェック**: 既存概念（CategorizedAppUser, Worker, AppUser 等）と新規概念の関係を明確にする。

### Phase 3: コンテキスト境界とマップ

既存モジュール（Force 単位）と新機能のコンテキスト境界を定義する。

#### 3.1 既存コンテキストの特定

本プロジェクトの主要コンテキスト:
- **it-force**: Software, AppUser, AppIntegration, ShadowIt, Access（TraceableApp, VisitHistory）
- **hr-force**: Worker, Person, WorkerTenure（BiTemporal）
- **shared-kernel**: Prisma 拡張、共通型

#### 3.2 新規コンテキストの定義

新機能が既存コンテキスト内に収まるか、新しいコンテキストが必要か判断する。

#### 3.3 関係パターンの選定

| パターン | 使用場面 | 本プロジェクトでの例 |
|---------|---------|-------------------|
| **Conformist** | Upstream のモデルをそのまま利用。同一チーム・安定した Upstream | Adapter 経由のモジュール間参照 |
| **Published Language** | Upstream が明示的な公開インターフェースを提供 | DomainEvent の型定義 |
| **Shared Kernel** | 共有部分が小さく安定。両チームが合意 | appUserId 等の ID 共有 |
| **ACL（Anti-Corruption Layer）** | 外部モデルが自ドメインと大きく異なる | 将来の外部 HR 連携等 |

ASCII テキスト図でコンテキストマップを描く（FigJam 転記用）。

### Phase 4: ドメインモデルの概念分類

#### 4.1 集約の特定

以下の基準で集約を絞り込む:

1. **独立した同一性（ID）を持つか** — PK が他 Entity と同じなら集約ではない
2. **不変条件を守る責務があるか** — 単なるデータ保存先は集約ではない（永続化モデル）
3. **独立したライフサイクルか** — バッチ再計算で上書きされるなら独立していない

**「動詞か名詞か」テスト**: 「除外する」「停止する」は動詞（Command の結果）。Command 結果を別 Entity にせず、対象 Entity のフィールドとして表現するのが本プロジェクトの標準パターン。

参照実装:
- `ShadowItApp.deletedAt` — 手動削除マーカーを Entity フィールドに保持
- `AppIntegrationFeature.forceRefresh` — 管理者オーバーライドを Entity フィールドに保持

#### 4.2 概念の分類

| 分類 | 判断基準 |
|------|---------|
| **集約ルート** | 独立した同一性 + 不変条件 + 独立ライフサイクル |
| **永続化モデル** | 同一性あるが集約の不変条件を持たない。計算結果の保存先 |
| **Domain Service** | ステートレスな判定ロジック。namespace + 純粋関数（ShadowEvaluationService パターン） |
| **Value Object** | 属性の組み合わせで定義。Prisma enum として実装されることが多い |
| **ReadModel** | 永続化しない。QueryService 層で導出。UI 表示用の合成概念 |

**MECE チェック**: enum を設計したら、その値が MECE（相互排他・網羅的）かを検証する。MECE にならない場合（複数の直交する軸がある場合）、ReadModel での導出を検討する。

### Phase 5: 振る舞いの定義

#### 5.1 ドメインイベント

`| イベント | 発生条件 | 消費者 | 副作用 |` のテーブルで記述する。

消費者と副作用を必ず明記する。既存の EventBridge → SQS パターンとの整合を確認する。

#### 5.2 コマンド

`| コマンド | 発行者 | 対象 | 意味 |` のテーブルで記述する。

コマンドハンドラは 1 Handler = 1 Execute パターン（既存規約）。

### Phase 6: 未決定事項の整理

戦術設計（`/design`）に先送りする項目を明示する。

`| # | 項目 | 選択肢 | 判断基準 | 現時点の方針 |` のテーブルで記述する。

特に以下の観点を漏らさない:
- パフォーマンス設計（バッチのトランザクション範囲、ReadModel の導出コスト）
- コンテキスト境界を越える参照のコスト（他 Force のサービス呼び出し）
- 冪等性・エラー隔離の戦略

## レビューチェックリスト

出力完了後、以下を自己チェックする:

- [ ] enum を設計した場合、MECE を検証したか
- [ ] 「動詞か名詞か」テストで Entity と Command を区別したか
- [ ] 既存コードベースのパターン（ShadowIt, AppIntegration 等）と整合するか
- [ ] コンテキスト境界を越える関心事の持ち込みがないか（例: HR-Force の概念を IT-Force に混入）
- [ ] 各集約の不変条件を定義したか
- [ ] ドメインイベントの消費者と副作用を明記したか
- [ ] 暗黙の除外ケース（NO_SIGNAL 等のエッジケース）を明示したか
- [ ] ReadModel で導出すべき概念と永続化すべき概念を区別したか

---
description: 実装後の修正項目から修正計画（fix-plan.md）を生成する（/fix）
---

実装後のレビューで発見した問題や修正項目をもとに、修正実装計画を生成する。

**重要**: `./tmp/fix-plan.md` もしくはCursorのplanモードで作成した`***.plan.md`のみを作成・変更する。ソースコードは変更しない。
**IMPORTANT**: Always respond in Japanese.

## ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| `/implement` 実装完了 + `/review` or 手動レビュー | `/fix` 修正計画作成 | `/implement ./tmp/fix-plan.md` 修正実行 |

## パラメーター

`$ARGUMENTS` には以下のいずれかを指定できる。省略時は `./tmp/fixes.md` を使用する。

- **修正項目ファイルのパス**: `./tmp/fixes.md` 等のローカルファイル
- **PR の URL または番号**: `https://github.com/org/repo/pull/123` または `123`

`$ARGUMENTS` が PR URL/番号の場合、PR 上の未解決レビューコメントを取得し、修正項目として取り込む。

`$ARGUMENTS` の種別に関わらず、`./tmp/review/unified.md` が存在すれば併せて参照する。

## 実行条件

以下の条件を確認する:

1. **修正項目の入力が存在すること**
   - `$ARGUMENTS` で指定されたファイル、または `./tmp/fixes.md` が存在する
   - **または** `$ARGUMENTS` が PR URL/番号であり、該当 PR が存在しアクセス可能である
   - **または** `./tmp/review/unified.md` が存在する
   - いずれも該当しない場合: 即座に停止し、ユーザーに報告する

2. **`./tmp/plan.md` の存在**（ギャップ分析に使用）
   - 存在しない場合: 警告を出すが処理は続行する

いずれかの必須条件が満たされない場合:

- プロセスを即座に停止する
- 満たされていない条件を具体的にユーザーへ報告する
- 処理を続行しない

## `./tmp/fixes.md` の推奨フォーマット

ユーザーはレビュー中に以下のフォーマットで修正項目を記録する（自由形式も受け付ける）:

```markdown
# 修正項目

## 修正 1: [修正タイトル]
- **ファイル**: `src/path/to/file.ts:L42`
- **問題**: [何が問題か]
- **期待する動作**: [どうあるべきか]

## 修正 2: [修正タイトル]
...
```

区切り文字 `=====` による列挙形式も許容する。

## 実行プロセス

### フェーズ 1: 入力の収集と前処理

#### 1.1 `$ARGUMENTS` の種別判定

`$ARGUMENTS` の内容を解析し、以下のいずれかに分類する:

| パターン | 種別 | 例 |
|----------|------|-----|
| `https://github.com/...` を含む | PR URL | `https://github.com/org/repo/pull/123` |
| 数字のみ | PR 番号 | `123` |
| 上記以外 | ファイルパス | `./tmp/fixes.md` |
| 未指定 | デフォルト | `./tmp/fixes.md` を使用 |

#### 1.2 修正項目の読み込み

以下を読み込む:

- [ ] `$ARGUMENTS` がファイルパスの場合: 該当ファイルを読み込む（デフォルト: `./tmp/fixes.md`）
- [ ] `$ARGUMENTS` が PR URL/番号の場合: PR 上の未解決レビューコメントを取得する（後述の 1.3 参照）
- [ ] `./tmp/review/unified.md` — `/review` 出力（存在する場合のみ）
- [ ] `./tmp/plan.md` — 元の実装計画（存在する場合のみ）

#### 1.3 PR レビューコメントの取得（`$ARGUMENTS` が PR URL/番号の場合）

`$ARGUMENTS` が PR URL または番号の場合、PR 上の未解決レビューコメントを取得して修正項目に含める。

```bash
# PR の基本情報を取得
pr_url="$ARGUMENTS"
gh pr view "$pr_url" --json number,title,baseRefName,headRefName,author,state
```

```bash
# GraphQL で未解決のレビュースレッドを取得
gh api graphql -f query='
query($owner: String!, $repo: String!, $number: Int!) {
  repository(owner: $owner, name: $repo) {
    pullRequest(number: $number) {
      reviewThreads(first: 100) {
        nodes {
          isResolved
          isOutdated
          path
          line
          startLine
          comments(first: 50) {
            nodes {
              author {
                login
              }
              body
              createdAt
              url
              originalPosition
              diffHunk
            }
          }
        }
      }
    }
  }
}' -f owner="$owner" -f repo="$repo" -F number="$number"
```

**フィルタリングルール**:

1. **resolved 済みを除外**: `isResolved: true` のスレッドはスキップする
2. **bot コメントを除外**: コメント作成者の login が以下に該当するものはスキップする
   - `[bot]` を含むユーザー名（例: `github-actions[bot]`, `dependabot[bot]`）
   - `copilot`、`gemini`、`coderabbit` など既知の AI レビュー bot
3. **outdated スレッドの扱い**: `isOutdated: true` のスレッドは取得するが、outdated であることを明記する

取得した各コメントを修正項目として統合する。コメントの `path` と `line` をファイル参照として、`body` を問題の説明として扱う。

#### 1.4 現在の実装状態の把握

```bash
# 実装済み変更の確認
git diff HEAD --stat
git log --oneline -10
```

`./tmp/plan.md` が存在する場合、元の計画との差分を分析する（計画に含まれているが実装されていない項目を特定する）。

#### 1.5 修正項目の統合と分類

収集した入力から修正項目を統合する:

- `./tmp/fixes.md` の明示的な修正指示（存在する場合）
- PR 上の未解決レビューコメント（PR URL/番号が指定された場合）
- `./tmp/review/unified.md` の ❌ Critical Issues と ⚠️ Minor Issues（存在する場合）
- `./tmp/plan.md` との比較で特定した未実装・実装漏れ項目（存在する場合）

各修正項目を以下に分類する:

| 種別 | 説明 |
|------|------|
| **バグ修正** | 動作が仕様・意図と異なる誤りの修正 |
| **実装漏れ** | plan.md に定義されたが実装されなかった項目 |
| **品質改善** | コード品質・可読性・型安全性の向上 |
| **テスト追加** | カバレッジ不足や漏れたエッジケースへの対応 |

重複項目は統合し、出典を記録する（例: `[fixes.md + review.md]`）。

### フェーズ 2: ユーザー確認（必要に応じて）

以下の問題を検出した場合、**プロセスを即座に停止**してユーザーに報告する:

- **修正意図が不明**: 修正項目の「何をどう直すか」が複数解釈可能な場合
  - 具体的にどの項目が不明確かを提示し、確認を求める
- **修正間の矛盾**: 複数の修正項目が互いに矛盾する場合
  - 矛盾する項目を並べて提示し、優先度の判断を依頼する
- **影響範囲が不明確**: 修正が他のコンポーネントへ大きく波及する可能性がある場合
  - 波及範囲を提示し、対応方針を確認する

複数の問題がある場合のフォーマット:

```
## 問題 1: [問題の要約]
[該当する修正項目の引用と、何が問題なのかの説明]

## 問題 2: [問題の要約]
[該当する修正項目の引用と、何が問題なのかの説明]
```

### フェーズ 3: コードベース調査

各修正項目について、サブエージェントを使って**並列に**調査を実施する:

- [ ] **該当コードの確認**: 指摘されたファイル・行番号の現在のコードを読む
- [ ] **関連コードの調査**: Serena MCP でセマンティック検索を行い、修正の影響範囲を特定する
- [ ] **既存パターンの確認**: 修正後のコードが従うべき既存実装パターンを特定する
- [ ] **外部情報の調査**: Web search tool で外部ドキュメントや事例を検索する
- [ ] **ライブラリ情報**: Context7 MCP でライブラリ・フレームワークの使用方法を確認する
- [ ] **テスト状況の確認**: 修正対象のテストカバレッジ状況を確認する

### フェーズ 4: 修正計画の策定

各修正項目について:

- [ ] 修正の具体的な手順を決定する
- [ ] 修正による影響範囲を評価する
- [ ] TDD 方針を決定する（バグ修正・実装漏れには必ずテストを先に追加）
- [ ] 修正の依存関係と実行順序を決定する（依存関係のある修正は順序を明示する）
- [ ] 優先度を決定する:
  - **P1 (Critical)**: 動作バグ・データ不整合・セキュリティ問題
  - **P2 (High)**: 実装漏れ・型安全性の違反
  - **P3 (Medium)**: 品質改善・リファクタリング
  - **P4 (Low)**: コメント整理・スタイル調整
- [ ] **要件カバレッジを検証する**: すべての修正項目が修正ステップでカバーされているか確認し、漏れがあれば追加する

### フェーズ 5: 修正計画の出力

#### Claude Code の場合

`./tmp/fix-plan.md` に以下の構成で修正計画を出力する:

```markdown
# 修正計画: [対象機能名]

## 概要
[修正の背景と目的]

## 前提条件
[修正に必要な前提知識・環境]

## 修正サマリー

| 優先度 | 件数 |
|--------|------|
| P1 Critical | X件 |
| P2 High | Y件 |
| P3 Medium | Z件 |
| P4 Low | W件 |

## 実装ステップ

### ステップ 1: [修正タイトル] [P1]
- **種別**: バグ修正 / 実装漏れ / 品質改善 / テスト追加
- **出典**: [fixes.md / review.md Critical C-1 / ギャップ分析]
- 変更対象ファイル: `src/path/to/file.ts:L42-50`
- テスト方針: [追加・修正すべきテストケース]
- 詳細: [現在の問題と具体的な修正内容]
- 影響範囲: [修正による波及範囲]

### ステップ 2: [修正タイトル] [P2]
...

## テスト方針
[全体のテスト戦略と修正後の動作確認方法]

## 影響範囲
[修正全体が影響を与えるモジュール・機能]

## 注意事項
[修正実施時に気をつけるべきアーキテクチャ・規約上の制約]
```

#### Cursor の場合

plan モードで実行し、`.cursor/plan.md` または指定された plan ファイルに修正計画として書き出す。
各修正項目をステップとして構造化し、Cursor の実装フローに乗せられる形式にする。

既存の plan ファイルが存在する場合は、修正ステップを既存の計画に追記する形式で更新する。
存在しない場合は、上記の Claude Code と同じフォーマットで新規作成する。

### フェーズ 6: コミット

```bash
# Claude Code の場合
git add ./tmp/fix-plan.md
git commit -m "docs: 修正計画を作成"

# Cursor の場合
git add .cursor/plan.md  # または該当する plan ファイル
git commit -m "docs: 修正計画を作成"
```

コミットに失敗した場合はユーザーに報告し、手動での対応を依頼する。

### フェーズ 7: 完了報告

以下の内容をユーザーに報告する:

- 修正項目の総数と優先度別の件数
- 入力ソース（fixes.md / review.md / PR コメント / ギャップ分析）の取込み結果
- 最優先で対応すべきステップ（P1 Critical の一覧）
- 次のステップの案内

```
修正計画を生成しました: ./tmp/fix-plan.md

修正サマリー:
- P1 Critical: X件（即時対応）
- P2 High: Y件
- P3 Medium: Z件
- P4 Low: W件

最優先対応:
1. ステップ 1: [概要] (src/path/file.ts:L42)
2. ステップ 2: [概要] (src/path/file.ts:L100)

次のステップ:
- /implement ./tmp/fix-plan.md — 修正を実行する
- /revise ./tmp/fix-plan.md — 修正計画を調整する（省略可）
```

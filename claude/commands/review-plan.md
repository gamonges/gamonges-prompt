---
description: 実装計画をスタッフエンジニアの視点でレビューする（/review-plan）
---

実装計画（plan.md）をスタッフエンジニアの視点でレビューし、実装前にアーキテクチャ整合性・リスク・実装順序・代替案を体系的に評価する。

**重要**: ソースコードは一切変更しない。`./tmp/plan-review.md` のみ出力する。
**重要**: Always respond in Japanese.

## パラメーター

`$ARGUMENTS` で実装計画ファイルのパスを指定できる。省略時は `./tmp/plan.md` を使用する。

## ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| `/design` 計画作成 | `/review-plan` 計画レビュー | `/revise` 計画修正 → `/implement` 実装 |

## 実行条件

以下のファイルが存在することを確認する:

- `$ARGUMENTS`（指定がない場合は `./tmp/plan.md`）— 実装計画

ファイルが存在しない場合:

- プロセスを即座に停止する
- どの条件が満たされていないかをユーザーに報告する
- レビューを続行しない

## スタッフエンジニアの思考フレームワーク

通常のコードレビュー（`/review`）とは異なり、以下の視座で計画を評価する:

- **システム思考**: 変更がシステム全体に与える影響（他モジュールへの波及、データフローの変化）
- **長期的視点**: 技術的負債の蓄積、将来のメンテナンスコスト、拡張性への影響
- **横断的関心事**: セキュリティ、パフォーマンス、オブザーバビリティ、障害耐性
- **組織的影響**: チーム境界をまたぐ変更、必要な知識・スキルの有無
- **プラグマティズム**: 適切な複雑度か（オーバーエンジニアリング / アンダーエンジニアリング）

## 指摘の重要度

| レベル | 意味 |
|--------|------|
| ❌ Critical | 計画の修正が必須。このまま実装すると問題が生じる |
| ⚠️ Warning | 修正を推奨。品質・保守性に影響する |
| ℹ️ Note | 検討・確認を推奨。判断はユーザーに委ねる |
| 💡 代替案 | 検討に値する別アプローチの提案 |
| ✅ Good | 計画の良い点（積極的に評価する） |

## Finding Format Standard

**すべての指摘で以下の形式を使用する:**

```
- `plan.md:L42` — 指摘内容
```

- 計画内の記述への参照は `plan.md:L{number}` 形式
- コードベース内のファイルへの参照は `src/full/path/to/file.tsx:L42` 形式（相対パス必須）
- 行番号が特定できない場合はセクション名で参照: `plan.md (ステップ 3)`

## 総合判定（3段階）

| 判定 | 意味 | 次のアクション |
|------|------|----------------|
| **LGTM** | 軽微な指摘のみ。実装に進んでよい | `/implement` |
| **要修正** | 指摘事項を反映してから実装すべき | `/revise ./tmp/plan-review.md` |
| **要再設計** | 根本的なアプローチの見直しが必要 | `/design`（再設計） |

## 実行プロセス

### フェーズ 1: 実装計画の読み込み

- [ ] 実装計画ファイル（`$ARGUMENTS` または `./tmp/plan.md`）を読み込む
- [ ] `./tmp/context.md` が存在する場合は、元の要件・コンテキストも読み込む
- [ ] 計画の全体構造を把握する:
  - 概要・目的
  - 実装ステップの数と依存関係
  - 変更対象ファイル一覧
  - テスト方針
  - 影響範囲

### フェーズ 2: コードベース調査

計画が前提としている内容を実コードと照合し、机上の空論を防ぐ。

- [ ] **ファイル・モジュールの存在確認**: 計画で参照されているファイルパスが実際に存在するか
- [ ] **既存パターンとの整合**: Serena MCP でセマンティック検索を行い、計画のアプローチが既存のコードパターンと整合するか
- [ ] **依存関係の検証**: 計画が前提としている依存関係（ライブラリ、モジュール間の関係）が正しいか
- [ ] **ライブラリ情報**: Context7 MCP で、計画が使用するライブラリ・API の使い方が正しいか確認（必要に応じて）

### フェーズ 3: 四観点レビュー

単一エージェントで以下の4観点を順次評価する。観点間の相互参照が重要なため、分割せず一貫した視点で行う。

#### 3.1 アーキテクチャ整合性

- [ ] 既存のアーキテクチャパターン（レイヤー構成、命名規則、ディレクトリ構造）と整合するか
- [ ] 新しいパターンを導入している場合、その必然性と一貫性は妥当か
- [ ] モジュール間の責務分担は明確か
- [ ] インターフェース設計（API 境界、型定義）は適切か

#### 3.2 リスク・見落とし検出

- [ ] エッジケースの考慮漏れはないか
- [ ] 障害シナリオ（ネットワークエラー、データ不整合、競合状態）への対処が計画されているか
- [ ] セキュリティ上の懸念（認証・認可、入力バリデーション、クロステナント）はないか
- [ ] パフォーマンスへの影響（N+1 クエリ、不要な再レンダリング、大量データ処理）は考慮されているか
- [ ] マイグレーション・デプロイ時のリスク（ダウンタイム、ロールバック手順）は考慮されているか

#### 3.3 実装順序・依存関係

- [ ] ステップ間の依存関係は正しく定義されているか
- [ ] 並列化可能なステップが直列になっていないか
- [ ] テストファーストのアプローチが適切に組み込まれているか
- [ ] 各ステップの粒度は適切か（大きすぎ / 小さすぎ）
- [ ] 途中で動作確認できるマイルストーンが設定されているか

#### 3.4 代替アプローチ

- [ ] よりシンプルな解決策が存在しないか
- [ ] 既存の機能・ユーティリティを活用できる箇所はないか
- [ ] ライブラリやフレームワークの標準機能で代替できる部分はないか
- [ ] YAGNI 原則に反する過剰な設計はないか

### フェーズ 4: レビューレポート生成

`./tmp/plan-review.md` に以下のテンプレートで出力する:

```markdown
# 実装計画レビュー

**対象**: {計画ファイルパス}
**レビュー日**: {date}
**総合判定**: {LGTM / 要修正 / 要再設計}

---

## 計画の概要

{レビュー対象の計画を簡潔に要約（2-3行）}

---

## ❌ Critical（修正必須）

### C-1. {問題タイトル}
**参照**: `plan.md:L{number}`

{詳細説明: 何が問題で、なぜ修正が必要か}

**推奨対応**:
{具体的な修正方法}

---

## ⚠️ Warning（修正推奨）

### W-1. {問題タイトル}
**参照**: `plan.md:L{number}`

{詳細説明}

**推奨対応**:
{具体的な修正方法}

---

## ℹ️ Note（検討推奨）

### N-1. {タイトル}
**参照**: `plan.md:L{number}`

{詳細説明}

---

## 💡 代替アプローチ

### A-1. {提案タイトル}
**関連**: `plan.md:L{number}`

{代替案の説明とトレードオフ}

---

## ✅ Good Points（計画の良い点）

1. **{タイトル}** — {説明}
2. ...

---

## 📊 サマリー

| 区分 | 件数 |
|------|------|
| ❌ Critical | X件 |
| ⚠️ Warning | Y件 |
| ℹ️ Note | Z件 |
| 💡 代替案 | W件 |
| ✅ Good | V件 |

---

## /revise への入力ガイド

以下の指摘を `/revise` に渡して計画を修正してください:

### 修正必須（Critical）
- C-1: {概要と修正方針}
- C-2: ...

### 修正推奨（Warning）
- W-1: {概要と修正方針}
- W-2: ...

実行コマンド:
\```
/revise ./tmp/plan-review.md
\```
```

### フェーズ 5: 完了報告

以下の内容をユーザーに報告する:

- **総合判定**（LGTM / 要修正 / 要再設計）
- 各区分の指摘件数
- 特に重要な指摘事項（Critical がある場合はその要約）
- 次のステップの案内:
  - **LGTM の場合**: `/implement` で実装開始
  - **要修正の場合**: `/revise ./tmp/plan-review.md` で計画を修正
  - **要再設計の場合**: `/design` で再設計

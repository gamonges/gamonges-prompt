## 実装計画ファイル

引数 `$ARGUMENTS` で指定したファイルを参照すること。

## テスト駆動開発ポリシー

- すべての実装はテストから書き始め、Red → Green → リファクタのサイクルを徹底する
- ドメイン層（entity/command/repository/value）を変更・追加するときは TDD を必須とする
- 複雑なビジネスロジック、クエリ、あるいは外部 I/F に依存した振る舞いを追加する場合も必ずテストを先に書く
- 既存テストでカバーできない場合は統合テストやエンドツーエンドテストを追加し、仕様を守る証拠として残す

### テスト品質検証（偽陽性・偽陰性の防止）

新規テストを書いた場合、以下の検証を必ず行い、テストが本当に実装を検証していることを担保する。

**偽陽性（False Positive）の検出** — テストが常に通ってしまい、バグを見逃す状態:
- **Red フェーズ検証**: テストを書いた直後（実装前）に実行し、**必ず失敗する**ことを確認する。実装前に通るテストは無価値
- **ミューテーションチェック**: Green 到達後、実装の核心ロジック（条件分岐・計算・戻り値など）を一時的に壊して、テストが**失敗に転じる**ことを確認する。壊しても通るテストはアサーション不足

**偽陰性（False Negative）の検出** — 正しい実装なのにテストが落ちる状態:
- **環境依存の排除**: テスト結果が実行順序・タイミング・外部状態に依存していないか確認する
- **期待値の妥当性**: ハードコードされた期待値が仕様に基づいているか、実装の偶然の出力をコピーしただけでないか確認する
- **過剰な厳密さの回避**: 本質でない詳細（オブジェクトの全プロパティ一致、メッセージ文言の完全一致など）に依存していないか確認する

### バグ修正時のリグレッションテスト必須化

バグ修正ステップでは、リグレッションテストを必須とする。

**検出条件（以下のいずれかに該当）:**
- plan.md のステップ説明に「バグ修正」「不具合修正」「fix」「bug」「regression」を含む
- `./tmp/fix-plan.md` から生成されたステップである
- ユーザーが「バグ修正」と明示した場合

※ 単独の「修正」はバグ修正以外（設計修正、テスト修正等）にも該当するため検出条件に含めない。
  判断が曖昧な場合は AskUserQuestion でユーザーに確認する。

**必須テストフロー:**
1. **再現テスト**: バグを再現するテストを最初に書き、**失敗する**ことを確認
   - テスト名に `regression` または `bug` を含める
   - バグの発生条件を明確にコメントで記述
2. **修正実装**: バグを修正し、再現テストが**通る**ことを確認
3. **回帰テスト**: 修正箇所の周辺機能の既存テストを実行し、デグレがないことを確認
   - 同一モジュール内のテストスイートを全実行
   - 修正ファイルを import しているファイルのテストも実行
4. **ミューテーションチェック**: 修正を元に戻し、再現テストが再び**失敗する**ことを確認

### TDD ユーザーチェックポイント

テスト作成後にユーザー確認を挟み、テストが仕様を正しく表現しているか検証する。

**適用範囲（以下に該当するテストのみ）:**
- ドメイン層（entity/command/repository/value）のビジネスロジックテスト
- 仕様の解釈が曖昧で、複数の実装が考えられるテスト
- ユーザーが明示的にチェックを要求した場合

**適用しない:**
- CRUD の基本操作テスト
- 既存パターンを踏襲するテスト
- リファクタリング時の既存テスト維持

**フロー:**
① テストを作成する（Red フェーズ前）
② AskUserQuestion でテストコードをユーザーに提示する
   提示内容:
   - テストファイルのパスと作成したテストの要約
   - テストが検証する仕様項目
   - 「このテストは仕様を正しく表現していますか？」
③ ユーザーの応答:
   - 承認 → Red フェーズに進む
   - 修正指示 → テストを修正し、再度 ② に戻る

## レビュー・修正サイクルポリシー

**重要**: 各実装ステップ完了後、必ず **レビュー → 修正** を **2回繰り返す** こと。

### レビューサイクルの手順

```
ステップ実装完了
    ↓
【1回目レビュー】セルフレビュー実行
    ↓
【1回目修正】指摘事項を修正
    ↓
【2回目レビュー】再度セルフレビュー実行
    ↓
【2回目修正】残存指摘事項を修正
    ↓

実装計画書に記載しているTODOリストに完了マークをつけて次のステップへ
```

### レビュー観点チェックリスト

各レビューでは以下の観点を確認する：

1. **コード品質**
   - 型安全性（any 型の使用禁止）
   - 命名規則の遵守
   - 単一責任原則の遵守
   - 不要なコメントの削除

2. **エラーハンドリング**
   - サイレント失敗がないか
   - 適切なエラーメッセージ
   - エッジケースの考慮

3. **プロジェクト規約**
   - CLAUDE.md の規約遵守
   - .cursor/rules/ の規約遵守
   - 既存コードパターンとの整合性

4. **テスト**
   - テストカバレッジの確認
   - テストケースの網羅性
   - テストの可読性
   - 偽陽性: 実装を壊してもテストが通り続けないか（アサーション不足の兆候）
   - 偽陰性: 期待値が仕様に基づいているか、過剰な厳密さで脆くなっていないか

## 実行プロセス

### Phase 1: 実装計画の読込

- 実装計画ファイルを読み込む
- 両方のファイルからすべてのコンテキストと要件を理解する
- 実装計画のすべての指示に従い、必要に応じて調査を拡張する
- 実装計画を完全に実装するために必要なすべてのコンテキストを確保する
- 必要に応じて Web 検索とコードベース探索を追加で行う
- セマンティックなコード検索には Serena MCP を使用する
- 外部ドキュメントや例の検索には Web search tool を使用する
- ライブラリ・フレームワークの使用方法には Context7 MCP を使用する

### Phase 2: 設計と計画

- 計画を実行する前によく考える。すべての要件に対応する包括的な計画を作成する
- 複雑なタスクを小さく管理可能なフェーズに分解し、todos tools を使用する
- TodoWrite tool を使用して実装計画を作成・追跡する
- 既存コードから従うべき実装パターンを特定する
- TDD で書くべきテストの種類（単体/統合）と失敗パターンをプラン内で明示する
- **各ステップにレビューサイクル（2回）を組み込んだ計画を立てる**
- plan.md の各ステップに `- [ ]` チェックボックスがない場合は付与する

#### Phase 2.5: エージェント選択

プロジェクトタイプと変更内容に基づき、実装で使用する専門エージェントを決定する。

##### プロジェクトタイプ検出
1. `package.json` の dependencies（`@nestjs/core` → Backend, `react` → Frontend）
2. CLAUDE.md のキーワード（"NestJS" / "Prisma" → Backend, "React" / "TanStack" → Frontend）
3. ディレクトリ構造（`v2/src/` → Backend, `src/apps/` → Frontend）

##### エージェント選択マトリクス
| 変更内容 | 起動エージェント |
|---------|---------------|
| ドメイン層（entity/command/repository/value） | backend-developer |
| Prisma スキーマ / マイグレーション / SQL | database-administrator + sql-pro |
| API 層（controller/dto/usecase） | backend-developer + api-designer |
| フロントエンド UI コンポーネント | frontend-developer + ui-designer |

##### 差分規模によるスケーリング
- **Small**（ステップの変更ファイル < 3）: メインエージェントが直接実装。専門エージェント不要
- **Medium**（3-8 ファイル）: メインエージェントがタスク分割し、1-2 専門エージェントを起動
- **Large**（> 8 ファイル）: メインエージェントが分割し、複数専門エージェントを並列起動

### Phase 3: 実装の実行（ステップごとにレビューサイクル適用）

各ステップで以下のサイクルを実行する：

#### 3.0 コンテキスト再読込

各ステップの開始時に以下のファイルを再読込し、コンテキストドリフトを防止する:

1. **CLAUDE.md**: プロジェクト規約の最新状態を確認
2. **実装計画ファイル**: 現在のステップの詳細と前後の依存関係を再確認
3. **関連仕様ファイル**: openspec/specs/ 配下の該当仕様（存在する場合）
4. **前ステップの出力**: 直前ステップのレビュー結果や修正内容

#### 3.1 ステップ実装

- 実装計画のステップを実行する
- セマンティックなコード検索には Serena MCP を使用する
- 実装は Claude, GPT-5-Codex など使用モデルに依存しない汎用的な記法で行う

**Small ステップ（変更ファイル < 3）:**
- メインエージェントが直接 TDD サイクルを実行する（3.2 に進む）

**Medium/Large ステップ（変更ファイル >= 3）:**
1. メインエージェントが Phase 2.5 のエージェント選択マトリクスに基づきタスクを分割する
2. 各専門エージェントの担当ファイルを**排他的に**割り当てる（同一ファイルの並列編集は原則禁止）
3. 各エージェントを `run_in_background: true` で並列起動する
4. 各エージェントの出力: `./tmp/step-{n}-{agent-name}-result.md`
5. 全エージェント完了後、結果マージプロトコル（3.1.1）を実行する

##### 3.1.1 結果マージプロトコル

マルチエージェント並列実装の結果を安全にマージする:

1. **排他的ファイル割当の検証**: 各エージェントが割り当て外のファイルを変更していないか確認
2. **ファイル間整合性チェック**: import パス、型定義、インターフェースの一致を検証
3. **型チェック**: IDE の型エラー表示や `tsc --noEmit` の部分実行で確認（L-2: `pnpm run typecheck` のフル実行は避ける）
4. **テスト実行**: 変更箇所に関連するテストスイートを実行
5. **問題検出時**: メインエージェントが修正。修正不能な場合は該当エージェントを再起動

worktree 使用時の追加手順:
- 各 worktree の変更を `git diff` で確認
- メインブランチに `git merge --no-ff` で明示的にマージ
- コンフリクト発生時はメインエージェントが解決

#### 3.2 TDD サイクル

まずテスト（ドメイン層なら必ず）を追加・失敗させ、その後で最小の実装を加える。

**新規テストを書いた場合、以下のサイクルを実行する:**

```
① Red: 仕様に基づいてテストを作成する
   - plan.md / openspec のステップ仕様からテストケースを導出する

   【ユーザーチェックポイント — 適用対象テストのみ】
   TDD ユーザーチェックポイントポリシーの適用対象に該当する場合:
   AskUserQuestion でテストコードをユーザーに提示し、承認を得る。
   非該当の場合はスキップして ② に進む。

② Red 確認: テスト実行 → 失敗を確認
   └─ 通ってしまった場合 → 偽陽性。テストを見直す
      （アサーション漏れ、テスト対象の未呼出、モックの過剰設定など）

③ Green: 最小の実装を書く → 実行 → 成功を確認
   └─ 失敗する場合 → 偽陰性の可能性を検証
      （期待値の妥当性、環境依存、過剰な厳密さを確認）

④ Mutation: 実装の核心を一時的に壊す → テスト実行 → 失敗を確認
   └─ まだ通る場合 → 偽陽性。アサーションを強化する
   └─ 確認後、壊した実装を元に戻す

⑤ Refactor: リファクタ → テスト実行 → 成功を確認
```

**バグ修正ステップの場合:**
上記サイクルの代わりに「バグ修正時のリグレッションテスト必須化」フローに従う。

#### 3.3 レビューサイクル 1回目

**レビュー実行**:
```
1. 変更したファイルの差分を確認（git diff）
2. レビュー観点チェックリストに基づいて自己レビュー
3. 問題点をリストアップ
```

**修正実行**:
```
1. リストアップした問題点を優先度順に修正
2. 修正内容を記録
3. lint/format を変更対象ファイルのみに実行して整合性確認
   ※ プロジェクト全体への lint/format 実行は禁止（L-1）
```

#### 3.4 レビューサイクル 2回目

**レビュー実行**:
```
1. 1回目の修正結果を確認
2. 見落とした問題がないか再度チェック
3. 新たに発生した問題がないか確認
```

**修正実行**:
```
1. 残存する問題点を修正
2. 最終的なコード品質を確認
3. lint/format を変更対象ファイルのみに実行
   ※ typecheck は自動実行しない（L-2: 実行に数分かかりタイムアウトの原因になる）
```

#### 3.5 ステップ完了確認

- テストがすべて通ることを確認
- TodoWrite で該当ステップを completed にマーク
- 完了したステップの `- [ ]` を `- [x]` に更新する（plan.md 内）
- 次のステップへ進む

### Phase 4: 最終検証

- format、lint を**変更対象ファイルのみ**に実行してコード品質を確保する（L-1: プロジェクト全体への実行は禁止）
- test を実行し、すべての失敗を修正する
- typecheck は自動実行しない。ユーザーに手動実行を促すか、CI に委ねる（L-2: 実行に数分かかりタイムアウトの原因になる）
- ドメイン層・複雑ロジックの追加箇所は新規テストが緑になることを確認し、必要に応じて回帰テストも走らせる
- 受入基準が満たされていることを確認する

### Phase 5: AI コメントの整理

- 実装コード内のすべての AI 生成コメントをレビューする
- コードが既に表現している内容を単純に繰り返す冗長なコメントを削除する
- 特定の目的がない限り、コメントアウトされたコードブロックを削除する
- コード構造では表現できない価値あるコンテキストを提供するコメントのみ残す
- 過剰なコメントではなく、明確な命名と構造によってコードを自己説明的にする
- 原則を適用する：「コードが言えないことをコメントし、コードが言わないことを単にコメントしない」

参考: https://github.com/97-things/97-things-every-programmer-should-know/tree/master/en/thing_16

### Phase 5.5: ドキュメント同期

Phase 6 の全体レビュー前に、documenter エージェントを起動してドキュメントとコードの整合性を検証する。

**実行内容:**
1. documenter エージェントを起動し、全ステップの変更をまとめて検証
2. plan.md / openspec/specs/ との差分を検出
3. 差分がある場合、ドキュメント更新を実行
4. レビューサイクルで発見された問題パターンを `tmp/lessons.md` に記録
5. 知識の圧縮（冗長な記録の統合・要約）

**UX・アクセシビリティ改善（UI 変更がある場合のみ）:**
- 全体レビュー（Phase 6）の観点に含める形で統合
- Phase 6.1 のチェック項目に「WCAG 2.1 AA 準拠」「デザインシステム整合性」を追加

**パフォーマンス最適化（該当する場合のみ）:**
- 全体レビュー（Phase 6）の観点に含める形で統合
- Phase 6.1 のチェック項目に「N+1 クエリ」「不要な再レンダリング」を追加

### Phase 6: 全体レビューサイクル（最終2回）

実装全体に対して最終的なレビューサイクルを2回実行する：

#### 6.1 全体レビュー 1回目

```
1. 実装全体を俯瞰してレビュー
2. ステップ間の整合性を確認
3. アーキテクチャ的な問題がないか確認
4. パフォーマンス上の懸念がないか確認（N+1 クエリ、不要な再レンダリング）
5. セキュリティ上の問題がないか確認
6. UI 変更がある場合: WCAG 2.1 AA 準拠、デザインシステム整合性を確認
```

#### 6.2 全体修正 1回目

```
1. 発見した問題を修正
2. 必要に応じてリファクタリング
3. テストを再実行
```

#### 6.3 全体レビュー 2回目

```
1. 修正結果を確認
2. 最終的なコード品質を評価
3. ドキュメント（TSDoc等）の整合性を確認
```

#### 6.4 全体修正 2回目

```
1. 最終的な微調整
2. lint/format を変更対象ファイルのみに実行（L-1: プロジェクト全体への実行は禁止）
3. test を実行し、すべてのチェックがパスすることを確認
4. typecheck はユーザーに手動実行を促す（L-2: 自動実行しない）
```

### Phase 7: 完了報告

- すべてのチェックリスト項目が完了したことを確認する
- 実装計画を再度読み、すべてを実装したことを確認する
- レビューサイクルの実施記録を含めて完了ステータスを報告する

**tmp/ クリーンアップ:**
- `./tmp/step-*` ファイルを削除する（一時的なエージェント出力）
- `./tmp/lessons.md` は保持する
- `./tmp/plan.md` は保持する

**報告フォーマット:**

```markdown

- 使用した専門エージェント: [一覧]

- ドキュメント同期: Y 回

### TDD チェックポイント実施記録

### 品質確認結果

- [ ] format: パス（変更対象ファイルのみ）
```

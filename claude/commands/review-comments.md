---
description: PRレビューコメントの妥当性チェックと対応方針の検討（/review-comments <PR URL>）
---

PRに寄せられたレビューコメントを取得し、各指摘の妥当性を評価した上で、対応が必要なものについて具体的な修正方針を提案する。

**IMPORTANT**: Always respond in Japanese.
**IMPORTANT**: ソースコードは変更しない。分析結果の出力のみ行う。

## パラメーター

`$ARGUMENTS` にPRのURLまたはPR番号を指定する（必須）。

例:
- `/review-comments https://github.com/org/repo/pull/123`
- `/review-comments 123`

## 実行条件

以下を確認する:

- `$ARGUMENTS` が指定されていること
- 指定されたPRが存在し、アクセス可能であること

条件が満たされない場合:

- プロセスを即座に停止する
- 状況をユーザーに報告する
- 処理を続行しない

## 実行プロセス

### フェーズ 1: PRメタデータの取得

`$ARGUMENTS` からPR URLまたは番号を解析し、メタデータを取得する。

```bash
# URLからowner/repo/numberを抽出、または番号のみならカレントリポジトリを使用
pr_url="$ARGUMENTS"

# PRの基本情報を取得
gh pr view "$pr_url" --json number,title,baseRefName,headRefName,author,state
```

取得する情報:
- PR番号、タイトル、ステータス
- ベースブランチ、ヘッドブランチ
- PR作成者

### フェーズ 2: レビューコメントの取得（未解決・人間のみ）

GraphQL APIを使い、**未解決のレビュースレッド**のみを取得する。

```bash
# GraphQLでunresolvedなレビュースレッドを取得
gh api graphql -f query='
query($owner: String!, $repo: String!, $number: Int!) {
  repository(owner: $owner, name: $repo) {
    pullRequest(number: $number) {
      reviewThreads(first: 100) {
        nodes {
          isResolved
          isOutdated
          path
          line
          startLine
          comments(first: 50) {
            nodes {
              author {
                login
              }
              body
              createdAt
              url
              originalPosition
              diffHunk
            }
          }
        }
      }
    }
  }
}' -f owner="$owner" -f repo="$repo" -F number="$number"
```

**フィルタリングルール**:

1. **resolved済みを除外**: `isResolved: true` のスレッドは完全にスキップする
2. **botコメントを除外**: コメント作成者のloginが以下に該当するものはスキップする
   - `[bot]` を含むユーザー名（例: `github-actions[bot]`, `dependabot[bot]`）
   - `copilot`、`gemini`、`coderabbit` など既知のAIレビューbot
   - GitHub Actionsなどの自動化ツールによるコメント
3. **outdatedスレッドの扱い**: `isOutdated: true` のスレッドは取得するが、outdatedであることを明記する

フィルタリング後にコメントが0件の場合:
- 「未解決のレビューコメントはありません」と報告して終了する

### フェーズ 3: 該当コードの読み取り

各レビューコメントが指す**実際のコード**を読み取る。

1. diffの取得:
   ```bash
   gh pr diff "$pr_url"
   ```

2. 各コメントの `path` と `line` から、該当ファイルの現在の内容を読み取る
3. コメントに含まれる `diffHunk` で差分のコンテキストも確認する
4. 必要に応じて、周辺のコード（関連する関数やクラス全体）も読み取り、コンテキストを理解する

### フェーズ 4: 妥当性の評価

各レビューコメントについて、以下の観点で妥当性を評価する:

#### 評価基準

| 観点 | 確認内容 |
|------|---------|
| **技術的正確性** | 指摘の技術的根拠は正しいか？ |
| **コンテキスト理解** | レビュアーはコードの意図・背景を正しく理解しているか？ |
| **影響度** | 指摘を無視した場合のリスク（バグ、パフォーマンス、セキュリティ、保守性） |
| **代替案の妥当性** | レビュアーが提案する修正案は適切か？より良い方法はあるか？ |
| **プロジェクト慣習** | プロジェクトのコーディング規約やパターンに照らして妥当か？ |

#### 判定カテゴリ

各コメントを以下のいずれかに分類する:

- **✅ 対応必須**: 技術的に正しく、修正しないとバグやリスクにつながる
- **🔶 対応推奨**: 正しい指摘だが、緊急性は低い（コード品質・保守性の改善）
- **💬 議論が必要**: 妥当性が判断しづらい、またはトレードオフがある
- **⏭️ 対応不要**: 指摘が技術的に不正確、またはコンテキストの誤解に基づく

### フェーズ 5: 対応方針の策定

**✅ 対応必須** および **🔶 対応推奨** と判定したコメントについて、具体的な対応方針を策定する:

- 修正すべきファイルとコード箇所の特定
- 具体的な修正内容（コード例付き）
- 修正による影響範囲の評価
- テストへの影響

**💬 議論が必要** のコメントについては:

- 賛成・反対それぞれの論点を整理
- PR上で返信する際の推奨コメント案を作成

**⏭️ 対応不要** のコメントについては:

- 対応不要と判断した理由を明記
- PR上で返信する際の推奨コメント案を作成（丁寧に根拠を説明する内容）

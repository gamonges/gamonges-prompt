---
description: ユーザーのフィードバックに基づいて実装計画を修正する（/revise）
---

既存の実装計画をユーザーのフィードバックに基づいて修正する。
初期設計と実装の間のギャップを埋め、ユーザーの知見を計画に反映する。

**重要**: `./tmp/plan.md` もしくはCursorのplanモードで作成した`***.plan.md`のみ変更する

## パラメーター

`$ARGUMENTS` でフィードバックファイルのパスを指定できる。省略時は `./tmp/context.md` を使用する。

## ワークフロー上の位置付け

| 前工程 | 本コマンド | 後工程 |
|--------|-----------|--------|
| `/design` 計画作成 | `/revise` 計画修正 | `/revise` 再修正 or `/implement` 実装 |

## 実行条件

以下のファイルが存在することを確認する:

- `./tmp/plan.md` — 既存の実装計画
- `$ARGUMENTS`（指定がない場合は `./tmp/context.md`）— ユーザーのフィードバック・修正要件

いずれかのファイルが存在しない場合:

- プロセスを即座に停止する
- どの条件が満たされていないかをユーザーに報告する
- 実装を続行しない

## 実行プロセス

### フェーズ 1: 現行計画の読み込み

- [ ] `./tmp/plan.md` から既存の実装計画を読み込む
- [ ] 現在の構成・要件・アプローチを把握する
- [ ] 計画の各ステップと依存関係を特定する

### フェーズ 2: フィードバックの読み込みと分類

- [ ] フィードバックファイル（`$ARGUMENTS` または `./tmp/context.md`）からフィードバックを読み込む
- [ ] 個別のフィードバック項目を識別する（区切り: `=====` やリスト形式など）
- [ ] ファイル参照やコンテキスト情報を抽出する（例: `tmp/plan.md:L144`）
- [ ] 各フィードバックを以下に分類する:
  - **要件変更**: 機能要件・非機能要件の追加・変更・削除
  - **技術的懸念**: パフォーマンス・セキュリティ・保守性への懸念
  - **アプローチ変更**: 実装方式・設計方針の変更提案

### フェーズ 3: ユーザー確認（必要に応じて）

以下の問題を検出した場合、**プロセスを即座に停止**してユーザーに報告する:

- **曖昧なフィードバック**: 複数の解釈が可能な場合
  - どのフィードバックが不明確か、または複数の解釈が考えられるかを具体的に質問する
- **矛盾する要求**: フィードバック間で整合性がとれない場合
  - 矛盾を明確に提示し、優先度の判断を依頼する
- **技術的制約の不足**: 実装方式に影響する制約が不明な場合
  - 不足している技術的詳細や制約を具体的に質問する

複数の問題がある場合のフォーマット:

```
## 問題 1: [問題の要約]
[該当するフィードバックの引用と、何が問題なのかの説明]

## 問題 2: [問題の要約]
[該当するフィードバックの引用と、何が問題なのかの説明]
```

### フェーズ 4: 調査・分析

各フィードバック項目について、サブエージェントを使って**並列に**調査を実施する:

- [ ] Serena MCP でセマンティックなコード検索を行い、フィードバックに関連するコードパターンや実装を調査する
- [ ] Web search tool で外部ドキュメントや事例を検索する
- [ ] Context7 MCP でライブラリ・フレームワークの使用方法を確認する
- [ ] フィードバックで指摘されたファイルや箇所を直接確認する

### フェーズ 5: 計画の修正方針の決定

- [ ] 各フィードバックの実現可能性を検証する
- [ ] フィードバック間の矛盾を特定し解決する
- [ ] 既存計画の有効な部分を維持しつつ、修正方針を決定する
- [ ] 修正による他ステップへの波及影響を評価する

### フェーズ 6: plan.md の更新

- [ ] `./tmp/plan.md` を修正後の実装計画で上書きする
- [ ] すべてのフィードバックを反映しつつ、元の構成を維持する
- [ ] 調査で得た最新情報（ファイルパス、URL、コード例など）で古い情報を更新する
- [ ] 修正履歴は含めない（計画として完結した状態にする）

### フェーズ 7: コミット

以下のコマンドで変更をコミットする:

```bash
git add ./tmp/plan.md
git commit -m "docs: 実装計画を修正"
```

コミットに失敗した場合はユーザーに報告し、手動での対応を依頼する。

### フェーズ 8: 完了報告

以下の内容をユーザーに報告する:

- 計画に加えた主な変更点
- 各フィードバックへの対応結果

複数のフィードバックがある場合のフォーマット:

```
## フィードバック 1: [フィードバックの要約]
[この修正で何をどう変更したかの説明]

## フィードバック 2: [フィードバックの要約]
[この修正で何をどう変更したかの説明]
```

- 次のステップの案内: `/revise` で再度修正、`/implement` で実装開始
